# Convenience Extension Methods

While `builder.Eventing.Subscribe<T>()` gives you full control, Aspire provides streamlined extension methods in `DistributedApplicationEventingExtensions` that eliminate boilerplate for the most common resource event patterns.

## Available Methods

These methods hang directly off `IResourceBuilder<T>`, making them fluent and discoverable:

- **`OnBeforeResourceStarted<T>()`** — runs before the resource starts (migrations, seeding)
- **`OnResourceReady<T>()`** — runs after health checks pass (verification, warming)
- **`OnConnectionStringAvailable<T>()`** — runs when the connection string is resolved (dependent config)
- **`OnResourceStopped<T>()`** — runs when the resource stops (cleanup, notifications)
- **`OnInitializeResource<T>()`** — runs during resource initialization
- **`OnResourceEndpointsAllocated<T>()`** — runs when endpoints are allocated

## Before vs. Manual Subscribe

The convenience methods reduce a three-line subscription to a single fluent call:

```csharp
// Manual approach
builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    db.Resource,
    async (evt, ct) => { /* seed */ });

// Convenience method — same result, less ceremony
db.OnBeforeResourceStarted(async (resource, evt, ct) =>
{
    // 'resource' is already typed as your resource
    // 'evt' is the BeforeResourceStartedEvent
});
```

## Real-World Patterns

```csharp
var db = builder.AddPostgres("pg").AddDatabase("mydb");

// Seed the database before it starts accepting connections
db.OnBeforeResourceStarted(async (resource, evt, ct) =>
{
    // Run SQL migration scripts
});

// Verify the database is healthy and accessible
db.OnResourceReady(async (resource, evt, ct) =>
{
    // Run a quick SELECT 1 to confirm connectivity
});

// React when connection info becomes available
db.OnConnectionStringAvailable(async (resource, evt, ct) =>
{
    // Push connection string to external config store
});
```

## When to Use Which

Use convenience methods for straightforward single-resource handlers. Fall back to `builder.Eventing.Subscribe<T>()` when you need access to `evt.Services`, want to subscribe to custom event types, or need to control `EventDispatchBehavior`. Both approaches are fully compatible — you can mix them freely in the same AppHost.
