# Introduction to the Aspire Eventing System

Aspire's eventing system lets you hook into the application lifecycle at precise moments — before startup, after resources are created, or when a specific resource becomes ready. The central interface is `IDistributedApplicationEventing`, accessed through `builder.Eventing`.

## Subscribing to Events

Every event is a strongly-typed class. Call `Subscribe<T>(callback)` to listen for global events, or `Subscribe<T>(resource, callback)` to scope a subscription to a single resource. Each call returns a `DistributedApplicationEventSubscription` you can later dispose to unsubscribe.

```csharp
// Global subscription — fires for the entire app
var subscription = builder.Eventing.Subscribe<BeforeStartEvent>(
    async (evt, ct) =>
    {
        var logger = evt.Services.GetRequiredService<ILogger<Program>>();
        logger.LogInformation("App is about to start...");
    });

// Resource-scoped subscription
var db = builder.AddPostgres("pg").AddDatabase("mydb");
builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    db.Resource,
    async (evt, ct) =>
    {
        // Only fires for this specific database
    });
```

## What Events Carry

Every event object includes a `Services` property giving you access to the full `IServiceProvider`. Lifecycle events like `BeforeStartEvent` also carry a `Model` property — the `DistributedApplicationModel` containing all registered resources. Resource-scoped events expose a `Resource` property pointing to the specific resource that triggered the event.

## Unsubscribing

The returned `DistributedApplicationEventSubscription` implements `IDisposable`. Dispose it or call `builder.Eventing.Unsubscribe()` to remove the handler. This is useful for one-shot tasks like initial data seeding where you don't need the handler after it runs once.

Understanding eventing unlocks powerful patterns: validation gates, dynamic configuration, ordered startup, and database seeding — all without modifying your service code.
