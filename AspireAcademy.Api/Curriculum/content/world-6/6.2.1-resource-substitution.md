# Resource Substitution in Tests

How to replace, modify, or disable resources in your test environment to make tests faster, more isolated, and more reliable.

## Why Substitute Resources?

Your production AppHost might define resources that are expensive, slow, or unnecessary for a particular test:

- A full SQL Server container when your test only exercises the web frontend
- An Azure resource that requires cloud credentials
- A heavy container image that takes minutes to pull

The testing builder gives you full access to the resource graph before the app starts, so you can customize it for your test scenario.

## Modifying Configuration

The simplest form of resource substitution is overriding configuration values. The builder's `Configuration` property lets you set values that your AppHost reads:

```csharp
var builder = await DistributedApplicationTestingBuilder
    .CreateAsync<Projects.MyApp_AppHost>();

// Override a configuration value the AppHost uses
builder.Configuration["Parameters:DbPassword"] = "test-password-123";
builder.Configuration["Parameters:ApiKey"] = "test-key";
```

This is useful when your AppHost reads parameters or feature flags from configuration to decide how to set up resources.

## Replacing Services in DI

You can replace services registered in the AppHost's DI container. This is helpful for swapping out infrastructure services:

```csharp
var builder = await DistributedApplicationTestingBuilder
    .CreateAsync<Projects.MyApp_AppHost>();

// Replace a service with a test-specific implementation
builder.Services.AddSingleton<IEmailSender, FakeEmailSender>();
```

## Using the Builder's Eventing System

The `Eventing` property lets you hook into lifecycle events to modify resources before they start. You can subscribe to `BeforeResourceStartedEvent` to alter resource configuration:

```csharp
var builder = await DistributedApplicationTestingBuilder
    .CreateAsync<Projects.MyApp_AppHost>();

builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    "webfrontend",
    async (evt, ct) =>
    {
        // Modify the resource before it starts
        var env = evt.Resource.Annotations.OfType<EnvironmentCallbackAnnotation>();
        // ... customize environment variables
    });
```

## Overriding Connection Strings

If you want a service to connect to a pre-existing database instead of one managed by Aspire, override the connection string in configuration:

```csharp
var builder = await DistributedApplicationTestingBuilder
    .CreateAsync<Projects.MyApp_AppHost>();

// Point "catalogdb" at an existing test database
builder.Configuration["ConnectionStrings:catalogdb"] =
    "Host=localhost;Database=test_catalog;Username=test;Password=test";
```

Services that consume `catalogdb` via `WithReference` will receive this connection string instead of the one from the container resource.

## Configuring Builder Options

Use the `configureBuilder` overload to set up the host environment for testing:

```csharp
var builder = await DistributedApplicationTestingBuilder.CreateAsync<Projects.MyApp_AppHost>(
    args: [],
    configureBuilder: (appOptions, hostSettings) =>
    {
        hostSettings.EnvironmentName = "Testing";
    }
);
```

Your AppHost can then check the environment to conditionally configure resources:

```csharp
// In your AppHost's Program.cs
var builder = DistributedApplication.CreateBuilder(args);

if (builder.Environment.IsEnvironment("Testing"))
{
    // Use a lighter-weight setup for tests
    var db = builder.AddPostgres("pg").AddDatabase("catalogdb");
}
else
{
    // Full production setup
    var db = builder.AddAzurePostgresFlexibleServer("pg").AddDatabase("catalogdb");
}
```

## Pattern: Conditional Resource Setup

A common pattern is designing your AppHost to be test-aware from the start:

```csharp
// AppHost Program.cs
var builder = DistributedApplication.CreateBuilder(args);

var useEmulator = builder.Configuration.GetValue<bool>("UseEmulators");

IResourceBuilder<IResourceWithConnectionString> cosmos;
if (useEmulator)
{
    cosmos = builder.AddAzureCosmosDB("cosmos").RunAsEmulator();
}
else
{
    cosmos = builder.AddAzureCosmosDB("cosmos");
}

var api = builder.AddProject<Projects.Api>("api")
    .WithReference(cosmos);
```

Then in your test:

```csharp
var builder = await DistributedApplicationTestingBuilder
    .CreateAsync<Projects.MyApp_AppHost>();

builder.Configuration["UseEmulators"] = "true";

await using var app = await builder.BuildAsync();
```

## Pattern: Testing a Single Service

When testing just one service, you may not need the entire application graph. While you can't remove resources already defined by the AppHost, you can focus your test on a specific resource and only wait for its dependencies:

```csharp
public async Task InitializeAsync()
{
    var builder = await DistributedApplicationTestingBuilder
        .CreateAsync<Projects.MyApp_AppHost>();

    _app = await builder.BuildAsync();
    await _app.StartAsync();

    // Only wait for the service under test and its direct dependency
    await _app.ResourceNotifications.WaitForResourceHealthyAsync("catalogdb");
    await _app.ResourceNotifications.WaitForResourceHealthyAsync("apiservice");
}

[Fact]
public async Task CatalogApiReturnProducts()
{
    var httpClient = _app.CreateHttpClient("apiservice");
    var response = await httpClient.GetAsync("/api/catalog");

    response.EnsureSuccessStatusCode();
}
```

## Summary

| Technique | Use Case |
| --- | --- |
| Configuration overrides | Swap parameters, feature flags, connection strings |
| DI service replacement | Replace infrastructure services with fakes |
| Environment-based branching | Different resource setup for test vs production |
| Lifecycle events | Modify resources just before they start |
| Scoped waiting | Focus on a subset of resources for faster tests |

These techniques let you tailor the test environment without maintaining a separate AppHost for testing. Your production AppHost remains the source of truth, and tests layer customizations on top.
