# Docker and Kubernetes Deployment

Aspire's manifest can be transformed into platform-specific deployment artifacts for Docker Compose and Kubernetes. Each platform maps Aspire concepts to its native resource types.

## Docker Compose Publishing

When targeting Docker Compose, Aspire resources map naturally:

| Aspire Resource | Docker Compose Equivalent |
|---|---|
| `project.v0` | Service with `build` context |
| `container.v0` | Service with `image` reference |
| Bindings | `ports` mapping |
| Environment variables | `environment` section |
| Volumes | `volumes` mount |

A generated `docker-compose.yml` preserves the dependency graph from your AppHost. Resources referenced via `WithReference()` appear as `depends_on` entries, ensuring correct startup ordering.

```yaml
services:
  apiservice:
    build:
      context: ../ApiService
    ports:
      - "8080:8080"
    environment:
      ConnectionStrings__cache: "cache:6379"
    depends_on:
      - cache
  cache:
    image: redis:7.4
    ports:
      - "6379:6379"
```

## Kubernetes Publishing

Kubernetes publishing generates multiple resource manifests:

| Aspire Concept | Kubernetes Resource |
|---|---|
| `project.v0` | Deployment + Service |
| `container.v0` | Deployment + Service |
| Environment variables | ConfigMap entries |
| Secrets / Parameters | Secret objects |
| Bindings | Service port definitions |

Each project resource becomes a Deployment with a corresponding ClusterIP Service. Connection strings and service discovery URLs are injected via ConfigMaps or Secrets.

## Customization Hooks

Both publishers support customization. You can intercept the generation process to add platform-specific annotations (like Kubernetes ingress rules or Docker Compose resource limits) without modifying your AppHost code. This keeps the AppHost focused on application topology while deployment details stay in the publishing layer.

The key insight is that your AppHost is deployment-target agnostic. The same `builder.AddRedis("cache")` call works whether you deploy to Docker Compose, Kubernetes, or Azure — only the publisher changes.
