# Testing Patterns and Best Practices

AppHost startup is expensive — it launches containers, initializes databases, and starts service projects. Effective test design minimizes this cost while maintaining isolation between test cases.

## Sharing AppHost with xUnit Fixtures

Use xUnit class fixtures to start the AppHost once and share it across all tests in a class:

```csharp
public class AppHostFixture : IAsyncLifetime
{
    public DistributedApplication App { get; private set; } = null!;

    public async Task InitializeAsync()
    {
        var builder = await DistributedApplicationTestingBuilder
            .CreateAsync<Projects.MyAppHost>();
        App = await builder.BuildAsync();
        await App.StartAsync();
    }

    public async Task DisposeAsync()
    {
        await App.DisposeAsync();
    }
}

public class ApiTests : IClassFixture<AppHostFixture>
{
    private readonly DistributedApplication _app;
    public ApiTests(AppHostFixture fixture)
    {
        _app = fixture.App;
    }
}
```

For sharing across multiple test classes, use collection fixtures instead.

## Database Isolation

When tests share an AppHost, database state leaks between tests. Strategies for isolation include using transactions that roll back after each test, creating unique table prefixes per test, or inserting data with unique identifiers and querying only for those specific records.

## Common Pitfalls

**Port conflicts** occur when `randomizePorts` is disabled or tests launch multiple AppHosts simultaneously. Always use the default randomized ports.

**Stale state** happens when tests depend on a clean database but previous tests left data behind. Design tests to be additive rather than assuming empty tables.

**Timing assertions** are fragile. Never use `Task.Delay` to wait for readiness — use `WaitForResourceHealthyAsync` or poll with retry logic instead.

**Parallel execution** requires each test to use unique resource names and identifiers to prevent cross-test interference.
