# The Aspire Publish Command

The `aspire publish` command bridges the gap between local development and production deployment. It transforms your code-first distributed application definition into a declarative manifest file that deployment tools can consume.

## What It Does

When you run `aspire publish`, Aspire evaluates your AppHost project and generates `aspire-manifest.json` — a complete description of every resource, its bindings, environment variables, connection strings, and relationships. This manifest is the deployment contract: it tells infrastructure tooling exactly what your application needs without requiring access to your C# source code.

```bash
# Generate the manifest from your AppHost
aspire publish --output-path ./deploy
```

## Why a Manifest?

The manifest decouples your application definition from any specific deployment target. The same `aspire-manifest.json` can be consumed by:

- **Azure Developer CLI (azd)** to provision Azure resources
- **Docker Compose generators** to create `docker-compose.yml`
- **Kubernetes generators** to produce Deployments and Services
- **Custom tooling** that reads the JSON schema

## Manifest Overview

The generated JSON contains a `resources` object where each key is a resource name and each value describes its type, bindings, environment variables, and dependencies:

```json
{
  "resources": {
    "cache": {
      "type": "container.v0",
      "image": "redis:7.4",
      "bindings": {
        "tcp": { "scheme": "tcp", "protocol": "tcp", "targetPort": 6379 }
      }
    },
    "apiservice": {
      "type": "project.v0",
      "path": "../ApiService/ApiService.csproj",
      "env": {
        "ConnectionStrings__cache": "{cache.bindings.tcp.host}:{cache.bindings.tcp.port}"
      }
    }
  }
}
```

This declarative format makes it possible to inspect, validate, and transform your application topology before any infrastructure is provisioned.
