# Polyglot Support & Code Generation

Aspire isn't just for .NET — its app model supports orchestrating applications written in any language. Understanding how this works reveals the boundary between Aspire's .NET-native orchestration and its language-agnostic deployment capabilities.

## The App Model as Intermediate Representation

The Aspire app model — resources, endpoints, connection strings, and their relationships — is a **language-neutral graph**. When you call `builder.AddPostgres("pg").AddDatabase("mydb")`, you're building a dependency graph that describes infrastructure, not .NET-specific code.

This graph can be serialized to a JSON intermediate representation (the manifest), which any language ecosystem can consume. The `aspire publish` command generates this manifest, and language-specific code generators can transform it into typed SDK clients.

## Language-Specific Generators

Aspire includes code generation packages for different language ecosystems under `src/Aspire.Hosting.CodeGeneration.*`:

- **TypeScript** (`Aspire.Hosting.CodeGeneration.TypeScript`) — Generates typed configuration objects with connection strings
- **Python** (`Aspire.Hosting.CodeGeneration.Python`) — Generates configuration modules with type hints
- **Go** (`Aspire.Hosting.CodeGeneration.Go`) — Generates structs with connection metadata
- **Java** (`Aspire.Hosting.CodeGeneration.Java`) — Generates configuration classes for Spring Boot integration
- **Rust** (`Aspire.Hosting.CodeGeneration.Rust`) — Generates configuration with connection metadata

Each generator reads the same manifest but produces idiomatic code for its target language. Code generation is available via the `aspire sdk generate` subcommand (a hidden developer command).

## Executable Resources

For local development, non-.NET applications are modeled as `ExecutableResource` types. DcpExecutor launches them as child processes with environment variables containing resolved connection strings and endpoint URLs. The application reads these environment variables using its language's standard library.

## Environment Variable Convention

The bridge between Aspire's expression system and polyglot apps is simple: **environment variables**. When DcpExecutor starts a resource, `ExpressionResolver` (from `src/Aspire.Hosting/ApplicationModel/ExpressionResolver.cs`) resolves all expressions and injects them as environment variables. A Python app reads `ConnectionStrings__mydb` just like a .NET app would read `builder.Configuration.GetConnectionString("mydb")`.
