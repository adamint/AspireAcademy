# DashboardWebApplication: Wiring Up the Dashboard

The Aspire dashboard is a standalone ASP.NET Core application defined at `src/Aspire.Dashboard/DashboardWebApplication.cs`. The `public sealed class DashboardWebApplication : IAsyncDisposable` is the composition root that wires together telemetry ingestion, resource monitoring, and the Blazor UI.

## gRPC Service Registration

The dashboard acts as an OpenTelemetry collector by mapping three gRPC services for OTLP ingestion:

- `MapGrpcService<OtlpGrpcTraceService>()` — Receives distributed traces
- `MapGrpcService<OtlpGrpcLogsService>()` — Receives structured logs
- `MapGrpcService<OtlpGrpcMetricsService>()` — Receives metrics

These services listen on a dedicated OTLP port (default 18889), separate from the browser-facing UI port. Applications instrumented with OpenTelemetry export telemetry directly to these endpoints.

## DashboardClient

The dashboard connects back to the AppHost via `DashboardClient`, which calls the AppHost's resource service gRPC endpoint. This is how the dashboard receives `ResourceNotificationService` updates (covered in 6.7.4) — through a streaming gRPC connection that pushes `CustomResourceSnapshot` changes.

## Authentication

DashboardWebApplication configures multiple authentication modes:

- **BrowserToken** — A one-time token displayed in the console for local development
- **OpenIdConnect** — For production deployments with identity providers
- **Certificate** — Mutual TLS for secure environments
- **Unsecured** — For trusted network scenarios

## Blazor Server Rendering

The UI is a Blazor Server application, meaning all rendering happens server-side with SignalR pushing DOM diffs to the browser. When `TelemetryRepository` receives new traces or `DashboardClient` receives resource updates, Blazor components re-render automatically. This architecture eliminates client-side complexity while enabling real-time updates — the dashboard never polls for data.
