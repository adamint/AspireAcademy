# Pipeline Architecture

How PipelineStep, PipelineContext, and PipelineStepContext orchestrate the publish/deploy lifecycle.

Aspire uses a **pipeline architecture** to transform your app model into deployable artifacts. The `IDistributedApplicationPipeline` orchestrates an ordered sequence of steps that produce manifests, Docker Compose files, Kubernetes manifests, or cloud deployments.

## Core Types

### PipelineStep

A `PipelineStep` is a named unit of work in the pipeline. Each step has an `Id`, optional dependencies, and an execution function:

```csharp
public class PipelineStep
{
    public string Id { get; }
    public Func<PipelineStepContext, Task> ExecuteAsync { get; }
}
```

### PipelineContext

The `PipelineContext` carries shared state across all steps — the app model, configuration, output directory, and any artifacts produced by earlier steps.

### PipelineStepContext

Each step receives a `PipelineStepContext` with access to the pipeline context plus step-specific metadata like the step ID and cancellation token.

## How the Pipeline Runs

When you execute `aspire publish` or `aspire deploy`, Aspire:

1. **Builds the app model** by running your AppHost's `Program.cs`
2. **Creates a pipeline** with the appropriate steps for the target environment
3. **Orders the steps** based on declared dependencies
4. **Executes each step** sequentially, passing context between them

```
AppHost Program.cs
    │
    ▼
App Model Built
    │
    ▼
Pipeline Created
    │
    ▼
Step 1: publish-manifest → aspire-manifest.json
    │
    ▼
Step 2: build → Container images built
    │
    ▼
Step 3: push → Images pushed to registry
    │
    ▼
Step 4: generate → Compose/K8s files generated
```

## Extensibility

You can add custom pipeline steps to inject your own logic — generating custom configuration files, running database migrations, or notifying external systems — by registering steps with the pipeline and declaring their dependencies on well-known steps.
