# DcpHost: Bootstrapping the Developer Control Plane

The Developer Control Plane (DCP) is the local orchestration engine that makes `dotnet run` on your AppHost actually start containers and executables. The entry point lives at `src/Aspire.Hosting/Dcp/DcpHost.cs`, where `internal sealed class DcpHost` manages the entire DCP lifecycle.

## The Startup Chain

When you call `DistributedApplication.RunAsync()`, the hosting infrastructure eventually invokes `DcpHost.StartAsync(CancellationToken)`. This method performs several critical steps:

1. **Container runtime detection** — DcpHost queries `IContainerRuntime` (defined at `src/Aspire.Hosting/Publishing/IContainerRuntime.cs`) to detect whether Docker or Podman is available via `CheckIfRunningAsync()`.
2. **DCP sidecar launch** — It starts the DCP process as a sidecar, which exposes a Kubernetes-style gRPC API server on a local port.
3. **gRPC client initialization** — `DcpKubernetesClient` connects to the DCP API, enabling the host to create and manage Container, Executable, and Service custom resources.

## Why Kubernetes-Style?

DCP models local resources using Kubernetes-inspired custom resource definitions (CRDs). A Postgres container becomes a `Container` CRD spec. A .NET project becomes an `Executable` CRD spec. This abstraction lets the same orchestration logic work across different container runtimes.

## Key Relationship

DcpHost is the **bootstrapper** — it starts DCP itself. The actual work of translating your `builder.AddPostgres()` calls into running containers is handled by `DcpExecutor` (covered in 6.7.2). Think of DcpHost as the ignition and DcpExecutor as the engine.

The full startup trace flows: `DistributedApplication.RunAsync()` → `DcpHost.StartAsync()` → container runtime init → gRPC API ready → `DcpExecutor.RunApplicationAsync()`.
