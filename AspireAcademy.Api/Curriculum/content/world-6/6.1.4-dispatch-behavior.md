# Event Dispatch Behavior

When multiple handlers subscribe to the same event, the order and concurrency of their execution matters. Aspire's `EventDispatchBehavior` enum gives you four strategies to control how handlers run when you publish events with `PublishAsync<T>()`.

## The Four Dispatch Modes

### BlockingSequential

Handlers execute one at a time, in subscription order, and the publisher awaits all of them. Use this when Handler B depends on the result of Handler A.

```csharp
await builder.Eventing.PublishAsync(
    myEvent,
    EventDispatchBehavior.BlockingSequential,
    cancellationToken);
```

### BlockingConcurrent

All handlers run in parallel, but the publisher still awaits their completion. This is the best choice when handlers are independent but the caller needs to know they all finished before proceeding.

### NonBlockingSequential

Handlers execute in order, but the publisher does not await them — it fires and forgets. Use this for ordered side-effects like audit logging where you want sequence guarantees but don't want to block startup.

### NonBlockingConcurrent

Handlers run in parallel with no awaiting. This is the most performant option for fully independent, non-critical handlers like telemetry emission or cache warming.

## Choosing the Right Mode

| Scenario | Recommended Mode |
|---|---|
| Migration before seeding | `BlockingSequential` |
| Independent health probes | `BlockingConcurrent` |
| Ordered audit logging | `NonBlockingSequential` |
| Telemetry & metrics fire | `NonBlockingConcurrent` |

## Default Behavior

Built-in Aspire lifecycle events use `BlockingSequential` by default, ensuring predictable ordering. When you publish custom events, you choose the behavior explicitly as the second parameter to `PublishAsync`.

The dispatch behavior only affects execution when there are multiple subscribers. A single handler behaves identically under all four modes. Choose the simplest mode that satisfies your ordering and completion requirements.
