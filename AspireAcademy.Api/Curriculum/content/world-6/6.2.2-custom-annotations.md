# Custom Annotations

Annotations are the metadata backbone of Aspire resources. Every resource carries a collection of `IResourceAnnotation` objects that describe its endpoints, environment variables, container images, health checks, and more. You can create custom annotations to attach any domain-specific metadata.

## Built-In Annotations

Aspire ships dozens of annotation types. The most commonly encountered include:

- **`EndpointAnnotation`** — defines a network endpoint (port, protocol, scheme)
- **`EnvironmentCallbackAnnotation`** — registers a callback that injects environment variables
- **`ContainerImageAnnotation`** — specifies the Docker image, tag, and registry
- **`HealthCheckAnnotation`** — wires an `IHealthCheck` to the resource
- **`ResourceCommandAnnotation`** — adds an interactive command button to the dashboard

## Querying Annotations

Use `resource.TryGetAnnotationsOfType<T>(out var annotations)` to retrieve all annotations of a given type. This returns `true` if any exist and populates an enumerable:

```csharp
if (resource.TryGetAnnotationsOfType<EndpointAnnotation>(out var endpoints))
{
    foreach (var endpoint in endpoints)
    {
        Console.WriteLine($"{endpoint.Name}: {endpoint.Port}");
    }
}
```

## Creating Custom Annotations

Implement `IResourceAnnotation` to define your own metadata. Annotations are composable — a resource can carry multiple annotations of the same type.

```csharp
public record FeatureFlagAnnotation(
    string FlagName,
    bool DefaultValue) : IResourceAnnotation;

// Attach to a resource
builder.AddResource(new MyResource("svc"))
    .WithAnnotation(new FeatureFlagAnnotation("dark-mode", false))
    .WithAnnotation(new FeatureFlagAnnotation("beta-api", true));
```

## Design Guidelines

Keep annotations as simple data carriers — records or plain classes with read-only properties. Avoid embedding behavior or service dependencies in annotations. If you need runtime logic, pair your annotation with a lifecycle hook or event handler that reads the annotation and acts on it. This separation keeps the resource model declarative and testable.
