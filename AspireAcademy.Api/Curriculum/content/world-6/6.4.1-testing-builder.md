# The Testing Builder

Integration testing in Aspire starts with `DistributedApplicationTestingBuilder`, a static class that creates a fully functional AppHost environment for your tests. Unlike unit tests that mock dependencies, Aspire integration tests spin up real resources and validate actual service interactions.

## Creating a Testable Application

The entry point is `DistributedApplicationTestingBuilder.CreateAsync<TEntryPoint>()`, where `TEntryPoint` is your AppHost project class. This method returns a builder you can configure before constructing the application:

```csharp
var builder = await DistributedApplicationTestingBuilder
    .CreateAsync<Projects.MyAppHost>();
await using var app = await builder.BuildAsync();
await app.StartAsync();
```

The `CreateAsync` call discovers your AppHost's `Program.cs`, executes its resource registration logic, and prepares the application model — all without actually starting any resources yet.

## Builder Configuration

Before calling `BuildAsync()`, you can modify the builder to override configuration values, substitute resources, or adjust behavior for the test environment:

```csharp
var builder = await DistributedApplicationTestingBuilder
    .CreateAsync<Projects.MyAppHost>();

builder.Configuration["Parameters:adminPassword"] = "test-password";
```

This is essential for injecting test-specific connection strings, feature flags, or credentials that differ from your development defaults.

## Lifecycle Management

`BuildAsync()` produces a `DistributedApplication` instance. Calling `StartAsync()` launches all registered resources — containers, projects, and executables. The application implements `IAsyncDisposable`, so wrapping it in `await using` ensures all resources are properly torn down when the test completes, preventing orphaned containers or port conflicts between test runs.

The testing builder randomizes ports by default, enabling safe parallel test execution without address conflicts.
