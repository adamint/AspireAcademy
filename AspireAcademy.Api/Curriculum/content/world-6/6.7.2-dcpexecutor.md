# DcpExecutor: The Orchestration Engine

At `src/Aspire.Hosting/Dcp/DcpExecutor.cs`, the ~2000-line `internal sealed partial class DcpExecutor` implements `IDcpExecutor, IConsoleLogsService, IAsyncDisposable`. This is the beating heart of Aspire's local orchestration — it translates your app model into running infrastructure.

## RunApplicationAsync: The Main Loop

When DcpHost finishes bootstrapping, it calls `DcpExecutor.RunApplicationAsync()`. This method iterates every resource in the `DistributedApplicationModel` and dispatches each one:

- **Container resources** → `CreateContainerAsync()` builds a container spec with image, ports, volumes, environment variables, and network aliases, then submits it to DCP's gRPC API.
- **Executable resources** → Creates executable specs for .NET projects and other processes.
- **Port allocation** → Calls `IPortAllocator.AllocatePort()` (from `src/Aspire.Hosting/ApplicationModel/PortAllocator.cs`) starting at port 8000 by default.

## Tracing a Real Example

When you write `builder.AddPostgres("pg")`, here's what DcpExecutor does:

1. Reads the `ContainerImageAnnotation` to get image `postgres:latest`
2. Reads `EndpointAnnotation` to find port 5432
3. Calls `PortAllocator.AllocatePort()` to assign a host port (e.g., 8001)
4. Resolves environment variable expressions (password parameters, etc.)
5. Calls `CreateContainerAsync()` to build the DCP Container spec
6. Submits the spec via `DcpKubernetesClient`

## StartResourceAsync

Individual resources can be started or restarted via `StartResourceAsync()`. This is what powers the "Start" button in the dashboard — it flows through `ResourceNotificationService` back to DcpExecutor.

## Port Allocation Detail

`PortAllocator` at `src/Aspire.Hosting/ApplicationModel/PortAllocator.cs` is a `public sealed class PortAllocator(int startPort = 8000)`. It tracks used ports via `AddUsedPort(int)` and hands out sequential ports via `AllocatePort()`, avoiding conflicts across all resources.
