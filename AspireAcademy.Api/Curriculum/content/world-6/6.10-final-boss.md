# Final Boss: Aspire Architect — Internals Edition

You've explored DcpHost bootstrapping, DcpExecutor orchestration, expression resolution, resource notifications, the dashboard's telemetry pipeline, container runtime abstraction, and the manifest publishing system. Now it's time to prove you can apply this knowledge to diagnose real problems.

## The Scenario

A developer has created a custom hosting integration for a cache service. The container starts but three things are broken:

1. **No endpoints appear in the dashboard** — The resource has no `EndpointAnnotation`, so `DcpExecutor.CreateContainerAsync()` (at `src/Aspire.Hosting/Dcp/DcpExecutor.cs`) never allocates ports via `PortAllocator` (`src/Aspire.Hosting/ApplicationModel/PortAllocator.cs`).

2. **Connection string doesn't resolve** — The `ConnectionStringExpression` references an `EndpointReference` for endpoint `"redis"`, but since no endpoint annotation exists, `ExpressionResolver.EvalExpressionAsync()` (at `src/Aspire.Hosting/ApplicationModel/ExpressionResolver.cs`) can't resolve it.

3. **Health status stays "Starting" forever** — Without a health check, `ResourceNotificationService.PublishUpdateAsync()` (at `src/Aspire.Hosting/ApplicationModel/ResourceNotificationService.cs`) never receives a healthy `CustomResourceSnapshot` update.

## What You Need to Fix

Your challenge (in `challenges/6.10-final-boss.yaml`) requires you to:

- Add `.WithEndpoint("redis", 6379)` to create the `EndpointAnnotation` that DcpExecutor reads
- Verify the `ConnectionStringExpression` matches the endpoint name
- Wire up health checks so `ResourceNotificationService` can transition the resource to healthy
- Explain **which source file** handles each fix and **why** the fix works

This isn't about memorizing APIs — it's about understanding the flow from app model configuration through DcpExecutor processing to dashboard rendering. Every fix traces back to a specific internal mechanism you've studied in Modules 6.7–6.9.
