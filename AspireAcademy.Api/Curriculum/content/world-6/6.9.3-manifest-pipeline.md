# The Manifest Publishing Pipeline

When you move from local development to deployment, Aspire's manifest pipeline transforms your app model into a deployment-ready JSON document. The key files are `src/Aspire.Hosting/Publishing/ManifestPublisher.cs` and `src/Aspire.Hosting/Publishing/ManifestPublishingContext.cs`.

## ManifestPublisher

`ManifestPublisher` is invoked when you run `dotnet run --publisher manifest` or when a deployment tool requests the manifest. It walks the entire `DistributedApplicationModel` and serializes each resource into a standardized JSON format.

## ManifestPublishingContext

`ManifestPublishingContext` provides the `Utf8JsonWriter` that resources use to write their manifest representation. Each resource type knows how to serialize itself — container resources write their image and port bindings, project resources write their project path and launch settings.

## The Manifest Format

The output JSON contains every resource with its:

- **Type** — `container.v0`, `project.v0`, `value.v0`, etc.
- **Connection strings** — Using expression syntax like `{postgres.connectionString}`
- **Endpoint bindings** — Declared ports, schemes, and transport protocols
- **Environment variables** — With expression references to other resources
- **Container details** — Image name, tag, volumes, bind mounts

## Expression Serialization

Connection string expressions like `{postgres.connectionString}` are serialized in their **unresolved form**. Deployment targets (Azure Developer CLI, Kubernetes tooling, etc.) resolve these expressions in the target environment. This is the key difference from local development where `ExpressionResolver` resolves everything eagerly.

## Deployment Target Consumption

Different deployment targets consume the same manifest:

- **Azure Developer CLI (azd)** — Generates Bicep/ARM templates
- **Kubernetes** — Generates Deployment, Service, and ConfigMap resources
- **Docker Compose** — Generates `docker-compose.yml` files

This single-manifest-multiple-targets approach means your `Program.cs` is truly the single source of truth for your distributed application's architecture.
