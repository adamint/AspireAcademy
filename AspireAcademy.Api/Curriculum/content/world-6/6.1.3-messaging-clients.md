# Messaging Clients

Client integrations for message brokers — connecting your .NET services to RabbitMQ, Kafka, and NATS.

---

## RabbitMQ

```csharp
builder.AddRabbitMQClient("messaging");
```

Registers `IConnection` in DI. Usage:

```csharp
app.MapPost("/publish", (IConnection connection) =>
{
    using var channel = connection.CreateModel();
    channel.BasicPublish("events", "content.created", body: messageBytes);
});
```

## Kafka

```csharp
builder.AddKafkaProducer<string, string>("kafka");
builder.AddKafkaConsumer<string, string>("kafka");
```

Registers `IProducer<TKey, TValue>` and `IConsumer<TKey, TValue>` in DI.

## NATS

```csharp
builder.AddNatsClient("nats");
```

Registers `INatsConnection` in DI.


> 🌍 **Real World:** In high-throughput systems, a single RabbitMQ connection can multiplex hundreds of channels. Aspire's pattern of registering `IConnection` in DI mirrors how production systems manage broker connections — one connection, many channels.

## Non-.NET Messaging

For TypeScript, Python, and Go services, use native client libraries:

```python
# Python with pika (RabbitMQ)
import os, pika
connection = pika.BlockingConnection(
    pika.URLParameters(os.environ["ConnectionStrings__messaging"]))
```

```typescript
// TypeScript with amqplib
import amqp from 'amqplib';
const conn = await amqp.connect(process.env.ConnectionStrings__messaging!);
```

```go
// Go with amqp091-go
conn, _ := amqp.Dial(os.Getenv("ConnectionStrings__messaging"))
```

The connection string format is the same — only the client library changes.
