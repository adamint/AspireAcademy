# PipelineStep Dependencies

Use DependsOn and RequiredBy to order pipeline steps with prerequisite relationships.

Pipeline steps don't execute in arbitrary order — they declare **dependency relationships** that the pipeline uses to determine execution order. This ensures steps run only after their prerequisites complete.

## DependsOn

A step can declare that it depends on another step by ID. The pipeline will not execute the dependent step until all dependencies have completed:

```csharp
var generateStep = new PipelineStep("generate-config")
{
    DependsOn = ["publish-manifest", "process-parameters"],
    ExecuteAsync = async context =>
    {
        // This only runs after both publish-manifest
        // and process-parameters have completed
        var manifest = context.GetManifest();
        await GenerateConfigAsync(manifest);
    }
};
```

## RequiredBy

The inverse relationship — a step can declare that it must run **before** another step:

```csharp
var validationStep = new PipelineStep("validate-resources")
{
    RequiredBy = ["publish-manifest"],
    ExecuteAsync = async context =>
    {
        // This runs BEFORE publish-manifest
        ValidateAllResources(context.AppModel);
    }
};
```

## Dependency Resolution

The pipeline topologically sorts all steps based on their `DependsOn` and `RequiredBy` declarations. If a circular dependency is detected, the pipeline throws an error at startup rather than deadlocking.

## Practical Example: Custom Migration Step

A common scenario is running database migrations after the manifest is generated but before deployment:

```csharp
var migrateStep = new PipelineStep("run-migrations")
{
    DependsOn = ["publish-manifest"],
    RequiredBy = ["deploy"],
    ExecuteAsync = async context =>
    {
        var connectionString = context.GetParameterValue("dbconnection");
        await RunMigrationsAsync(connectionString);
    }
};
```

This ensures migrations run in the correct window: after the manifest resolves connection strings, but before the deploy step provisions infrastructure.

## Error Handling

If a step fails, all steps that depend on it are automatically skipped. The pipeline reports which steps succeeded, which failed, and which were skipped due to failed dependencies.
