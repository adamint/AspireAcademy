# Manifest Structure Deep Dive

The `aspire-manifest.json` file uses a structured schema where every resource is described by its type, bindings, environment variables, and relationships. Understanding this structure is essential for debugging deployments and writing custom publishers.

## Resource Types

Each resource has a `type` field that identifies its kind:

| Type | Description |
|---|---|
| `project.v0` | A .NET project built and deployed as a service |
| `container.v0` | A Docker container with image, tag, and port config |
| `value.v0` | A computed value (connection strings, derived config) |
| `parameter.v0` | A user-supplied parameter (secrets, settings) |

## Bindings Section

Bindings describe the network endpoints a resource exposes. Each binding includes `scheme`, `protocol`, `transport`, and optionally `targetPort`:

```json
"bindings": {
  "http": {
    "scheme": "http",
    "protocol": "tcp",
    "transport": "http",
    "targetPort": 8080
  },
  "https": {
    "scheme": "https",
    "protocol": "tcp",
    "transport": "http",
    "targetPort": 8443
  }
}
```

## Environment Variable Placeholders

Environment variables use a placeholder syntax that references other resources' bindings at deployment time:

```json
"env": {
  "ConnectionStrings__cache": "{cache.bindings.tcp.host}:{cache.bindings.tcp.port}",
  "services__api__http__0": "{apiservice.bindings.http.url}"
}
```

The `{resource.bindings.endpoint.property}` pattern lets deployment tools resolve actual values — hostnames, ports, URLs — when infrastructure is provisioned.

## Tracing WithReference to Manifest

When you call `.WithReference(cache)` in your AppHost, Aspire generates the appropriate environment variable entry in the manifest. For `IResourceWithConnectionString` resources, it creates a `ConnectionStrings__name` entry. For service discovery, it generates `services__name__scheme__index` entries. This direct mapping means your AppHost code is the single source of truth for the deployment topology.
