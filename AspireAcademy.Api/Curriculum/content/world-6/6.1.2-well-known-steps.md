# WellKnownPipelineSteps

The built-in pipeline steps: Publish, Deploy, Build, Push, ProcessParameters, and Diagnostics.

Aspire defines a set of **well-known pipeline steps** that handle the standard publish and deploy workflow. These steps have fixed IDs so you can declare dependencies on them from custom steps.

## The publish-manifest Step

The most fundamental step — it serializes the app model to `aspire-manifest.json`:

```
publish-manifest → writes aspire-manifest.json
```

This manifest is the canonical representation of your distributed application: every resource, its type, configuration, bindings, and connection strings. Nearly all other steps depend on this manifest.

## Build and Push Steps

These steps handle container images:

- **build**: Builds container images for project resources using Docker. Each `.csproj` with a Dockerfile or SDK container support gets built.
- **push**: Pushes built images to a container registry. Requires registry configuration.

```
build → push (push depends on build)
```

## ProcessParameters Step

Resolves parameter values — prompting for missing values, reading from configuration, or generating secrets. This step ensures all `parameter.v0` resources in the manifest have concrete values before deployment artifacts are generated.

## Environment-Specific Steps

Different publish targets add their own steps:

- **Docker Compose**: Generates `docker-compose.yaml` from the manifest
- **Kubernetes**: Generates Kubernetes manifests (Deployments, Services, ConfigMaps)
- **Azure**: Provisions Azure resources via Bicep templates

## Step Execution Order

Steps execute in dependency order. The typical flow for `aspire publish`:

```
1. publish-manifest    (no dependencies)
2. process-parameters  (depends on publish-manifest)
3. build               (depends on publish-manifest)
4. push                (depends on build)
5. generate-compose    (depends on publish-manifest, process-parameters)
```

## Listing Steps

You can see which steps will run by examining the pipeline before execution. Each step logs its ID and status, making it easy to debug pipeline issues in the console output.

Understanding well-known step IDs is essential for adding custom steps, since you declare dependencies using these string identifiers.
