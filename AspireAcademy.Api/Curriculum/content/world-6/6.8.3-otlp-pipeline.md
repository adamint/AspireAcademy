# The OTLP Pipeline: From Application Code to Dashboard UI

Understanding the full telemetry pipeline reveals how a single `Activity.Start()` in your application becomes a rendered trace span in the dashboard. Let's trace the complete journey.

## Step 1: Application Instrumentation

Your application creates an `Activity` (the .NET equivalent of a span). The OpenTelemetry SDK's `BatchActivityExportProcessor` batches spans and exports them via the OTLP gRPC exporter, targeting the dashboard's OTLP endpoint.

## Step 2: gRPC Ingestion

At `src/Aspire.Dashboard/Otlp/Grpc/OtlpGrpcTraceService.cs`, the `Export()` method receives the protobuf-encoded `ExportTraceServiceRequest`. It deserializes the spans and performs **resource matching** — associating each span with the correct Aspire resource by matching the OpenTelemetry `service.name` resource attribute.

## Step 3: Storage

Matched spans flow into `TelemetryRepository.AddTraces()` (at `src/Aspire.Dashboard/Otlp/Storage/TelemetryRepository.cs`). The repository stores them in `CircularBuffer` ring buffers and links parent-child span relationships to build trace trees.

## Step 4: Subscriber Notification

After storage, TelemetryRepository notifies watchers. `WatchSpansAsync()` pushes the new span through a `BoundedChannel` to all active subscribers — typically Blazor components rendering the traces page.

## Step 5: Blazor Rendering

The subscribed Blazor component receives the notification, queries `GetTraces()` for updated data, and Blazor Server pushes the DOM diff to the browser via SignalR.

## Peer Resolution

For uninstrumented services (like databases), `ResourceOutgoingPeerResolver` resolves outgoing span attributes (`net.peer.name`, `db.system`) to known Aspire resources. This is how the dashboard draws connection lines between your API and its Postgres database — even though Postgres itself doesn't emit traces. The resolver matches peer metadata against known resource endpoints to infer relationships.
