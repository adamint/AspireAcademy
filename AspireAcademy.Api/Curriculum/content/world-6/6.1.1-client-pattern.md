# Add vs AddKeyed — Client Registration

**Client integrations** are the service-side complement to hosting integrations. While hosting integrations add infrastructure to your AppHost, client integrations register typed clients in your service's DI container.

---

## The Pattern

```csharp
// In your service's Program.cs
builder.AddRedisClient("cache");
builder.AddNpgsqlDataSource("atlas-db");
builder.AddRabbitMQClient("messaging");
```

Each `Add*Client` method:
1. Reads the connection string from `ConnectionStrings::<name>`
2. Configures the client library (Npgsql, StackExchange.Redis, etc.)
3. Registers the client in DI
4. Adds OpenTelemetry instrumentation (tracing, metrics)
5. Adds health checks

## Keyed Clients

When you have multiple instances of the same service type:

```csharp
builder.AddKeyedRedisClient("session-cache");
builder.AddKeyedRedisClient("output-cache");
```

Inject with `[FromKeyedServices]`:

```csharp
app.MapGet("/data", ([FromKeyedServices("session-cache")] IConnectionMultiplexer sessionCache) =>
{
    // Use the session-specific Redis instance
});
```

## Client Integrations in Non-.NET Services

Client integrations are a .NET concept (DI registration). Non-.NET services use their language's native client libraries with the connection string from the environment:

```python
# Python — no client integration, just env vars
import os, redis
r = redis.from_url(f"redis://{os.environ['ConnectionStrings__cache']}")
```

```typescript
// TypeScript — same pattern
import Redis from 'ioredis';
const redis = new Redis(process.env.ConnectionStrings__cache!);
```

The key insight: **hosting integrations** work in the AppHost (any language). **Client integrations** work in .NET services only. Other languages use native clients with injected connection strings.
