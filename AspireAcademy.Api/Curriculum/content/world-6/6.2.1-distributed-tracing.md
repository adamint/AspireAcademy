# Distributed Tracing

Distributed tracing lets you follow a single request as it crosses service boundaries. Aspire configures tracing automatically via OpenTelemetry.

---

## How It Works

> 📝 **Trace ID** — a unique identifier assigned to every incoming request. All work done for that request across all services shares the same trace ID.

> 📝 **Span** — a single unit of work within a trace (e.g., an HTTP handler, a database query, a message publish). Spans link together via parent references to form a tree.

Every request gets a trace ID. Each unit of work within that request gets a span. Spans link together across services via the **W3C TraceContext** header.

```
User Request                trace-id: abc123
    ↓
Frontend (TypeScript)       span: "GET /search" (parent: none)
    ↓ HTTP call
API (C#)                    span: "GET /api/search" (parent: frontend span)
    ↓ SQL query
PostgreSQL                  span: "SELECT * FROM articles" (parent: API span)
    ↓ HTTP call
ML Service (Python)         span: "POST /recommend" (parent: API span)
```

> 💡 The Dashboard shows this as a **waterfall timeline** — you see exactly where time is spent. Long spans immediately reveal bottlenecks.

> 🐛 **Common Gotcha:** If you see disconnected spans in the Dashboard (same request but separate traces), check that W3C TraceContext headers are being propagated. Custom HTTP clients or reverse proxies can strip the `traceparent` header.

## Adding Custom Spans by Language

### C# — Activity and ActivitySource

```csharp
private static readonly ActivitySource s_source = new("Bazaar.Api");

app.MapGet("/search", async (string query) =>
{
    using var activity = s_source.StartActivity("SearchArticles");
    activity?.SetTag("search.query", query);
    // ... perform search
    activity?.SetTag("search.results", results.Count);
    return results;
});
```

### Python — OpenTelemetry SDK

```python
from opentelemetry import trace

tracer = trace.get_tracer("bazaar.ml")

@app.post("/recommend")
async def recommend(request: Request):
    with tracer.start_as_current_span("compute_recommendations") as span:
        span.set_attribute("input.count", len(items))
        recommendations = compute(items)
        span.set_attribute("output.count", len(recommendations))
        return recommendations
```

### TypeScript — OpenTelemetry API

```typescript
import { trace } from '@opentelemetry/api';

const tracer = trace.getTracer('bazaar.frontend');

async function fetchSearch(query: string) {
    const span = tracer.startSpan('fetchSearch');
    span.setAttribute('search.query', query);
    const result = await fetch(`/api/search?q=${query}`);
    span.end();
    return result;
}
```

### Go — OpenTelemetry SDK

```go
tracer := otel.Tracer("bazaar.worker")

func processMessage(ctx context.Context, msg Message) {
    ctx, span := tracer.Start(ctx, "processMessage")
    defer span.End()
    span.SetAttributes(attribute.String("message.id", msg.ID))
    // ... process
}
```

## Automatic vs Manual Instrumentation

| What's automatic (via ServiceDefaults) | What you add manually |
|---|---|
| HTTP calls (`HttpClient`, `fetch`, `requests`) | Business-specific spans (`SearchArticles`, `compute_recommendations`) |
| Database queries (with client integrations) | Custom attributes (`search.query`, `results.count`) |
| Message broker operations | Domain-level context |
| gRPC calls | Error tagging and status codes |

---

> 📋 **Key Takeaways**
> - Every request gets a trace ID; each unit of work is a span — together they form a tree
> - Aspire automatically instruments HTTP, database, and messaging spans via ServiceDefaults
> - Custom spans add business context — name them after the operation, tag them with domain-specific attributes
> - All four languages (C#, Python, TypeScript, Go) use the same OpenTelemetry model
