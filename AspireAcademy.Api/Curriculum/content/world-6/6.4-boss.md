# Boss Challenge: Test Architect

Time to prove your testing mastery. In this challenge, you will write a complete integration test suite for an AppHost that orchestrates an API service backed by PostgreSQL.

## The Scenario

Your AppHost defines a `CatalogApi` project that connects to a PostgreSQL database named `catalogdb`. The API exposes CRUD endpoints at `/api/products` for managing a product catalog. Your job is to write integration tests that verify the entire stack works end-to-end.

## Requirements

**Setup and Teardown:** Implement `IAsyncLifetime` to manage the `DistributedApplication` lifecycle. Use `DistributedApplicationTestingBuilder.CreateAsync<Projects.MyAppHost>()` to create the builder, build the application, and start it during `InitializeAsync`. Dispose the application in `DisposeAsync`.

**Readiness Checks:** Before running any assertions, wait for the API service and database to be healthy. Do not rely on arbitrary delays — use structured readiness checks to ensure resources are fully initialized.

**HTTP Integration Tests:** Use `app.CreateHttpClient("catalogapi")` to obtain a pre-configured HTTP client. Write at least one test that performs a POST to create a product, then a GET to retrieve it, asserting that the created product is returned correctly.

**Response Validation:** Verify HTTP status codes with `EnsureSuccessStatusCode()`. Deserialize response bodies and assert on specific property values, not just that the response was successful.

## Success Criteria

Your test class must create and start the distributed application, use `CreateHttpClient` for HTTP communication, perform write and read operations with meaningful assertions, and properly dispose all resources on completion. The tests should be reliable and free from timing-dependent failures.

Think about what you learned regarding fixtures, port randomization, and readiness patterns — they all apply here.
