# aspire-manifest.json

The manifest format — resource definitions, connection strings, bindings, and expression syntax.

The **Aspire manifest** (`aspire-manifest.json`) is the serialized representation of your entire distributed application. It's the bridge between your code-first app model and deployment targets like Docker Compose or Kubernetes.

## Structure

The manifest is a JSON file with a top-level `resources` object. Each key is a resource name, and the value describes its type and configuration:

```json
{
  "resources": {
    "cache": {
      "type": "container.v0",
      "image": "docker.io/library/redis:7.4",
      "bindings": {
        "tcp": {
          "scheme": "tcp",
          "protocol": "TCP",
          "transport": "tcp",
          "targetPort": 6379
        }
      }
    },
    "catalogdb": {
      "type": "parameter.v0",
      "connectionString": "{cache.bindings.tcp.host}:{cache.bindings.tcp.port}"
    },
    "webfrontend": {
      "type": "project.v0",
      "path": "../WebFrontend/WebFrontend.csproj",
      "env": {
        "ConnectionStrings__cache": "{cache.connectionString}"
      }
    }
  }
}
```

## Resource Types

Each resource has a versioned type string:

- **`project.v0`**: A .NET project resource with a path to its `.csproj`
- **`container.v0`**: A container resource with an image reference
- **`parameter.v0`**: A parameter value (secrets, connection strings)
- **`value.v0`**: A computed value derived from expressions

## Bindings

Bindings describe the network endpoints a resource exposes:

```json
"bindings": {
  "http": {
    "scheme": "http",
    "protocol": "TCP",
    "transport": "http",
    "targetPort": 8080
  }
}
```

Each binding has a scheme, protocol, transport, and target port. The actual host and port are resolved at deployment time.

## Generating the Manifest

Run `aspire publish` to generate the manifest, or use the `publish-manifest` pipeline step programmatically. The manifest is the input for all downstream deployment generators.
