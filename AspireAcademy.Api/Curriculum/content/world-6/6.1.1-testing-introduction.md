# Introduction to Testing with Aspire

Why integration testing matters for distributed applications, and what Aspire provides to make it practical.

## The Challenge of Testing Distributed Apps

Distributed applications are hard to test. A single request might flow through a web frontend, an API service, a database, and a message broker. Unit tests can verify individual components, but they can't catch issues that only appear when services interact — mismatched connection strings, serialization bugs, broken service discovery, or race conditions during startup.

Traditional integration testing for distributed apps requires manually spinning up infrastructure (databases, caches, message brokers), configuring connection strings, starting services in the right order, and tearing everything down afterward. This is fragile, slow, and often skipped entirely.

## What Aspire Provides

The `Aspire.Hosting.Testing` package solves this by letting you reuse your AppHost as the test fixture. Since your AppHost already defines all the resources, connections, and dependencies your app needs, Aspire can spin up the entire application graph in a test — containers included.

| Feature | Description |
| --- | --- |
| `DistributedApplicationTestingBuilder` | Creates a test-aware version of your AppHost |
| `DistributedApplication` | The running app instance with `StartAsync()` / `StopAsync()` |
| `CreateHttpClient()` | Gets an `HttpClient` pre-configured to talk to a resource |
| `GetConnectionStringAsync()` | Retrieves connection strings for running resources |
| `ResourceNotificationService` | Waits for resources to reach healthy/ready states |

## The Testing Workflow

Every Aspire integration test follows the same pattern:

```
Create builder → Build → Start → Test → Stop
```

Here's the minimal version in code:

```csharp
// 1. Create a testing builder pointing at your AppHost
var builder = await DistributedApplicationTestingBuilder.CreateAsync<Projects.MyAppHost>();

// 2. Build the distributed application
await using var app = await builder.BuildAsync();

// 3. Start all resources (containers, projects, etc.)
await app.StartAsync();

// 4. Wait for a resource to be ready
await app.ResourceNotifications.WaitForResourceHealthyAsync("webfrontend");

// 5. Test against the running application
var httpClient = app.CreateHttpClient("webfrontend");
var response = await httpClient.GetAsync("/");
response.EnsureSuccessStatusCode();

// 6. Cleanup happens automatically via `await using`
```

## Installing the Package

Add `Aspire.Hosting.Testing` to your test project:

```dotnetcli
dotnet add package Aspire.Hosting.Testing
```

Your test project also needs a project reference to your AppHost so the testing builder can locate it:

```xml
<ItemGroup>
  <ProjectReference Include="..\MyApp.AppHost\MyApp.AppHost.csproj" />
</ItemGroup>
```

## What Happens Under the Hood

When you call `DistributedApplicationTestingBuilder.CreateAsync<TEntryPoint>()`, Aspire:

1. **Loads your AppHost** — it runs your `Program.cs` to discover all resource definitions
2. **Suspends execution** — the builder pauses before the app starts, letting you modify resources
3. **Randomizes ports** — test resources get random ports to avoid conflicts with other tests or local dev
4. **Manages lifecycle** — containers start, health checks run, and everything tears down on disposal

This means your tests run against real infrastructure. If your app uses PostgreSQL, the test spins up a real PostgreSQL container. If it uses Redis, a real Redis container starts. No mocks, no fakes — the real thing.

## When to Use Integration Tests

Integration tests with Aspire are best for verifying:

- **End-to-end request flows** — does a request through the frontend reach the API and database?
- **Service discovery** — can services find and communicate with each other?
- **Resource connectivity** — do connection strings work? Are health checks passing?
- **Startup behavior** — does the application start successfully with all dependencies?

They complement (but don't replace) unit tests. Use unit tests for business logic; use Aspire integration tests for verifying that your distributed system works as a whole.
