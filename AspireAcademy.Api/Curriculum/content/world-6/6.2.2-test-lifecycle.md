# Test Resource Lifecycle

Managing the lifecycle of Aspire resources in tests — from startup through teardown — and best practices for reliable test execution in local and CI environments.

## How Async Disposal Works

`DistributedApplication` implements `IAsyncDisposable`. When you dispose it, Aspire:

1. Sends stop signals to all running projects and executables
2. Stops and removes all containers it created
3. Releases allocated ports
4. Cleans up temporary files and volumes

### Disposal in a Single Test

For one-off tests, use `await using` to scope the application to the test method:

```csharp
[Fact]
public async Task QuickSmokeTest()
{
    var builder = await DistributedApplicationTestingBuilder
        .CreateAsync<Projects.MyApp_AppHost>();

    await using var app = await builder.BuildAsync();
    await app.StartAsync();
    await app.ResourceNotifications.WaitForResourceHealthyAsync("api");

    var client = app.CreateHttpClient("api");
    var response = await client.GetAsync("/health");

    response.EnsureSuccessStatusCode();
    // app is disposed here — containers stop, ports are freed
}
```

### Disposal in a Test Class

When sharing the application across multiple test methods, use `IAsyncLifetime`:

```csharp
public class ApiTests : IAsyncLifetime
{
    private DistributedApplication _app = null!;

    public async Task InitializeAsync()
    {
        var builder = await DistributedApplicationTestingBuilder
            .CreateAsync<Projects.MyApp_AppHost>();

        _app = await builder.BuildAsync();
        await _app.StartAsync();
        await _app.ResourceNotifications.WaitForResourceHealthyAsync("api");
    }

    public async Task DisposeAsync()
    {
        await _app.DisposeAsync();
    }

    [Fact]
    public async Task GetProducts() { /* ... */ }

    [Fact]
    public async Task CreateProduct() { /* ... */ }
}
```

All test methods in the class run against the same application instance. The application starts once in `InitializeAsync` and stops once in `DisposeAsync`.

## Avoiding Port Conflicts

Aspire's testing builder automatically randomizes ports for all resources. This means multiple test classes can run in parallel without port collisions — each gets its own set of ports.

However, if you're running tests alongside a local `dotnet run` of the same AppHost, you might still see conflicts. The testing builder's port randomization only applies to the test instance.

### Parallel Test Execution

xUnit runs test classes in parallel by default. Because each class creates its own `DistributedApplication` with randomized ports, parallel execution works out of the box:

```
Test Class A → AppHost instance 1 (ports 54001, 54002, ...)
Test Class B → AppHost instance 2 (ports 54010, 54011, ...)
Test Class C → AppHost instance 3 (ports 54020, 54021, ...)
```

If you need to limit parallelism (e.g., to reduce resource usage), use xUnit's `[Collection]` attribute to serialize certain test classes:

```csharp
[Collection("Sequential")]
public class HeavyResourceTests : IAsyncLifetime
{
    // These tests run one at a time
}
```

## Using CancellationTokens for Timeouts

### Startup Timeout

Protect against indefinite hangs by passing a `CancellationToken` to startup methods:

```csharp
public async Task InitializeAsync()
{
    var builder = await DistributedApplicationTestingBuilder
        .CreateAsync<Projects.MyApp_AppHost>();

    _app = await builder.BuildAsync();

    using var cts = new CancellationTokenSource(TimeSpan.FromMinutes(3));

    await _app.StartAsync(cts.Token);
    await _app.ResourceNotifications
        .WaitForResourceHealthyAsync("webfrontend", cts.Token);
}
```

If startup takes longer than three minutes, the `CancellationToken` fires, `OperationCanceledException` is thrown, and the test fails with a clear timeout instead of hanging.

### Per-Test Timeout

You can also add timeouts to individual test methods:

```csharp
[Fact]
public async Task ApiRespondsWithinTimeout()
{
    using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(30));
    var client = _app.CreateHttpClient("api");

    var response = await client.GetAsync("/api/products", cts.Token);
    response.EnsureSuccessStatusCode();
}
```

## Best Practices for CI/CD

### Container Image Caching

The first test run on a CI agent pulls container images (PostgreSQL, Redis, etc.), which can be slow. Strategies to mitigate this:

- **Cache Docker layers** — most CI systems support Docker layer caching
- **Use smaller images** — prefer Alpine-based images where available
- **Set generous timeouts** — first-run timeouts should account for image pulls

### Resource Cleanup

Disposal normally cleans up everything, but if a test crashes or the CI agent terminates unexpectedly, containers may be left running. Mitigations:

- **Container runtime cleanup** — configure your CI to prune containers between runs
- **Unique naming** — Aspire uses unique names for test containers, reducing collision risk

### Memory and CPU

Each `DistributedApplication` starts real containers and processes. On resource-constrained CI agents:

- **Limit parallel test classes** — reduce xUnit's `maxParallelThreads` in `xunit.runner.json`
- **Share app instances** — use `IAsyncLifetime` to share one app across many test methods
- **Test what matters** — not every test needs the full application graph

```json
{
  "$schema": "https://xunit.net/schema/current/xunit.runner.schema.json",
  "maxParallelThreads": 2
}
```

## Lifecycle Summary

The following shows the complete lifecycle of resources during a test run:

| Phase | What Happens | Your Code |
| --- | --- | --- |
| Create | AppHost loads, resources discovered | `CreateAsync<T>()` |
| Configure | Modify resources, override config | Set `builder.Configuration`, `builder.Services` |
| Build | Application graph finalized | `builder.BuildAsync()` |
| Start | Containers launch, projects compile and run | `app.StartAsync()` |
| Wait | Health checks pass, resources become ready | `WaitForResourceHealthyAsync()` |
| Test | Interact with running resources | `CreateHttpClient()`, `GetConnectionStringAsync()` |
| Stop | Signals sent to stop all resources | `app.StopAsync()` or `app.DisposeAsync()` |
| Dispose | Containers removed, ports freed, cleanup | `await using` or `DisposeAsync()` |

## Handling Flaky Infrastructure

Containers occasionally fail to start on the first attempt, especially in CI. Consider retry logic at the test fixture level:

```csharp
public async Task InitializeAsync()
{
    var retries = 3;
    for (var i = 0; i < retries; i++)
    {
        try
        {
            var builder = await DistributedApplicationTestingBuilder
                .CreateAsync<Projects.MyApp_AppHost>();
            _app = await builder.BuildAsync();

            using var cts = new CancellationTokenSource(TimeSpan.FromMinutes(3));
            await _app.StartAsync(cts.Token);
            await _app.ResourceNotifications
                .WaitForResourceHealthyAsync("api", cts.Token);
            return;
        }
        catch (Exception) when (i < retries - 1)
        {
            if (_app is not null)
            {
                await _app.DisposeAsync();
            }
        }
    }
}
```

Use this pattern sparingly — it's a safety net for infrastructure instability, not a substitute for fixing underlying issues.

## Key Takeaways

- Always dispose `DistributedApplication` — use `await using` or `IAsyncLifetime`
- Port randomization makes parallel test execution safe by default
- Use `CancellationToken` timeouts to prevent indefinite hangs
- In CI, account for container image pull times and resource constraints
- Share application instances across test methods in a class to reduce startup overhead
