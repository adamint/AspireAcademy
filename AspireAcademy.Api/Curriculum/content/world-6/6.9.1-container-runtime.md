# Container Runtime Abstraction

Aspire doesn't hardcode Docker — it abstracts container operations behind `IContainerRuntime`, defined at `src/Aspire.Hosting/Publishing/IContainerRuntime.cs`. This enables support for Docker, Podman, and potentially other container runtimes.

## The IContainerRuntime Interface

The interface exposes the core operations any container runtime must support:

- **`Name`** — Returns "docker" or "podman"
- **`CheckIfRunningAsync()`** — Verifies the runtime daemon is available
- **`BuildImageAsync()`** — Builds a container image from a Dockerfile
- **`TagImageAsync()`** — Tags an image for registry pushing
- **`PushImageAsync()`** — Pushes an image to a container registry
- **`LoginToRegistryAsync()`** — Authenticates with a container registry

## How DcpHost Uses It

During startup (see Module 6.7.1), `DcpHost.StartAsync()` queries `IContainerRuntime.CheckIfRunningAsync()` to detect which container runtime is available. If neither Docker nor Podman is running, Aspire reports a clear error. The `aspire doctor` command also uses this interface for environment diagnostics.

## Container Networking

When DcpExecutor creates containers, it configures them on a shared Docker network. **DNS aliases** enable inter-container service discovery — a container named `postgres` is reachable at hostname `postgres` from other containers on the same network. This is how your Redis container can connect to your Postgres container by name.

## Port Mapping

Container ports follow a two-layer model:

1. **Container port** — The port inside the container (e.g., 5432 for Postgres)
2. **Host port** — The port on your machine, allocated by `PortAllocator` (from `src/Aspire.Hosting/ApplicationModel/PortAllocator.cs`)

DcpExecutor maps host→container ports so your .NET projects (running outside containers) can connect via `localhost:{hostPort}`, while container-to-container traffic uses DNS aliases and container ports directly.
