# Container Image Building and Pushing

How the Build and Push pipeline steps create and distribute container images.

Before your Aspire app can run on Docker Compose or Kubernetes, project resources need to be built into container images and pushed to a registry. The pipeline's **build** and **push** steps handle this automatically.

## The Build Step

The build step creates container images for each project resource. .NET projects use the SDK's built-in container support:

```bash
# What the build step effectively runs for each project resource
dotnet publish /t:PublishContainer -p ContainerImageName=catalog-api
```

This produces a local container image tagged with the project name. No Dockerfile is needed — the .NET SDK generates an optimized container image directly.

## The Push Step

After images are built locally, the push step uploads them to a container registry:

```bash
aspire publish --output-path ./output \
    --publisher docker-compose \
    --container-registry myregistry.azurecr.io
```

The registry URL is applied to all project images:

```
catalog-api:latest → myregistry.azurecr.io/catalog-api:latest
```

## Image Naming and Tagging

By default, images are named after their resource name. You can customize the image name and tag:

```csharp
var api = builder.AddProject<Projects.CatalogApi>("api")
    .WithContainerImage("my-catalog-api", "v1.2.3");
```

## Container Resources vs Project Resources

- **Container resources** (e.g., `AddRedis`) already have images — they skip the build step entirely
- **Project resources** (e.g., `AddProject<T>`) need to be built into images

## Build Context

The build step uses the project's directory as the build context. It reads container configuration from the project file:

```xml
<PropertyGroup>
  <ContainerRepository>catalog-api</ContainerRepository>
  <ContainerImageTag>latest</ContainerImageTag>
</PropertyGroup>
```

## Pipeline Dependencies

The build and push steps have a strict ordering:

```
publish-manifest → build → push → generate-compose/k8s
```

The generate step needs to know final image names (including registry prefix), which is why it runs after push.
