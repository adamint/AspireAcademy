# Volumes and Bind Mounts

Containers are ephemeral by default — data is lost when they stop. Volumes and bind mounts provide persistence.

## Named Volumes with WithVolume

`WithVolume()` creates a Docker-managed named volume:

```csharp
var postgres = builder.AddContainer("db", "postgres:16")
    .WithVolume("pgdata", "/var/lib/postgresql/data");
```

Named volumes are managed by the container runtime. They persist across container restarts and are portable between machines. The first argument is the volume name; the second is the mount path inside the container.

You can also create anonymous volumes (auto-named by Aspire) by omitting the name:

```csharp
var db = builder.AddContainer("db", "postgres:16")
    .WithVolume("/var/lib/postgresql/data");
```

Add `isReadOnly: true` for read-only mounts:

```csharp
.WithVolume("config-vol", "/etc/app/config", isReadOnly: true)
```

## Bind Mounts with WithBindMount

`WithBindMount()` maps a host directory directly into the container:

```csharp
var nginx = builder.AddContainer("web", "nginx:latest")
    .WithBindMount("./nginx.conf", "/etc/nginx/nginx.conf", isReadOnly: true);
```

The source path is relative to the AppHost project directory. Bind mounts reflect host file changes immediately — ideal for development configs.

## Choosing Between Them

| Feature | WithVolume | WithBindMount |
|---------|-----------|---------------|
| Managed by | Container runtime | Host filesystem |
| Portable | Yes | No (host-dependent) |
| Performance | Optimized | Host-dependent |
| Best for | Database data, persistent state | Config files, dev-time assets |

## Key Points

- Use `WithVolume()` for persistent data (databases, caches)
- Use `WithBindMount()` for config files and development assets
- Both support `isReadOnly` for safety
- Volume names should be descriptive and consistent across restarts
