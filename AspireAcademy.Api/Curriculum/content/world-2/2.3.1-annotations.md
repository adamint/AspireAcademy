# Annotations: Behavior Without Inheritance

Instead of creating subclasses for every combination of features, Aspire uses **annotations** — small metadata objects attached to a resource's `Annotations` collection. The orchestrator and publishers inspect these annotations at runtime.

---

## The Interface

```csharp
public interface IResourceAnnotation { }
```

A marker interface. Any class implementing it can be attached to any resource.

## How Annotations Are Added

Most annotations are added by extension methods. When you call:

```csharp
builder.AddContainer("nginx", "nginx")
       .WithHttpEndpoint(port: 80);
```

The `WithHttpEndpoint` method creates an `EndpointAnnotation` and adds it to the container's `Annotations` collection.

## Why This Matters

Annotations make Aspire **extensible without modifying base classes**. To add a new capability:

1. Define an annotation class implementing `IResourceAnnotation`
2. Add it to resources via an extension method
3. Read it in a pipeline step, lifecycle hook, or custom publisher

This is how every hosting integration works. `Aspire.Hosting.Redis` adds Redis-specific annotations to a `ContainerResource` — health checks, connection string format, default ports — without subclassing `ContainerResource`.

The pattern is language-agnostic. When a TypeScript AppHost calls `cache.withLifetime(ContainerLifetime.Persistent)`, the RemoteHost bridge adds a `ContainerLifetimeAnnotation` to the resource on the .NET side.
