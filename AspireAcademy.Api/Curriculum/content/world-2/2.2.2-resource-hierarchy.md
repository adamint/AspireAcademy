# The Resource Type Hierarchy

While every resource implements `IResource`, there are concrete base classes for common resource categories.

---

## The Class Tree

```
IResource
├── Resource (abstract base)
│   ├── ContainerResource     — Docker/Podman containers
│   ├── ProjectResource       — .NET project references
│   ├── ExecutableResource    — Local processes (Go, Python, etc.)
│   └── ParameterResource    — Configuration parameters and secrets
```

### Resource

The abstract base class that most resources inherit from:

```csharp
public abstract class Resource : IResource
{
    public string Name { get; }
    public ResourceAnnotationCollection Annotations { get; }
}
```

### ContainerResource

For Docker or Podman containers. Added by `AddContainer()` and by hosting integrations like `AddRedis()`, `AddPostgres()`, etc.

```csharp
var nginx = builder.AddContainer("proxy", "nginx", "1.27");
```

### ProjectResource

For .NET projects. Added by `AddProject<T>()`. The generic type parameter `T` comes from generated project references.

```csharp
var api = builder.AddProject<Projects.BazaarApi>("api");
```

### ExecutableResource

For local processes — anything that runs as a command. Go, Python, Ruby, or any executable.

```csharp
var worker = builder.AddExecutable("worker", "go", workingDir, "run", ".");
```

### ParameterResource

For configuration values — secrets, API keys, feature flags.

```csharp
var apiKey = builder.AddParameter("api-key", secret: true);
```

> 🌍 **Real World:** In production Aspire apps, it's common to see a mix of all four resource types in a single AppHost — .NET projects for APIs, containers for databases, executables for polyglot services, and parameters for secrets. The hierarchy is designed to cover these real-world patterns.

## Hosting Integration Resources

Hosting integrations often define their own resource types that inherit from `ContainerResource`:

- `RedisResource` — from Aspire.Hosting.Redis
- `PostgresServerResource` — from Aspire.Hosting.PostgreSQL
- `RabbitMQServerResource` — from Aspire.Hosting.RabbitMQ

These add integration-specific behavior (connection strings, health checks) while inheriting container lifecycle management.
