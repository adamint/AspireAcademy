# Port Allocation and Proxying

Aspire's port allocation and proxying system prevents port conflicts and enables features like load balancing for replicas.

## Automatic Port Allocation

When you don't specify a port, Aspire finds an available port on the host:

```csharp
// Both port and targetPort are auto-allocated
var api = builder.AddProject<Projects.ApiService>("api")
    .WithHttpEndpoint(name: "api");
```

This eliminates the "port already in use" errors that plague local development. Each resource gets unique ports, even when running multiple AppHosts.

## Port vs TargetPort

These two concepts are often confused:

- **`Port`**: The external-facing port on the **proxy**. This is what consumers connect to.
- **`TargetPort`**: The port the **application** listens on inside the resource.

```csharp
.WithHttpEndpoint(port: 8080, targetPort: 5000)
// Consumers connect to :8080 → proxy forwards to :5000
```

When you specify only `targetPort`, the proxy port is auto-allocated. When you specify only `port`, both are set to the same value.

## The Aspire Reverse Proxy

By default (`IsProxied = true`), traffic flows through Aspire's built-in reverse proxy:

```
Consumer → Proxy (:8080) → App (:5000)
```

The proxy provides:
- **Load balancing**: Distributes requests across replicas
- **Port mapping**: Decouples external ports from internal ports
- **Health awareness**: Routes only to healthy instances

## Bypassing the Proxy

Set `isProxied: false` when you need direct access:

```csharp
.WithHttpEndpoint(port: 5000, isProxied: false)
```

Without the proxy, `Port` and `TargetPort` must match. The app binds directly to the external port. Load balancing and port mapping are unavailable.

## Key Points

- Auto-allocation prevents port conflicts across resources
- `Port` = proxy/external; `TargetPort` = app/internal
- The proxy enables load balancing, health-aware routing, and port decoupling
- `isProxied: false` bypasses the proxy for direct access
