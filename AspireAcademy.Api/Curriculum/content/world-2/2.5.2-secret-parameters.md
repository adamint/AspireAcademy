# Secret Parameters

Secret parameters add security to your Aspire configuration. Values are masked in the dashboard and logs, making them safe for passwords, API keys, and tokens.

## Creating a Secret Parameter

```csharp
var dbPassword = builder.AddParameter("db-password", secret: true);
```

The `secret: true` flag tells Aspire to treat this value as sensitive. The behavior is identical to a regular parameter — it resolves from `Parameters:db-password` in configuration — but with additional protections.

## What Secret Does

- **Dashboard masking**: The value is displayed as `●●●●●●●●` in the dashboard
- **Log redaction**: The value is not written to logs
- **User secrets integration**: During development, store in .NET user secrets

## Storing Secrets in Development

Use .NET user secrets for the AppHost project:

```bash
dotnet user-secrets set "Parameters:db-password" "SuperSecret123!"
```

This keeps sensitive values out of `appsettings.json` and source control.

## Using Secret Parameters

Wire them into resources just like regular parameters:

```csharp
var password = builder.AddParameter("db-password", secret: true);

builder.AddContainer("db", "postgres:16")
    .WithEnvironment("POSTGRES_PASSWORD", password);

// The actual value is injected, but never displayed in the UI
```

## Combining with Regular Parameters

```csharp
var dbHost = builder.AddParameter("db-host");           // visible
var dbPassword = builder.AddParameter("db-password", secret: true); // masked

builder.AddProject<Projects.ApiService>("api")
    .WithEnvironment("DB_HOST", dbHost)
    .WithEnvironment("DB_PASSWORD", dbPassword);
```

In the dashboard, `DB_HOST` shows its value; `DB_PASSWORD` shows `●●●●●●●●`.

## Key Points

- `secret: true` masks the value in dashboard and logs
- Store secrets in user secrets during development, not in `appsettings.json`
- The actual value is still injected into the resource's environment at runtime
- Both secret and non-secret parameters use the same `Parameters:{name}` configuration path
