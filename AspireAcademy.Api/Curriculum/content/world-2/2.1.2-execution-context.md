# ExecutionContext: Run vs Publish

The builder has an `ExecutionContext` that tells you whether Aspire is running your application locally or generating deployment artifacts. This distinction matters more than you might think — it's the seam between your local development environment and your production infrastructure.

---

## Two Modes

| Mode | Activated By | Purpose |
|---|---|---|
| **Run mode** | `aspire run` | Start DCP, run services locally, open Dashboard |
| **Publish mode** | `aspire publish` | Generate artifacts (Compose, K8s, Bicep) without starting anything |

## Checking the Mode

```csharp
var builder = DistributedApplication.CreateBuilder(args);

if (builder.ExecutionContext.IsRunMode)
{
    // Only when running locally
    builder.AddContainer("mailhog", "mailhog/mailhog")
        .WithHttpEndpoint(port: 8025, targetPort: 8025);
}

if (builder.ExecutionContext.IsPublishMode)
{
    // Only when publishing
    // e.g., use a managed Azure service instead of a container
}
```

## When to Branch

Common patterns:
- **Dev-only services**: Add mailhog, pgAdmin, or Redis Commander only in run mode
- **Cloud vs local infrastructure**: Use a container in run mode, an Azure managed service in publish mode
- **Seeding**: Run database seed scripts only in run mode

Most of the time, you do not need to check the execution context. The hosting integration packages and publishing pipeline handle the differences automatically. But when you need to customize behavior for a specific environment, `ExecutionContext` is the tool.

---

> 📋 **Key Takeaways**
> - `ExecutionContext.IsRunMode` = local development, `IsPublishMode` = generating deployment artifacts
> - Use it to add dev-only tools (pgAdmin, mailhog) or switch between local containers and managed cloud services
> - Most integrations handle the run/publish distinction automatically — you only branch when you need to override
