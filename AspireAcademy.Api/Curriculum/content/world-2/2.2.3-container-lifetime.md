# Container Lifetime Management

By default, containers start and stop with the AppHost. `WithLifetime()` lets you change this behavior for containers that should persist across restarts.

## ContainerLifetime Enum

Aspire defines two lifetime modes:

- **`ContainerLifetime.Session`** (default): The container starts when the AppHost starts and stops when the AppHost stops. Every restart creates a fresh container.
- **`ContainerLifetime.Persistent`**: The container survives AppHost restarts. If a container with the same name already exists and is running, Aspire reuses it instead of creating a new one.

## Using WithLifetime

```csharp
var postgres = builder.AddContainer("db", "postgres:16")
    .WithVolume("pgdata", "/var/lib/postgresql/data")
    .WithLifetime(ContainerLifetime.Persistent);
```

When you restart the AppHost, the `db` container remains running with its data intact. No restart delay, no data loss.

## When to Use Persistent Lifetime

Persistent containers are ideal for:

- **Databases**: Avoid re-seeding data on every AppHost restart
- **Caches**: Preserve cached data during development iterations
- **Message brokers**: Keep queued messages across restarts
- **Containers with slow startup**: Skip initialization time

## Session Lifetime Is the Safe Default

Session lifetime ensures clean state on every restart. This is important when:

- Testing requires a fresh environment
- Container configuration changed and needs to be reapplied
- You want reproducible startup behavior

## Key Points

- Default is `ContainerLifetime.Session` — containers stop with the AppHost
- Persistent containers are reused if already running with the same name
- Combine persistent lifetime with volumes for durable development databases
- The dashboard shows persistent containers as "Existing" when reused
