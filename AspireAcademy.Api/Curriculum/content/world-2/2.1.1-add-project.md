# Adding Projects with AddProject\<T\>

`AddProject<TProject>()` is the primary way to add .NET projects to your Aspire app model. The type parameter `TProject` refers to a class auto-generated in the `Projects` namespace from your AppHost's `<ProjectReference>` items.

## How It Works

When you add a `<ProjectReference>` to your AppHost `.csproj`, the Aspire SDK generates a metadata class (e.g., `Projects.ApiService`) implementing `IProjectMetadata`. This class contains the path to the referenced project, allowing Aspire to locate the binary, apply launch settings, and manage the process lifecycle.

```csharp
var builder = DistributedApplication.CreateBuilder(args);

var api = builder.AddProject<Projects.ApiService>("apiservice");

builder.Build().Run();
```

The string `"apiservice"` is the **resource name** — it identifies this resource in the dashboard, logs, and environment variable injection. Choose a name that's lowercase and descriptive.

## What AddProject Returns

`AddProject<TProject>()` returns an `IResourceBuilder<ProjectResource>`, enabling fluent chaining:

```csharp
var api = builder.AddProject<Projects.ApiService>("apiservice")
    .WithExternalHttpEndpoints()
    .WithReplicas(2);
```

## Non-Generic Overload

If you don't have a project reference (e.g., the project path is dynamic), use the non-generic overload with an explicit path:

```csharp
var api = builder.AddProject("apiservice", "../ApiService/ApiService.csproj");
```

## Key Points

- `TProject` must implement `IProjectMetadata` — this is auto-generated from project references
- The resource name becomes the DNS alias and environment variable prefix
- The returned builder supports all `IResourceWithEndpoints` and `IResourceWithEnvironment` extensions
- Aspire discovers endpoints from `launchSettings.json` by default
