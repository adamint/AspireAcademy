# AddCSharpApp — File and Path-Based C# Apps

Use AddCSharpApp(name, path) to add C# applications by path instead of project reference.

> ⚠️ **Experimental API** — `AddCSharpApp` is marked with `[Experimental("ASPIRECSHARPAPPS001")]`. You must suppress this diagnostic to use it: add `<NoWarn>ASPIRECSHARPAPPS001</NoWarn>` to your project or use `#pragma warning disable ASPIRECSHARPAPPS001`.

## The API

```csharp
[Experimental("ASPIRECSHARPAPPS001")]
public static IResourceBuilder<ProjectResource> AddCSharpApp(
    this IDistributedApplicationBuilder builder,
    string name,
    string path)
```

The `path` parameter can be:
- A `.csproj` file path
- A `.cs` single-file app path
- A directory containing a project

All paths are resolved relative to the AppHost directory.

## When to Use AddCSharpApp

Use `AddCSharpApp` when you need to reference a C# project that isn't part of your solution:

```csharp
var builder = DistributedApplication.CreateBuilder(args);

// External C# project — no ProjectReference needed
var api = builder.AddCSharpApp("api", "../services/Api/Api.csproj")
                 .WithHttpEndpoint(targetPort: 5000);

// Single-file C# app
var worker = builder.AddCSharpApp("worker", "../tools/worker.cs");

builder.Build().Run();
```

## How It Differs from AddProject

| Feature | `AddProject<T>` | `AddCSharpApp` |
|---------|------------------|----------------|
| Requires `<ProjectReference>` | ✅ Yes | ❌ No |
| Compile-time safety | ✅ Generated types | ❌ String paths |
| Single-file .cs apps | ❌ | ✅ |
| Same fluent API | ✅ | ✅ |

## Same Capabilities

Once created, `AddCSharpApp` returns an `IResourceBuilder` with the same methods:

```csharp
builder.AddCSharpApp("api", "../Api/Api.csproj")
       .WithReference(redis)
       .WithEnvironment("MODE", "prod")
       .WithHttpEndpoint(targetPort: 5000)
       .WaitFor(redis);
```

Under the hood, the orchestrator uses `dotnet run` to launch the app — identical to `AddProject`.
