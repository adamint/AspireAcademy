# AddConnectionString — External Resources

Use AddConnectionString(name, environmentVariableName) to reference pre-existing external resources.

## The Problem

Sometimes your database or service already exists — you don't want Aspire to create it. You just need to pass its connection string to your services.

## The API

```csharp
var externalDb = builder.AddConnectionString("orders-db");
```

This creates a resource that reads from `ConnectionStrings:orders-db` in configuration:

```json
{
  "ConnectionStrings": {
    "orders-db": "Server=production-server;Database=Orders;..."
  }
}
```

## Using It

```csharp
var builder = DistributedApplication.CreateBuilder(args);

// External database — already exists, just pass the connection string
var ordersDb = builder.AddConnectionString("orders-db");

builder.AddProject<Projects.Api>("api")
       .WithReference(ordersDb);

builder.Build().Run();
```

The API service receives `ConnectionStrings__orders-db` as an environment variable, exactly as if Aspire had created the database.

## Custom Environment Variable Name

If the external service uses a non-standard environment variable:

```csharp
var externalDb = builder.AddConnectionString("orders-db",
    environmentVariableName: "DATABASE_URL");
```

## Mixing External and Managed Resources

```csharp
var builder = DistributedApplication.CreateBuilder(args);

// Aspire manages Redis locally
var cache = builder.AddRedis("cache");

// External production database
var db = builder.AddConnectionString("orders-db");

builder.AddProject<Projects.Api>("api")
       .WithReference(cache)    // Aspire-managed
       .WithReference(db);      // External

builder.Build().Run();
```

## In the Manifest

`AddConnectionString` resources appear in the manifest as values that must be provided at deployment time. The deployer supplies the actual connection string for the target environment.
