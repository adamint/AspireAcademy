# CreateBuilder() — The Entry Point

Every Aspire AppHost starts with a builder. In C#, it is `DistributedApplication.CreateBuilder(args)`. In TypeScript, `createBuilder()`. In Python, `create_builder()`. The API surface is the same — the language is different.

---

## C#

```csharp
var builder = DistributedApplication.CreateBuilder(args);
// Add resources...
builder.Build().Run();
```

`CreateBuilder(args)` does the following:
1. Creates a `HostApplicationBuilder` with the standard .NET host (configuration, DI, logging)
2. Registers Aspire-specific services (DCP, resource notification, eventing, health checks)
3. Parses `args` to determine the `ExecutionContext` (run mode vs publish mode)
4. Returns a `IDistributedApplicationBuilder` with methods for adding resources

## TypeScript

```typescript
import { createBuilder } from './.modules/aspire.js';

const builder = await createBuilder();
// Add resources...
await builder.build().run();
```

`createBuilder()` connects to the RemoteHost process over a Unix socket and returns a builder handle. Every method call on `builder` is an RPC to the .NET hosting SDK.

## Python

```python
from aspire_app import create_builder

with create_builder() as builder:
    # Add resources...
    builder.run()
```

Python uses a context manager. `create_builder()` returns a builder that connects to the RemoteHost. When the `with` block exits, cleanup runs automatically.

## The Builder Pattern

Regardless of language, the builder follows the same pattern:

1. **Create** the builder
2. **Add resources** — databases, caches, services, containers
3. **Wire resources** — `WithReference()`, `WaitFor()`
4. **Build and run** — passes the model to DCP for execution

The builder is the root of the **app model** — the declarative graph you are constructing.
