# CreateBuilder() — The Entry Point

Every Aspire AppHost starts with a builder. In C#, it is `DistributedApplication.CreateBuilder(args)`. In TypeScript, `createBuilder()`. In Python, `create_builder()`. The API surface is the same — the language is different.

> 🎯 **By the end of this lesson, you will be able to:**
> - Call the correct builder factory for C#, TypeScript, and Python
> - Explain what `CreateBuilder` does under the hood (host setup, DCP registration, execution context)
> - Describe the four-step builder pattern that every AppHost follows

---

## C#

```csharp
var builder = DistributedApplication.CreateBuilder(args);
// Add resources...
builder.Build().Run();
```

> 🔍 **What `CreateBuilder(args)` does under the hood:**
> 1. Creates a `HostApplicationBuilder` with the standard .NET host (configuration, DI, logging)
> 2. Registers Aspire-specific services (DCP, resource notification, eventing, health checks)
> 3. Parses `args` to determine the `ExecutionContext` (run mode vs publish mode)
> 4. Returns a `IDistributedApplicationBuilder` with methods for adding resources

## TypeScript

```typescript
import { createBuilder } from './.modules/aspire.js';

const builder = await createBuilder();
// Add resources...
await builder.build().run();
```

> 📝 **RemoteHost** — a .NET process that TypeScript and Python AppHosts connect to over a Unix socket. Every method call on the builder is an RPC to the .NET hosting SDK.

## Python

```python
from aspire_app import create_builder

with create_builder() as builder:
    # Add resources...
    builder.run()
```

Python uses a context manager. `create_builder()` returns a builder that connects to the RemoteHost. When the `with` block exits, cleanup runs automatically.

## The Builder Pattern

> 🏗️ **Every AppHost follows four steps:**
> 1. **Create** the builder — factory method returns a configured builder instance
> 2. **Add resources** — databases, caches, services, containers
> 3. **Wire resources** — `WithReference()` for dependencies, `WaitFor()` for startup ordering
> 4. **Build and run** — passes the assembled app model to DCP for execution

The builder is the root of the **app model** — the declarative graph you are constructing.

> 🔧 **Try it:** Create a minimal AppHost in your preferred language that calls `CreateBuilder`, adds nothing, and runs. What does the Dashboard show?

---

> 📋 **Key Takeaways**
> - `CreateBuilder` is the universal entry point — every language has an equivalent factory
> - C# builds a full .NET host; TypeScript and Python connect to a RemoteHost over Unix sockets
> - The builder pattern is always: create → add → wire → run
