# Adding Parameters with AddParameter

`AddParameter()` creates a typed parameter resource that externalizes configuration values. Parameters are read from `IConfiguration` at startup and can be passed to environment variables, connection strings, and other resource builders.

## Basic Usage

```csharp
var apiKey = builder.AddParameter("api-key");
```

This creates a parameter that resolves from `Parameters:api-key` in the configuration hierarchy. In `appsettings.json`:

```json
{
  "Parameters": {
    "api-key": "my-secret-key-123"
  }
}
```

## Using Parameters

Pass parameters to resources via `WithEnvironment()`:

```csharp
var apiKey = builder.AddParameter("api-key");

builder.AddProject<Projects.ApiService>("api")
    .WithEnvironment("API_KEY", apiKey);
```

At runtime, `API_KEY` resolves to the configured value. The parameter acts as an indirection layer — the AppHost defines *what* configuration is needed; the environment defines the *value*.

## Parameter with a Default Value

Provide a static default for development convenience:

```csharp
var region = builder.AddParameter("region", "us-east-1");
```

## The Dashboard Prompts for Missing Parameters

When a parameter has no configured value and no default, the Aspire dashboard prompts you to enter it at startup. This is a safety net — you won't accidentally run with missing configuration.

## Parameters in Resource Configuration

Parameters can flow into container environment variables, build arguments, and composed expressions:

```csharp
var dbUser = builder.AddParameter("db-user");

builder.AddContainer("db", "postgres:16")
    .WithEnvironment("POSTGRES_USER", dbUser);
```

## Key Points

- Parameters read from `Parameters:{name}` in configuration
- Use `WithEnvironment()` to inject parameter values into resources
- Missing parameters are prompted in the dashboard
- Parameters decouple resource definitions from environment-specific values
