# AddParameter — Configurable Values

Use AddParameter(name, secret) to define configurable values that can be set at deployment time.

## The API

```csharp
// Simple parameter (prompted at runtime if not configured)
builder.AddParameter("api-key");

// Secret parameter (masked in Dashboard, stored securely)
builder.AddParameter("db-password", secret: true);

// Parameter with a default value
builder.AddParameter("region", "eastus");
```

## How Parameters Work

Parameters are **not running processes** — they're configuration values. They appear in the Dashboard as resources but don't have a state like Running or Exited.

When the AppHost starts, parameters are resolved from:
1. **Configuration** — `appsettings.json`, environment variables, user secrets
2. **Dashboard prompt** — if not configured, the Dashboard prompts you to enter the value

## Using Parameters

Inject parameters into resources as environment variables:

```csharp
var apiKey = builder.AddParameter("api-key", secret: true);

builder.AddProject<Projects.Api>("api")
       .WithEnvironment("API_KEY", apiKey);
```

Or use them in connection strings:

```csharp
var password = builder.AddParameter("db-password", secret: true);

builder.AddPostgres("pg", password: password);
```

Many hosting integrations accept `IResourceBuilder<ParameterResource>` for credentials.

## Configuration Key

Parameters read from `Parameters:{name}` in configuration:

```json
{
  "Parameters": {
    "api-key": "my-secret-key",
    "db-password": "P@ssw0rd!"
  }
}
```

For local development, use user secrets:
```bash
dotnet user-secrets set "Parameters:api-key" "my-secret-key"
```

## In the Manifest

When publishing, parameters become deployment-time values:

```json
{
  "resources": {
    "api-key": {
      "type": "parameter.v0",
      "value": "{api-key.value}",
      "inputs": {
        "value": { "secret": true }
      }
    }
  }
}
```

The deployment tool (Azure, Kubernetes) resolves these values at deploy time.
