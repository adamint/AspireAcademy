# IResource — Everything Is a Resource

In Aspire, every component of your distributed application is a **resource**. A PostgreSQL database is a resource. A Redis cache is a resource. Your C# API project is a resource. A Python script is a resource. Even a configuration parameter is a resource.

This is a design decision that has deep consequences — and once you understand it, everything else in Aspire clicks.

---

## The Interface

```csharp
public interface IResource
{
    string Name { get; }
    ResourceAnnotationCollection Annotations { get; }
}
```

That is it. Two members.

> 📝 **Name** — a unique identifier used in the Dashboard, service discovery, and deployment artifacts.

> 📝 **Annotations** — a collection of metadata objects that describe the resource's behavior. Think of them as "capability tags" that tell Aspire how to handle the resource.

Every resource in Aspire implements this interface. The simplicity is intentional — behavior is attached through **annotations** (composition), not through deep class hierarchies (inheritance).

## Why Everything Is a Resource

By treating everything as a resource, Aspire can:

| Capability | How it uses `IResource` |
|---|---|
| **Dashboard display** | Every resource gets a row in the Resources table |
| **State tracking** | Starting → Running → Exited → FailedToStart |
| **Wiring** | `WithReference()` works between any two resources |
| **Publishing** | The pipeline generates artifacts for each resource type |
| **Health checks** | Any resource can have health check annotations |

When you call `builder.AddPostgres("pg")`, you are adding a resource named `"pg"` to the app model. When you call `builder.AddViteApp("frontend", "./web")`, you are adding another resource. They are different types, but they share the same `IResource` foundation.

## The Resource Name Contract

Resource names must be unique within an AppHost. The name is used as:

| Context | Example |
|---|---|
| Dashboard resource table | Row labeled `"api"` |
| Service discovery hostname | `http://api` resolves to the resource named `"api"` |
| Connection string env var | `ConnectionStrings__cache` |
| Deployment manifest | K8s Deployment named `"api"` |

> ⚠️ If two resources share the same name, the AppHost will throw at startup. Choose names that are short, lowercase, and meaningful — they appear everywhere.

---

> 📋 **Key Takeaways**
> - `IResource` has two members: `Name` and `Annotations` — everything else is composition
> - Annotations attach capabilities (endpoints, health checks, volumes) without class inheritance
> - The resource name is a universal key: Dashboard, DNS, env vars, and deployment all use it
> - This design means any new resource type automatically gets Dashboard display, state tracking, wiring, and publishing
