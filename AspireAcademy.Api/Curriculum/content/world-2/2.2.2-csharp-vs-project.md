# CSharpAppResource vs ProjectResource

When to use AddCSharpApp vs AddProject — trade-offs and use cases.

## AddProject&lt;T&gt; — The Standard Approach

```csharp
var api = builder.AddProject<Projects.ApiService>("api");
```

Requires a `<ProjectReference>` in your AppHost `.csproj`. The SDK generates a `Projects.ApiService` type at build time that implements `IProjectMetadata`. If you remove the reference, you get a **build error**.

## AddCSharpApp — Path-Based Flexibility

```csharp
var api = builder.AddCSharpApp("api", "../external/Api.csproj");
```

No project reference needed. The path is a string — typos are runtime errors, not build errors. Supports `.csproj` files, `.cs` single-file apps, and directories.

## Decision Matrix

| Scenario | Use |
|----------|-----|
| Project in your solution | `AddProject<T>` |
| Project outside your solution | `AddCSharpApp` |
| Single-file `.cs` app | `AddCSharpApp` |
| CI/CD with strict build validation | `AddProject<T>` |
| Rapid prototyping | Either works |

## Both Create the Same Resource Type

Both methods ultimately produce a `ProjectResource`. The orchestrator treats them identically — same process launching, same environment variable injection, same service discovery. The only difference is how the project is discovered (generated type vs. string path).

## Recommendation

Default to `AddProject<T>` for projects in your solution. Use `AddCSharpApp` when you need to reference external projects or single-file C# applications.
