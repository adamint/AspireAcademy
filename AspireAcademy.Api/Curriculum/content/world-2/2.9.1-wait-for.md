# WaitFor — Health-Based Readiness

Use WaitFor to delay startup until a dependency reports healthy, with configurable WaitBehavior.

## The API

```csharp
public static IResourceBuilder<T> WaitFor<T>(
    this IResourceBuilder<T> builder,
    IResourceBuilder<IResource> dependency)
    where T : IResourceWithWaitSupport
```

## Basic Usage

```csharp
var db = builder.AddPostgres("pg").AddDatabase("mydb");
var api = builder.AddProject<Projects.Api>("api")
                 .WithReference(db)
                 .WaitFor(db);
```

The API won't start until PostgreSQL is **healthy** — not just running, but actually accepting connections.

## How It Works

1. The API enters the `Waiting` state
2. Aspire monitors the database's health checks
3. When the database reports `Healthy`, the API transitions to `Starting`
4. If the database enters a terminal state (`FailedToStart`, `Exited`), the waiting resource handles it based on `WaitBehavior`

## Chaining Dependencies

```csharp
var db = builder.AddPostgres("pg").AddDatabase("mydb");
var cache = builder.AddRedis("cache");

var api = builder.AddProject<Projects.Api>("api")
                 .WithReference(db)
                 .WithReference(cache)
                 .WaitFor(db)
                 .WaitFor(cache);

builder.AddProject<Projects.Web>("web")
       .WithReference(api)
       .WaitFor(api);
```

The API waits for both the database and cache. The web frontend waits for the API.

## Important Rules

- **No self-waits** — `api.WaitFor(api)` throws an exception
- **Child resources wait on parents** — if you wait on a database, Aspire automatically waits for the parent server too
- Resources without health checks are considered healthy when `Running`

## Dashboard Behavior

When a resource is waiting, the Dashboard shows:
- State: **Waiting**
- A dependency indicator showing what it's waiting for
- Transitions to **Starting** → **Running** once dependencies are ready
