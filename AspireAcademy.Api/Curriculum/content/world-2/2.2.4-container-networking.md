# Container Networking and DNS

Aspire creates a shared container network where resources discover each other using DNS aliases. Understanding this networking model helps you debug connectivity issues and design resource topologies.

## The Aspire Network

When the AppHost starts, Aspire's orchestrator (DCP) creates a Docker network. Every container resource joins this network automatically. Each container gets a DNS alias matching its **resource name** — the first argument to `AddContainer()`.

```csharp
var redis = builder.AddContainer("cache", "redis:7");
var api = builder.AddProject<Projects.ApiService>("api")
    .WithReference(redis);
```

Within the network, the Redis container is reachable at hostname `cache`.

## How Endpoints Map to Ports

When you add an endpoint, Aspire maps it through a proxy. The proxy listens on an allocated host port and forwards traffic to the container's target port:

```csharp
var redis = builder.AddContainer("cache", "redis:7")
    .WithEndpoint(targetPort: 6379, scheme: "tcp", name: "redis");
```

Other resources referencing this endpoint receive the proxy's address — not the container's internal address directly.

## Project-to-Container Communication

Projects reference containers via `WithReference()`. Aspire injects the connection information as environment variables, following the same patterns used for all resource types:

```csharp
var api = builder.AddProject<Projects.ApiService>("api")
    .WithReference(redis);
```

This mirrors production service discovery patterns — services don't hardcode addresses.

## Key Points

- Aspire creates a shared Docker network for all containers
- Resource names become DNS aliases within the network
- Endpoints are proxied through allocated host ports
- `WithReference()` injects connection information as environment variables
- This networking model works identically for containers and projects
