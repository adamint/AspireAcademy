# WaitForCompletion and WaitForStart

Wait for init containers to exit with WaitForCompletion(dependency, exitCode) or just start with WaitForStart.

## WaitForCompletion

```csharp
public static IResourceBuilder<T> WaitForCompletion<T>(
    this IResourceBuilder<T> builder,
    IResourceBuilder<IResource> dependency,
    int exitCode = 0)
    where T : IResourceWithWaitSupport
```

Use this for **init containers** or **setup tasks** that must finish before your service starts:

```csharp
var dbMigration = builder.AddContainer("migrate", "my-migration-image")
                         .WithReference(db);

var api = builder.AddProject<Projects.Api>("api")
                 .WithReference(db)
                 .WaitForCompletion(dbMigration);
```

The API waits until the migration container exits with code 0. If it exits with a different code, the wait fails.

## Custom Exit Codes

Some tools use non-zero exit codes for success:

```csharp
builder.AddProject<Projects.Api>("api")
       .WaitForCompletion(healthCheck, exitCode: 0);
```

## WaitFor vs WaitForCompletion

| Method | Waits Until | Use Case |
|--------|-------------|----------|
| `WaitFor` | Dependency is healthy (Running + health checks pass) | Long-lived services |
| `WaitForCompletion` | Dependency exits with expected code | Init containers, migrations |

## Example: Database Migration Pattern

```csharp
var builder = DistributedApplication.CreateBuilder(args);

var db = builder.AddPostgres("pg")
                .AddDatabase("mydb");

// Migration container runs once and exits
var migrate = builder.AddDockerfile("migrate", "../migrations")
                     .WithReference(db)
                     .WaitFor(db);  // Migration waits for DB

// API waits for migration to complete
var api = builder.AddProject<Projects.Api>("api")
                 .WithReference(db)
                 .WaitForCompletion(migrate);

builder.Build().Run();
```

Flow: DB starts → Migration runs → Migration exits (code 0) → API starts

## Important Notes

- `WaitForCompletion` is designed for **local development** scenarios
- The dependency must reach a terminal state (`Exited` or `Finished`)
- If the dependency never exits, the waiting resource stays in `Waiting` forever
