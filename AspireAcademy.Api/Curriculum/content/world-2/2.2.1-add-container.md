# Adding Containers with AddContainer

`AddContainer()` registers a container resource in the Aspire app model. Containers run alongside your .NET projects, appearing in the dashboard and participating in dependency graphs.

## Basic Usage

```csharp
var redis = builder.AddContainer("cache", "redis:7");
```

The first argument is the resource name; the second is the Docker image. The image string follows standard Docker conventions — `image:tag` or `registry/image:tag`.

You can also separate the image and tag:

```csharp
var redis = builder.AddContainer("cache", "redis", "7.2");
```

## How Containers Participate

Container resources integrate fully with the Aspire model:

- **Dashboard**: Containers appear with status, logs, and endpoints
- **Endpoints**: Add HTTP or custom endpoints that other resources can reference
- **Environment variables**: Inject configuration via `WithEnvironment()`
- **Dependencies**: Other resources can use `WithReference()` and `WaitFor()`

## Adding Endpoints to Containers

Containers often need explicit endpoint definitions since Aspire can't infer them from launch profiles:

```csharp
var redis = builder.AddContainer("cache", "redis:7")
    .WithEndpoint(targetPort: 6379, scheme: "tcp", name: "redis");
```

## When to Use AddContainer vs Hosting Integrations

For well-known infrastructure (Redis, PostgreSQL, RabbitMQ), prefer Aspire's hosting integration packages — they provide `AddRedis()`, `AddPostgres()`, etc. with pre-configured endpoints, health checks, and connection strings. Use `AddContainer()` for:

- Custom or niche container images
- Sidecar containers (logging agents, proxies)
- Containers without a dedicated Aspire hosting package

```csharp
// Hosting integration (preferred for Redis)
var redis = builder.AddRedis("cache");

// Generic container (for custom images)
var sidecar = builder.AddContainer("proxy", "envoyproxy/envoy:v1.30");
```
