# DistributedApplicationModel — The Complete Graph

When you call `builder.Build()`, Aspire creates a `DistributedApplicationModel` — a snapshot of every resource, annotation, and relationship in your application.

---

## The Model

```csharp
public class DistributedApplicationModel
{
    public IResourceCollection Resources { get; }
}
```

The model is the **complete graph** of your application. DCP reads it to know what to start. Publishers read it to know what artifacts to generate. The Dashboard reads it to know what to display.

## How It Is Built

```
CreateBuilder()
    ↓
AddPostgres("pg")  → adds PostgresServerResource to model
AddRedis("cache")  → adds RedisResource to model
AddProject("api")  → adds ProjectResource to model
    .WithReference(pg) → adds ConnectionStringRedirectAnnotation
    .WaitFor(pg)       → adds WaitAnnotation
    ↓
Build() → freezes the model into DistributedApplicationModel
    ↓
Run()   → passes model to DCP for execution
```

Every `Add*`, `With*`, and `WaitFor` call modifies the builder's internal state. `Build()` captures that state as an immutable model. `Run()` hands the model to the orchestrator.

## The Model Is the Source of Truth

The same model is consumed by:

1. **DCP** (run mode) — starts containers and processes
2. **Publishing pipeline** (publish mode) — generates Docker Compose, K8s, Bicep
3. **Testing harness** — creates an in-memory AppHost for integration tests
4. **Dashboard** — displays resources and their relationships

This is why one AppHost can serve multiple purposes — the model is a neutral representation.

> 🎲 **Fun Fact:** The DistributedApplicationModel is so central to Aspire that integration tests create one without ever calling Run(). Test frameworks call Build() and then inspect the model directly — verifying that resources, annotations, and relationships are correctly configured without starting any containers.
