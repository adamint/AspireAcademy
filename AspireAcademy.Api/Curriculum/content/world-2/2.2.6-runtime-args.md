# Container Runtime Arguments

`WithContainerRuntimeArgs()` passes arguments to the container runtime (Docker) itself — not to the application inside the container. These control resource limits, security, and runtime behavior.

## Basic Usage

```csharp
var gpu = builder.AddContainer("ml-worker", "ml-model:latest")
    .WithContainerRuntimeArgs("--gpus", "all");
```

Runtime args are added to the `docker run` command, appearing before the image name. This is distinct from:

- **`WithArgs()`**: Arguments passed to the container's entrypoint (after the image name)
- **`WithEnvironment()`**: Environment variables inside the container

## Common Use Cases

```csharp
// Memory limits
.WithContainerRuntimeArgs("--memory", "512m")

// CPU limits
.WithContainerRuntimeArgs("--cpus", "2.0")

// Privileged mode (use sparingly)
.WithContainerRuntimeArgs("--privileged")

// Custom network settings
.WithContainerRuntimeArgs("--dns", "8.8.8.8")
```

## Callback Overload

For dynamic or conditional runtime args:

```csharp
.WithContainerRuntimeArgs(context =>
{
    context.Args.Add("--memory");
    context.Args.Add("512m");
    if (builder.ExecutionContext.IsRunMode)
    {
        context.Args.Add("--gpus");
        context.Args.Add("all");
    }
});
```

## Runtime Args vs Entry-Point Args vs Env Vars

| Method | Where it goes | Example |
|--------|--------------|---------|
| `WithContainerRuntimeArgs()` | `docker run [HERE] image` | `--memory 512m` |
| `WithArgs()` | `docker run image [HERE]` | `--verbose --port 8080` |
| `WithEnvironment()` | `docker run -e KEY=VAL image` | `LOG_LEVEL=debug` |

## Key Points

- Use sparingly — most configuration should use env vars or entry-point args
- Runtime args affect the container host, not the application
- Useful for GPU access, resource limits, and advanced Docker features
