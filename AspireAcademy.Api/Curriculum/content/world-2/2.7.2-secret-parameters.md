# Secret Parameters and Generated Defaults

Mark parameters as secrets with AddParameter(name, secret: true) and provide generated default values.

## Secret Parameters

```csharp
var password = builder.AddParameter("db-password", secret: true);
```

When `secret: true`:
- The value is **masked** in the Dashboard (shown as `****`)
- The value is stored securely in the manifest as a secret input
- The Dashboard prompts for the value if it's not configured

## Generated Default Values

For passwords and keys, you often want a random default:

```csharp
var password = builder.AddParameter("db-password",
    new GenerateParameterDefault
    {
        MinLength = 22,
        Special = false  // no special characters
    },
    secret: true);
```

`GenerateParameterDefault` auto-generates a random value with configurable:
- `MinLength` — minimum character count
- `Special` — include special characters
- `MinLower`, `MinUpper`, `MinNumeric` — character class minimums

## Using Secrets with Hosting Integrations

Many integrations accept parameters for credentials:

```csharp
var pgPassword = builder.AddParameter("pg-password", secret: true);
var pg = builder.AddPostgres("pg", password: pgPassword);
```

If you don't provide a password parameter, `AddPostgres` generates one automatically. Providing your own gives you control over the value.

## Persist Parameter

Use `persist: true` to save a generated value so it doesn't change between restarts:

```csharp
builder.AddParameter("api-key",
    new GenerateParameterDefault { MinLength: 32 },
    secret: true,
    persist: true);
```

## Publish vs Default Values

The `publishValueAsDefault` flag controls whether a hardcoded value appears in the manifest:

```csharp
builder.AddParameter("region", "eastus", publishValueAsDefault: true);
```

This sets `"eastus"` as the default in the manifest, but deployers can override it.
