# 9.1.1 — What Is the Developer Control Plane (DCP)?

## Introduction

When you run an Aspire AppHost, you might assume the AppHost process itself is starting your containers and executables. In reality, Aspire delegates all local orchestration to a separate component called the **Developer Control Plane (DCP)**. DCP is the engine under the hood — the local orchestrator that manages containers, processes, networking, and port allocation on your development machine.

## DCP as a Separate Process

DCP runs as its own process alongside the AppHost. When you call `builder.Build().Run()`, the AppHost:

1. Builds the application model (the graph of resources and relationships)
2. Launches the DCP process
3. Communicates with DCP over **gRPC** to submit resource definitions
4. DCP takes over — creating containers, starting executables, allocating ports

This separation of concerns means the AppHost is declarative (it describes *what* to run) while DCP is imperative (it *executes* the plan).

```
┌─────────────────────┐       gRPC        ┌──────────────────────┐
│      AppHost         │ ───────────────▶  │        DCP           │
│  (App Model / C#)    │ ◀───────────────  │  (Local Orchestrator)│
└─────────────────────┘                    └──────────┬───────────┘
                                                      │
                                           ┌──────────▼───────────┐
                                           │  Container Runtime   │
                                           │  (Docker / Podman)   │
                                           └──────────────────────┘
```

## Kubernetes-Inspired Architecture

DCP borrows core concepts from Kubernetes but reimagines them for local development:

| Concept | Kubernetes | DCP |
|---|---|---|
| Resource model | YAML manifests with CRDs | In-memory objects derived from C# app model |
| Container orchestration | Pods, Deployments, Services | Container, Executable, ExecutableReplicaSet |
| Networking | ClusterIP, NodePort, Ingress | Port allocation and localhost binding |
| Desired state reconciliation | Control loops in controllers | Lightweight reconciliation for dev scenarios |
| Scope | Production clusters across nodes | Single developer machine |

Like Kubernetes, DCP uses **Custom Resource Definitions (CRDs)** — structured objects that describe the desired state of each resource. The DCP controller watches these definitions and reconciles the actual state to match.

## DCP Resource Types

DCP defines three primary resource types that map to Aspire resources:

### Container

A `Container` resource represents a Docker or Podman container. When you call `builder.AddContainer(...)` or use a hosting integration that wraps a container image (like `AddRedis` or `AddPostgres`), the AppHost creates a DCP `Container` object.

DCP handles:
- Pulling the container image if not already cached
- Creating and starting the container with the correct environment variables, ports, and volume mounts
- Monitoring the container's running state
- Stopping and removing the container on shutdown

### Executable

An `Executable` resource represents a locally running process. When you call `builder.AddProject<T>(...)`, the AppHost translates that into a DCP `Executable` — because .NET projects are ultimately `dotnet run` invocations.

DCP handles:
- Starting the process with the correct working directory and arguments
- Injecting environment variables (connection strings, service discovery URLs, etc.)
- Monitoring the process for crashes or exits
- Terminating the process on shutdown

### ExecutableReplicaSet

An `ExecutableReplicaSet` manages multiple identical instances of an `Executable`. When you call `.WithReplicas(3)` on a project resource, DCP creates three instances of the executable, each with its own allocated ports:

```csharp
var api = builder.AddProject<Projects.WeatherApi>("api")
    .WithReplicas(3);
```

DCP assigns unique ports to each replica and coordinates service discovery so that other resources can load-balance across them.

## How DCP Differs from Kubernetes

DCP is **not** a Kubernetes replacement — it is optimized for the local development experience:

- **Single-machine scope**: DCP runs everything on `localhost`. There is no cluster, no node scheduling, no multi-machine networking.
- **No production orchestration**: DCP does not handle rolling updates, auto-scaling, or fault tolerance strategies. It is designed for fast iteration.
- **Ephemeral by default**: Resources are created when the AppHost starts and destroyed when it stops (unless using persistent container lifetimes via `ContainerLifetime.Persistent`).
- **Direct container runtime access**: DCP talks directly to Docker or Podman — there is no kubelet, no CRI shim, no container sandboxing layer.
- **Speed over resilience**: DCP prioritizes startup speed and developer feedback over the consistency guarantees that Kubernetes provides in production.

## Container Lifetime Modes

DCP supports two container lifetime modes controlled by the `ContainerLifetime` enum:

```csharp
var redis = builder.AddRedis("cache")
    .WithLifetime(ContainerLifetime.Persistent);
```

| Mode | Behavior |
|---|---|
| `ContainerLifetime.Session` | Container is created on startup and destroyed on shutdown (default) |
| `ContainerLifetime.Persistent` | Container survives AppHost restarts, reusing existing state |

Persistent containers are useful for databases or caches where you want to preserve data across development sessions.

## The DCP Lifecycle

The complete lifecycle during a typical `dotnet run` of an AppHost:

1. **AppHost builds the app model** — resources, references, environment variables, endpoints
2. **DCP process starts** — the AppHost launches DCP as a child process
3. **gRPC channel established** — the AppHost connects to DCP's gRPC API
4. **Resources submitted** — the app model is translated to DCP CRDs and sent via gRPC
5. **DCP reconciles** — containers are pulled/started, executables are launched, ports are allocated
6. **State flows back** — DCP reports resource states (Starting, Running, Exited) back to the AppHost
7. **Dashboard displays** — the AppHost's `ResourceNotificationService` relays state to the dashboard
8. **Shutdown** — when the AppHost exits, DCP tears down all session-scoped resources

## Summary

DCP is the invisible workhorse of Aspire's local development experience. It takes the declarative app model you build in C# and turns it into running containers, processes, and network endpoints — all on your local machine, all orchestrated through a Kubernetes-inspired but development-optimized control plane.
