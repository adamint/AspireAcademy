# ATS Core Concepts

The Application Type System (ATS) defines how .NET types are exposed to polyglot AppHosts. It has three categories.

---

## Handle Types

Types marked with `[AspireExport]`. These are passed **by reference** — only an opaque handle ID crosses the wire.

Examples:
- `IDistributedApplicationBuilder` — the builder itself
- `IResourceBuilder<RedisResource>` — a resource builder
- `EndpointReference` — a reference to an endpoint

When TypeScript calls `builder.addRedis("cache")`, it gets back a handle ID referencing the .NET `IResourceBuilder<RedisResource>`. All subsequent calls on `cache` (like `cache.withLifetime(...)`) pass this handle ID back to the server.

## DTO Types

Types marked with `[AspireDto]`. These are passed **by value** — serialized as JSON.

Examples:
- `CreateBuilderOptions` — options for creating a builder
- `ResourceSnapshot` — snapshot of resource state

DTOs are simple data objects. They do not have methods — they are read as JSON on the client side.

## Enum Types

Discovered in capability signatures. Serialized as **strings**.

Examples:
- `ContainerLifetime.Persistent` → `"Persistent"`
- `UriScheme.Http` → `"http"`

## AtsContext

`AtsContext` is the central metadata store:

```csharp
public class AtsContext
{
    public Dictionary<string, AtsHandleType> HandleTypes { get; }
    public Dictionary<string, AtsDtoType> DtoTypes { get; }
    public Dictionary<string, AtsEnumType> EnumTypes { get; }
    public List<AtsCapability> Capabilities { get; }
}
```

It is built by scanning assemblies for `[AspireExport]` and `[AspireDto]` attributes. The code generators consume `AtsContext` to produce language-specific SDKs.
