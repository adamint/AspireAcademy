# 9.1.3 — Build-Time vs Run-Time Configuration

## Introduction

Aspire separates your application's lifecycle into two distinct phases: **build-time** (where you define the app model) and **run-time** (where that model is materialized into running infrastructure). Understanding this distinction is critical for writing correct AppHost configurations, especially when your app targets both local development and cloud deployment.

## Build-Time: Assembling the App Model

Build-time is the phase where you construct the **resource graph** — the declarative description of your entire distributed application. This happens in your AppHost's `Program.cs`:

```csharp
var builder = DistributedApplication.CreateBuilder(args);

// Build-time: defining resources and relationships
var cache = builder.AddRedis("cache");
var db = builder.AddPostgres("pg").AddDatabase("appdb");

var api = builder.AddProject<Projects.WeatherApi>("api")
    .WithReference(cache)
    .WithReference(db)
    .WaitFor(cache)
    .WaitFor(db);

var frontend = builder.AddProject<Projects.Frontend>("frontend")
    .WithReference(api);

// Nothing is running yet — we've only built a graph
var app = builder.Build();
app.Run();
```

During build-time, no containers are started, no ports are allocated, and no connection strings are resolved. You are building an **in-memory graph** that describes:

- **Resources**: What services, containers, and infrastructure exist
- **References**: Which resources depend on which others
- **Endpoints**: What network endpoints each resource exposes
- **Configuration**: Environment variables, arguments, and settings
- **Annotations**: Metadata attached to resources (health checks, lifecycle behavior, etc.)

### The Resource Graph

The app model is a directed graph where resources are nodes and references are edges:

```
  ┌────────┐         ┌────────┐
  │ cache  │         │   db   │
  │ (Redis)│         │ (PG)   │
  └───┬────┘         └───┬────┘
      │                   │
      │  WithReference    │  WithReference
      ▼                   ▼
  ┌──────────────────────────┐
  │          api             │
  │    (WeatherApi project)  │
  └────────────┬─────────────┘
               │
               │  WithReference
               ▼
  ┌──────────────────────────┐
  │        frontend          │
  │    (Frontend project)    │
  └──────────────────────────┘
```

Each edge carries information about what the dependent resource needs — typically a connection string or service discovery endpoint.

## Run-Time: Materializing the Model

When `app.Run()` is called, the build-time model is handed to DCP for materialization. This is where abstract definitions become concrete infrastructure:

### Port Allocation

Endpoints without explicit ports receive dynamically allocated ports:

```csharp
// Build-time: no port specified
var api = builder.AddProject<Projects.Api>("api")
    .WithHttpEndpoint();

// Run-time: DCP allocates port 52341 (example)
// Environment variable injected: services__api__http__0=http://localhost:52341
```

### Connection String Resolution

Connection strings are computed from the actual allocated ports and resource addresses:

```csharp
// Build-time: AddPostgres creates an abstract resource
var db = builder.AddPostgres("pg").AddDatabase("appdb");

// Run-time: DCP starts PostgreSQL on a random port (e.g., 54321)
// Connection string resolved: Host=localhost;Port=54321;Username=postgres;Password=<generated>;Database=appdb
```

### Environment Variable Computation

All environment variables — including connection strings, service discovery URLs, and OTEL endpoints — are computed from the materialized state and injected into each resource's environment.

## ExecutionContext: Knowing Your Phase

Aspire provides `ExecutionContext` to let your AppHost code behave differently depending on whether it's running locally or being published for deployment:

```csharp
var builder = DistributedApplication.CreateBuilder(args);

if (builder.ExecutionContext.IsPublishMode)
{
    // Publishing to a deployment manifest (e.g., azd, Kubernetes)
    // Resources may be configured differently for production
}
else
{
    // Running locally via `dotnet run` or F5
    // Use local containers, emulators, etc.
}
```

### IsPublishMode vs IsRunMode

| Property | When True | Use Case |
|---|---|---|
| `ExecutionContext.IsPublishMode` | Running `dotnet run` with publish arguments or generating a manifest | Configure production-ready resource definitions |
| `ExecutionContext.IsRunMode` | Normal local development (`dotnet run`, F5, `aspire run`) | Use local containers, emulators, dev settings |

This is commonly used with Azure resources to switch between emulators locally and real cloud services in production:

```csharp
var storage = builder.AddAzureStorage("storage");

if (builder.ExecutionContext.IsRunMode)
{
    // Local development: use the Azurite emulator
    storage.RunAsEmulator();
}
// In publish mode, it provisions a real Azure Storage account
```

## PublishAs Methods

Some resources provide `PublishAs*` methods that configure how the resource should be represented when publishing a deployment manifest, without affecting local development:

```csharp
var redis = builder.AddRedis("cache")
    .PublishAsAzureRedis();
```

This means:
- **Locally** (`IsRunMode`): A Redis Docker container runs on your machine
- **Published** (`IsPublishMode`): The manifest describes an Azure Cache for Redis resource

The `PublishAs*` methods modify the resource's metadata so that deployment tools (like `azd`) know how to provision the correct cloud resource.

## The Manifest: Serialized Build-Time Model

When you run in publish mode, the build-time app model is serialized into a **manifest** — a JSON file that describes the entire application:

```bash
dotnet run --project AppHost -- --publisher manifest --output-path ./manifest.json
```

The manifest captures:

```json
{
  "resources": {
    "cache": {
      "type": "container.v0",
      "image": "docker.io/library/redis:7.4",
      "bindings": {
        "tcp": { "scheme": "tcp", "protocol": "tcp", "transport": "tcp", "targetPort": 6379 }
      }
    },
    "api": {
      "type": "project.v0",
      "path": "../WeatherApi/WeatherApi.csproj",
      "env": {
        "ConnectionStrings__cache": "{cache.bindings.tcp.host}:{cache.bindings.tcp.port}"
      },
      "bindings": {
        "http": { "scheme": "http", "protocol": "tcp", "transport": "http" },
        "https": { "scheme": "https", "protocol": "tcp", "transport": "http" }
      }
    }
  }
}
```

Notice that the manifest uses **expression placeholders** like `{cache.bindings.tcp.host}` — these are resolved at deployment time, not build time.

## Build-Time Annotations

During build-time, resources accumulate **annotations** — metadata objects attached to the resource that influence run-time behavior:

```csharp
var api = builder.AddProject<Projects.Api>("api")
    .WithHttpEndpoint(port: 5000, name: "public")   // adds EndpointAnnotation
    .WithEnvironment("MODE", "debug")                 // adds EnvironmentCallbackAnnotation
    .WaitFor(cache)                                   // adds WaitAnnotation
    .WithHealthCheck("api-health");                   // adds HealthCheckAnnotation
```

Each `With*` method typically adds an annotation to the resource's `Annotations` collection. These annotations are read during run-time to configure the actual infrastructure.

## Common Build-Time vs Run-Time Mistakes

### Mistake 1: Accessing run-time values at build-time

```csharp
// WRONG: Port is not allocated yet at build-time
var endpoint = api.GetEndpoint("https");
var url = endpoint.Url; // This will throw or return incomplete data
```

Instead, use reference expressions that are resolved at run-time:

```csharp
// CORRECT: Use ReferenceExpression for deferred resolution
builder.AddProject<Projects.Frontend>("frontend")
    .WithEnvironment("API_URL", api.GetEndpoint("https"));
```

### Mistake 2: Assuming fixed ports

```csharp
// FRAGILE: Hardcoded port may conflict
var api = builder.AddProject<Projects.Api>("api")
    .WithHttpEndpoint(port: 5000);

// BETTER: Let DCP allocate ports dynamically
var api = builder.AddProject<Projects.Api>("api")
    .WithHttpEndpoint();
```

### Mistake 3: Not accounting for publish mode

```csharp
// WRONG: Emulator runs in production
var cosmos = builder.AddAzureCosmosDB("cosmos")
    .RunAsEmulator();

// CORRECT: Emulator only in development
var cosmos = builder.AddAzureCosmosDB("cosmos");
if (builder.ExecutionContext.IsRunMode)
{
    cosmos.RunAsEmulator();
}
```

## The Two-Phase Mental Model

Think of Aspire configuration in two phases:

```
Phase 1: BUILD-TIME                    Phase 2: RUN-TIME
─────────────────────                  ──────────────────
• Define resources                     • Start DCP
• Declare references                   • Allocate ports
• Set environment expressions          • Resolve connection strings
• Add annotations                      • Compute environment variables
• Configure endpoints                  • Pull images / build projects
• Build the resource graph             • Start containers & processes
                                       • Monitor health
                                       • Relay state to dashboard
```

All your AppHost code in `Program.cs` executes during Phase 1. The `app.Run()` call triggers Phase 2.

## Summary

The build-time/run-time separation is a fundamental architectural decision in Aspire. Build-time is declarative — you describe your system. Run-time is imperative — DCP makes it real. Understanding this boundary helps you write AppHost configurations that work correctly in both local development and cloud deployment scenarios, avoid common pitfalls around port allocation and connection string resolution, and leverage `ExecutionContext` to customize behavior per environment.
