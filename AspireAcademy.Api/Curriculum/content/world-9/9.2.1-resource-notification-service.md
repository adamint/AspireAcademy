# 9.2.1 — ResourceNotificationService Deep Dive

## Introduction

The `ResourceNotificationService` is the central nervous system of an Aspire application at run time. It is a **pub/sub system** that broadcasts resource state changes to all interested subscribers — the dashboard, lifecycle hooks, the `WaitFor` mechanism, and your own custom code. Every time a resource transitions from `Starting` to `Running`, or a health check fails, or a container exits — this service is how the rest of the system finds out.

## Architecture Overview

`ResourceNotificationService` is a public class that implements `IDisposable`:

```csharp
public class ResourceNotificationService : IDisposable
```

It sits between the orchestration layer (which produces state updates) and consumers (which react to them):

```
Producers                        Consumers
─────────                        ─────────
DCP state changes  ──┐     ┌──▶ Aspire Dashboard (real-time UI)
Health check polls ──┤     ├──▶ WaitFor mechanism (dependency gating)
Custom publishers  ──┼──▶──┤
Lifecycle hooks    ──┤     ├──▶ Lifecycle hooks (event reactions)
Resource commands  ──┘     └──▶ Custom subscribers (your code)
```

## CustomResourceSnapshot

At the heart of the notification system is `CustomResourceSnapshot` — a sealed record that captures the complete state of a resource at a point in time:

```csharp
public sealed record CustomResourceSnapshot
```

A snapshot contains information about:

- **Resource state** (e.g., `Starting`, `Running`, `Exited`, `FailedToStart`)
- **Properties** — key-value pairs visible in the dashboard (URLs, connection strings, container IDs)
- **Environment variables** — the resolved environment for the resource
- **Endpoint URLs** — allocated and active endpoints
- **Health status** — the last known health check result
- **Creation timestamp** — when the resource was first observed
- **Exit code** — for resources that have exited

Each state change produces a new `CustomResourceSnapshot` — snapshots are immutable records, so each update is a fresh snapshot rather than a mutation.

## Publishing State Updates

The orchestration layer publishes state changes using `PublishUpdateAsync`. This method takes a resource and a function that transforms the previous snapshot into a new one:

```csharp
await resourceNotificationService.PublishUpdateAsync(
    myResource,
    snapshot => snapshot with
    {
        State = new ResourceStateSnapshot("Running", KnownResourceStateStyles.Success)
    });
```

The callback receives the previous `CustomResourceSnapshot` and returns a new one. Using the `with` expression on the record ensures immutability — you create a new snapshot based on the previous state with only the changed fields updated.

### Publishing Custom Properties

You can publish custom properties that appear in the dashboard's resource details:

```csharp
await resourceNotificationService.PublishUpdateAsync(
    myResource,
    snapshot => snapshot with
    {
        Properties = [
            .. snapshot.Properties,
            new ResourcePropertySnapshot(
                "region",
                KnownProperties.Resource.Region,
                "East US")
        ]
    });
```

## Subscribing to State Changes

Subscribers can watch for state changes on specific resources. The subscription returns an `IAsyncEnumerable<ResourceEvent>` that yields events as they occur:

```csharp
await foreach (var resourceEvent in
    resourceNotificationService.WatchAsync(cancellationToken))
{
    var snapshot = resourceEvent.Snapshot;
    var resourceName = resourceEvent.Resource.Name;

    Console.WriteLine(
        $"Resource '{resourceName}' is now {snapshot.State?.Text}");
}
```

You can also watch a specific resource:

```csharp
await foreach (var resourceEvent in
    resourceNotificationService.WatchAsync(myResource, cancellationToken))
{
    // Only events for myResource
}
```

## Resource State Flow

Resources transition through a well-defined set of states:

```
                    ┌──────────────┐
                    │   Starting   │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
      ┌──────────┐  ┌───────────┐  ┌──────────────┐
      │ Running  │  │  Waiting  │  │ FailedToStart│
      └────┬─────┘  └───────────┘  └──────────────┘
           │
     ┌─────┼──────┐
     ▼            ▼
┌──────────┐ ┌────────┐
│ Stopping │ │ Exited │
└────┬─────┘ └────────┘
     ▼
┌──────────┐
│ Stopped  │
└──────────┘
```

Common states and their meanings:

| State | Meaning | Dashboard Display |
|---|---|---|
| `Starting` | Resource is being created or launched | Spinner / blue indicator |
| `Running` | Resource is active and healthy | Green indicator |
| `Waiting` | Resource is waiting for a dependency | Yellow / waiting indicator |
| `FailedToStart` | Resource could not start | Red indicator with error details |
| `Stopping` | Resource is shutting down | Transitional state |
| `Exited` | Resource has terminated | Gray indicator with exit code |
| `RuntimeUnhealthy` | Resource is running but health checks are failing | Orange/warning indicator |
| `Hidden` | Resource exists but should not be shown | Not displayed |

## How WaitFor Uses ResourceNotificationService

The `WaitFor` mechanism is a consumer of `ResourceNotificationService`. When you write:

```csharp
var db = builder.AddPostgres("db");
var api = builder.AddProject<Projects.Api>("api")
    .WaitFor(db);
```

At run-time, the system:

1. Starts the PostgreSQL container
2. Subscribes to `ResourceNotificationService` for the `db` resource
3. Holds the `api` executable in a `Waiting` state
4. Watches for the `db` resource to reach a `Running` state with a healthy health check
5. Once `db` is healthy, starts the `api` executable

This ensures that services do not start before their dependencies are ready, preventing connection failures during startup.

## How the Dashboard Subscribes

The Aspire Dashboard uses `ResourceNotificationService` to power its real-time UI. When you open the dashboard, it:

1. Retrieves the initial snapshot of all resources
2. Subscribes to `WatchAsync()` for live updates
3. Updates the UI whenever a new `CustomResourceSnapshot` arrives

This is why you see resource states update in real time — every state transition published through `ResourceNotificationService` is immediately reflected in the dashboard.

## Custom Subscriber Example

You can build your own subscriber in a lifecycle hook or background service. Here's a complete example of a background service that logs all resource state changes:

```csharp
public class ResourceStateLogger : BackgroundService
{
    private readonly ResourceNotificationService _notificationService;
    private readonly ILogger<ResourceStateLogger> _logger;

    public ResourceStateLogger(
        ResourceNotificationService notificationService,
        ILogger<ResourceStateLogger> logger)
    {
        _notificationService = notificationService;
        _logger = logger;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        await foreach (var resourceEvent in
            _notificationService.WatchAsync(stoppingToken))
        {
            var name = resourceEvent.Resource.Name;
            var state = resourceEvent.Snapshot.State?.Text ?? "Unknown";

            _logger.LogInformation(
                "Resource '{Name}' transitioned to '{State}'",
                name,
                state);
        }
    }
}
```

Register it in your AppHost:

```csharp
builder.Services.AddHostedService<ResourceStateLogger>();
```

## Thread Safety and Concurrency

`ResourceNotificationService` is designed for concurrent access:

- **Multiple publishers** can call `PublishUpdateAsync` concurrently — updates are serialized per-resource
- **Multiple subscribers** can watch simultaneously — each gets its own independent stream
- **Snapshot immutability** ensures no data races — each subscriber receives its own copy of the state

## Summary

`ResourceNotificationService` is the backbone of Aspire's run-time observability. It connects DCP's resource management to the dashboard, the `WaitFor` dependency system, and any custom subscribers you write. By publishing immutable `CustomResourceSnapshot` records through a pub/sub channel, it enables every part of the system to react to state changes in real time without tight coupling.
