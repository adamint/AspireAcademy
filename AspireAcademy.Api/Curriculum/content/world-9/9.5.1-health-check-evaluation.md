# 9.5.1 — Health Check Evaluation Internals

## Introduction

Health checks are the mechanism Aspire uses to determine whether a resource is truly ready — not just "process started" but "accepting connections and functioning correctly." This lesson dives into how the orchestrator discovers, executes, and reacts to health checks, and how they integrate with the `WaitFor` mechanism and the dashboard.

## The Health Check Polling Loop

Aspire runs a continuous polling loop that evaluates registered health checks for each resource at regular intervals:

```
┌─────────────────────────────────────────────────┐
│              Health Check Polling Loop           │
│                                                  │
│   For each resource with health checks:          │
│                                                  │
│   1. Execute all registered IHealthCheck impls   │
│   2. Aggregate results into HealthStatus         │
│   3. Map HealthStatus to resource state          │
│   4. Publish update via ResourceNotificationSvc  │
│   5. Wait for polling interval                   │
│   6. Repeat                                      │
│                                                  │
└─────────────────────────────────────────────────┘
```

The polling runs as a background task within the AppHost process. It starts as soon as a resource's container or process is launched and continues for the lifetime of the resource.

## How IHealthCheck Implementations Are Discovered

Health checks in Aspire are built on ASP.NET Core's `Microsoft.Extensions.Diagnostics.HealthChecks` framework. The standard `IHealthCheck` interface is used:

```csharp
public interface IHealthCheck
{
    Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext context,
        CancellationToken cancellationToken = default);
}
```

Health checks are registered in two ways:

### 1. Built-in Integration Health Checks

Hosting integrations register health checks automatically. When you call `AddRedis("cache")`, the Redis hosting integration adds a health check that verifies the Redis server is accepting connections:

```csharp
// Inside Aspire.Hosting.Redis (simplified)
builder.AddResource(resource)
    .WithImage("redis")
    .WithEndpoint(targetPort: 6379, name: "tcp")
    .WithHealthCheck("redis-health", (healthBuilder, context) =>
    {
        healthBuilder.AddRedis(
            connectionStringFactory: async (ct) =>
                await context.Resource.ConnectionStringExpression
                    .GetValueAsync(ct),
            name: "redis-health");
    });
```

Common built-in health checks:

| Integration | Health Check Type | What It Checks |
|---|---|---|
| Redis | TCP connection | Can connect to Redis port |
| PostgreSQL | Npgsql connection | Can execute a query |
| SQL Server | SQL connection | Can open a connection |
| RabbitMQ | AMQP connection | Can connect to broker |
| MongoDB | MongoDB connection | Can ping the server |
| Kafka | TCP connection | Can reach broker port |
| Azure Storage | HTTP endpoint | Blob service responds |

### 2. Custom Health Checks via WithHealthCheck

You can add custom health checks to any resource using `WithHealthCheck`:

```csharp
var api = builder.AddProject<Projects.Api>("api")
    .WithHealthCheck("api-ready", (healthBuilder, context) =>
    {
        healthBuilder.AddUrlGroup(
            uri: new Uri("http://localhost/health"),
            name: "api-ready",
            failureStatus: HealthStatus.Unhealthy);
    });
```

## The HealthCheckAnnotation

When a health check is associated with a resource, Aspire stores the relationship using `HealthCheckAnnotation`:

```csharp
public sealed class HealthCheckAnnotation : IResourceAnnotation
{
    // The name of the health check registration
    public string Key { get; }

    // Factory to register the health check with the health checks builder
    public Func<IHealthChecksBuilder, HealthCheckContext, IHealthChecksBuilder>? HealthCheckFactory { get; }
}
```

This annotation links a resource to one or more named health check registrations. When the polling loop runs, it looks up the `HealthCheckAnnotation` instances on each resource to know which checks to execute.

## HealthStatus to Resource State Mapping

The `HealthCheckResult` from each check contains a `HealthStatus` enum value. Aspire maps these to resource states:

| HealthStatus | Resource State | Dashboard Display | Effect on WaitFor |
|---|---|---|---|
| `Healthy` | `Running` | Green indicator | Dependency satisfied — dependents can start |
| `Degraded` | `Running` (with warning) | Yellow/orange indicator | Dependency satisfied — dependents can start |
| `Unhealthy` | Stays in `Starting` | Blue spinner / starting indicator | Dependency NOT satisfied — dependents keep waiting |

This mapping is critical: an `Unhealthy` health check means the resource is **not considered ready**, even if the process or container is running. The resource remains in `Starting` state until health checks pass.

### The Starting → Running Transition

```
Container Started     Health Check Polling     Health Check Passes
     │                       │                        │
     ▼                       ▼                        ▼
┌──────────┐          ┌─────────────┐          ┌───────────┐
│ Starting │──poll──▶ │  Unhealthy  │──poll──▶ │  Healthy  │
│          │          │ (still      │          │           │
│          │          │  Starting)  │          │ → Running │
└──────────┘          └─────────────┘          └───────────┘
                           │
                      (repeats until healthy
                       or timeout)
```

A resource only transitions to `Running` when **all** of its registered health checks report `Healthy` (or `Degraded`).

## How WaitFor Uses Health Checks

The `WaitFor` mechanism gates dependent resource startup on health check results:

```csharp
var db = builder.AddPostgres("db");
var api = builder.AddProject<Projects.Api>("api")
    .WaitFor(db);
```

The internal flow:

1. DCP starts the PostgreSQL container
2. The container enters `Starting` state
3. The health check polling loop begins executing the PostgreSQL health check
4. Each poll that returns `Unhealthy` keeps the resource in `Starting`
5. The `api` resource sees `db` is in `Starting` state and remains in `Waiting`
6. Once the PostgreSQL health check returns `Healthy`:
   - `db` transitions to `Running` via `ResourceNotificationService`
   - The `WaitFor` watcher observes the state change
   - `api` is released from waiting and DCP starts it

```
Timeline:
─────────────────────────────────────────────────────────────

db:    [Starting ──── health check polling ──── Running]
api:   [         Waiting                        Starting → Running]
                                                 ↑
                                          db became healthy
```

## Health Check Polling Configuration

The health check system uses sensible defaults but can be configured:

- **Polling interval**: Health checks are polled at regular intervals (typically every few seconds)
- **Timeout**: Individual health checks have a timeout to prevent hanging
- **Initial delay**: A brief delay before the first poll allows the resource time to initialize

## Multiple Health Checks per Resource

A resource can have multiple health checks. **All** must pass for the resource to be considered healthy:

```csharp
var myService = builder.AddContainer("myservice", "myimage")
    .WithHealthCheck("tcp-check", (b, ctx) =>
    {
        b.AddTcpHealthCheck(o => o.AddHost("localhost", 8080),
            name: "tcp-check");
    })
    .WithHealthCheck("http-check", (b, ctx) =>
    {
        b.AddUrlGroup(new Uri("http://localhost:8080/health"),
            name: "http-check");
    });
```

The resource remains in `Starting` until both `tcp-check` AND `http-check` return `Healthy`.

## Dashboard Health Status Display

The Aspire dashboard displays health status in real time:

```
┌──────────────────────────────────────────────────────────────┐
│ Resources                                                     │
├──────────────┬──────────┬────────────────────────────────────┤
│ Name         │ State    │ Health                              │
├──────────────┼──────────┼────────────────────────────────────┤
│ cache        │ Running  │ ● Healthy                          │
│ db           │ Running  │ ● Healthy                          │
│ api          │ Starting │ ○ Unhealthy (waiting for db)       │
│ frontend     │ Waiting  │ — (waiting for api)                │
└──────────────┴──────────┴────────────────────────────────────┘
```

The dashboard subscribes to `ResourceNotificationService` and receives `CustomResourceSnapshot` updates that include health status information. Each snapshot contains the latest health check results, which the dashboard renders as colored indicators.

## Health Checks and Container Readiness

It's important to distinguish between different levels of "ready":

```
Level 1: Container/Process Created     (DCP reports it exists)
Level 2: Container/Process Running     (the OS process is alive)
Level 3: Port Accepting Connections    (TCP health check passes)
Level 4: Application Ready             (HTTP health endpoint returns 200)
Level 5: Data Loaded                   (custom health check passes)
```

Without health checks, Aspire would consider a resource ready at Level 2. With health checks, you can gate readiness at any level — ensuring that dependent services don't start until their dependencies are truly functional.

## Writing Effective Custom Health Checks

When writing custom health checks, follow these guidelines:

```csharp
public class SmtpHealthCheck : IHealthCheck
{
    private readonly string _host;
    private readonly int _port;

    public SmtpHealthCheck(string host, int port)
    {
        _host = host;
        _port = port;
    }

    public async Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext context,
        CancellationToken cancellationToken = default)
    {
        try
        {
            using var client = new TcpClient();
            await client.ConnectAsync(_host, _port, cancellationToken);

            // Read the SMTP banner
            using var stream = client.GetStream();
            var buffer = new byte[1024];
            var bytesRead = await stream.ReadAsync(buffer, cancellationToken);
            var banner = Encoding.ASCII.GetString(buffer, 0, bytesRead);

            return banner.StartsWith("220")
                ? HealthCheckResult.Healthy("SMTP server ready")
                : HealthCheckResult.Unhealthy($"Unexpected banner: {banner}");
        }
        catch (Exception ex)
        {
            return HealthCheckResult.Unhealthy(
                "Cannot connect to SMTP server",
                exception: ex);
        }
    }
}
```

### Best Practices

| Practice | Rationale |
|---|---|
| Keep checks fast (< 2 seconds) | Slow checks delay the polling loop and startup |
| Check functional readiness, not just connectivity | A TCP connection doesn't mean the service is ready to handle requests |
| Handle exceptions gracefully | Return `Unhealthy` with details rather than throwing |
| Use cancellation tokens | Respect timeouts and shutdown signals |
| Avoid side effects | Health checks should read state, not modify it |

## The Health Check Lifecycle

The complete flow from registration to dashboard display:

```
1. AddRedis("cache") registers HealthCheckAnnotation
2. DCP starts the Redis container
3. Health check polling begins for "cache"
4. Poll 1: Unhealthy (container still initializing)
5. ResourceNotificationService publishes: cache = Starting
6. Dashboard shows: cache ○ Starting
7. Poll 2: Healthy (Redis accepting connections)
8. ResourceNotificationService publishes: cache = Running
9. Dashboard shows: cache ● Running
10. WaitFor dependents are released
11. Polling continues at regular intervals
12. If Redis crashes: poll returns Unhealthy
13. ResourceNotificationService publishes: cache = RuntimeUnhealthy
14. Dashboard shows: cache ⚠ Unhealthy
```

## Summary

Health checks are the bridge between "a process is running" and "a service is ready." Aspire's health check system builds on ASP.NET Core's `IHealthCheck` framework, adds automatic registration through hosting integrations, and integrates tightly with `ResourceNotificationService`, the `WaitFor` dependency system, and the dashboard. By understanding how health checks are discovered, polled, mapped to resource states, and displayed, you can build more reliable distributed applications that only start services when their dependencies are genuinely ready.
