# 9.2.3 — The Aspire Eventing System

## Introduction

While lifecycle hooks give you global control at three fixed phases, the Aspire **eventing system** provides fine-grained, per-resource control over the application lifecycle. Using `IDistributedApplicationEventing`, you can subscribe to specific events on individual resources — enabling targeted customization that lifecycle hooks cannot achieve.

## Eventing vs Lifecycle Hooks

Before diving in, it's important to understand when to use each:

| Feature | Lifecycle Hooks | Eventing System |
|---|---|---|
| Fires for | Entire application (once per phase) | Individual resources or global events |
| Granularity | 3 fixed phases | Many event types, can target specific resources |
| Access | Full `DistributedApplicationModel` | Event-specific data and the target resource |
| Registration | `builder.Services.AddLifecycleHook<T>()` | `builder.Eventing.Subscribe<TEvent>(...)` |
| Best for | App-wide validation, logging, seeding | Per-resource customization, conditional behavior |

The eventing system complements lifecycle hooks — use hooks for global concerns and events for resource-specific behavior.

## Subscribing to Events

The eventing system is accessed through `builder.Eventing`. You subscribe to an event type, optionally scoped to a specific resource:

```csharp
var builder = DistributedApplication.CreateBuilder(args);

var db = builder.AddPostgres("db");

// Subscribe to an event for a specific resource
builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    db,
    async (evt, cancellationToken) =>
    {
        // This runs before the 'db' resource starts
        Console.WriteLine($"About to start: {evt.Resource.Name}");
    });
```

The subscribe method returns a `DistributedApplicationEventSubscription` that can be used to unsubscribe later if needed.

## BeforeResourceStartedEvent

The most commonly used event is `BeforeResourceStartedEvent`. It fires **before each individual resource starts**, giving you a chance to customize, validate, or delay that specific resource:

```csharp
builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    myResource,
    async (evt, cancellationToken) =>
    {
        // Perform per-resource initialization
        // This runs after endpoints are allocated but before the resource starts
        await InitializeResourceAsync(evt.Resource, cancellationToken);
    });
```

### Use Cases for BeforeResourceStartedEvent

**Per-resource configuration:**

```csharp
var api = builder.AddProject<Projects.Api>("api");

builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    api,
    async (evt, cancellationToken) =>
    {
        // Inject a dynamic configuration value
        // based on external state
        var featureFlags = await LoadFeatureFlagsAsync(cancellationToken);

        evt.Resource.Annotations.Add(
            new EnvironmentCallbackAnnotation(context =>
            {
                context.EnvironmentVariables["FEATURE_FLAGS"] =
                    string.Join(",", featureFlags);
            }));
    });
```

**Conditional resource startup:**

```csharp
builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    expensiveService,
    async (evt, cancellationToken) =>
    {
        // Check if we actually need this service
        if (!ShouldStartExpensiveService())
        {
            // Log a warning — the resource will still start,
            // but you can take preparatory action
            Console.WriteLine(
                "Warning: Starting expensive service in dev mode");
        }
    });
```

**Database schema migration:**

```csharp
var db = builder.AddPostgres("db").AddDatabase("appdb");
var api = builder.AddProject<Projects.Api>("api")
    .WithReference(db);

builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    api,
    async (evt, cancellationToken) =>
    {
        // Run migrations before the API starts
        // The database is already running (due to WaitFor ordering)
        var connectionString = await db.Resource
            .GetConnectionStringAsync(cancellationToken);

        await RunMigrationsAsync(connectionString!, cancellationToken);
    });
```

## Global Event Subscriptions

You can also subscribe to events globally — without scoping to a specific resource. This fires the callback for **every** resource that triggers the event:

```csharp
builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    async (evt, cancellationToken) =>
    {
        // Fires before EVERY resource starts
        Console.WriteLine($"Starting resource: {evt.Resource.Name}");
    });
```

This is useful for cross-cutting concerns like logging or metrics collection.

## Event Execution Order

When both lifecycle hooks and events are registered, they execute in a predictable order:

```
1. BeforeStartAsync           (lifecycle hooks — global)
2. DCP allocates endpoints
3. AfterEndpointsAllocatedAsync (lifecycle hooks — global)
4. For each resource:
   a. BeforeResourceStartedEvent  (eventing — per resource)
   b. DCP starts the resource
5. AfterResourcesCreatedAsync  (lifecycle hooks — global)
```

This means `BeforeResourceStartedEvent` fires **after** `AfterEndpointsAllocatedAsync` but **before** `AfterResourcesCreatedAsync`. Endpoints are already allocated when your event handler runs.

## Practical Example: Audit Logging

Here's a complete example that uses the eventing system to log detailed startup information for each resource:

```csharp
var builder = DistributedApplication.CreateBuilder(args);

var cache = builder.AddRedis("cache");
var db = builder.AddPostgres("pg").AddDatabase("appdb");
var api = builder.AddProject<Projects.Api>("api")
    .WithReference(cache)
    .WithReference(db);

// Global event: log every resource start
builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    async (evt, cancellationToken) =>
    {
        var resource = evt.Resource;
        var endpointCount = resource.Annotations
            .OfType<EndpointAnnotation>()
            .Count();

        Console.WriteLine(
            $"[AUDIT] Starting '{resource.Name}' " +
            $"(type: {resource.GetType().Name}, " +
            $"endpoints: {endpointCount})");
    });

// Resource-specific event: special handling for API
builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    api,
    async (evt, cancellationToken) =>
    {
        Console.WriteLine("[AUDIT] API resource starting — verifying dependencies...");

        // At this point, cache and db should already be running
        // because of WaitFor ordering
    });

builder.Build().Run();
```

## Publishing Custom Events

The eventing system also supports publishing custom events. You can define your own event types and publish them:

```csharp
// Define a custom event
public class DatabaseSeededEvent : IDistributedApplicationEvent
{
    public required IResource Resource { get; init; }
    public required int RecordCount { get; init; }
}
```

This extensibility enables you to build event-driven workflows within your AppHost.

## Combining Events with ResourceNotificationService

Events and `ResourceNotificationService` work at different levels but complement each other:

- **Events** fire at specific lifecycle points (before start, etc.)
- **ResourceNotificationService** continuously streams state changes (Running, Exited, etc.)

You might use an event to prepare a resource before it starts, then use `ResourceNotificationService` to monitor it continuously afterward:

```csharp
// Before start: prepare the resource
builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    myResource,
    async (evt, cancellationToken) =>
    {
        await PrepareResourceAsync(evt.Resource, cancellationToken);
    });

// After start: monitor continuously
builder.Services.AddHostedService<ResourceMonitor>();
// (ResourceMonitor subscribes to ResourceNotificationService.WatchAsync)
```

## Multiple Subscriptions

You can register multiple handlers for the same event and resource. They execute in registration order:

```csharp
builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    api,
    async (evt, ct) =>
    {
        Console.WriteLine("Handler 1: Validating configuration...");
    });

builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    api,
    async (evt, ct) =>
    {
        Console.WriteLine("Handler 2: Running migrations...");
    });

// Output order:
// Handler 1: Validating configuration...
// Handler 2: Running migrations...
```

## Error Handling

If an event handler throws an exception, it prevents the resource from starting. This is by design — if your pre-start validation or migration fails, the resource should not start in a broken state:

```csharp
builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
    api,
    async (evt, cancellationToken) =>
    {
        var isReady = await CheckPrerequisitesAsync(cancellationToken);
        if (!isReady)
        {
            throw new InvalidOperationException(
                "Prerequisites not met — cannot start API");
        }
    });
```

The error will be surfaced in the dashboard and the resource will show a `FailedToStart` state.

## Summary

The Aspire eventing system provides per-resource lifecycle control through a subscribe/publish pattern. The `BeforeResourceStartedEvent` is the primary event, letting you run custom logic before each resource starts. Combined with lifecycle hooks for global concerns and `ResourceNotificationService` for continuous monitoring, the eventing system completes the toolkit for controlling how your Aspire application starts and behaves.
