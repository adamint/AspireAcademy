# 9.4.1 — Service Discovery Internals

## Introduction

When you call `builder.AddProject<Projects.Frontend>("frontend").WithReference(api)`, the frontend can magically call `http://api` and reach the API service — even though the API is running on a random localhost port. This is **service discovery** in action. This lesson explains every step of the resolution pipeline, from environment variable injection in the AppHost to the `HttpClient` interceptor in the consuming service.

## The Resolution Pipeline

Service discovery in Aspire is a multi-step pipeline:

```
Step 1: AppHost injects service endpoint environment variables
             │
             ▼
Step 2: Consuming service calls AddServiceDiscovery()
             │
             ▼
Step 3: HttpClient request targets http://resourcename
             │
             ▼
Step 4: ServiceDiscoveryHttpClientHandler intercepts the request
             │
             ▼
Step 5: ConfigurationServiceEndpointProvider resolves the name
             │
             ▼
Step 6: Resolved endpoint URL replaces the service name
             │
             ▼
Step 7: HTTP request goes to the actual endpoint
```

Let's walk through each step in detail.

## Step 1: Environment Variable Injection

When the AppHost starts, DCP computes the actual endpoints for each resource (including dynamically allocated ports) and injects them as environment variables into every resource that references them.

The environment variable naming convention is:

```
services__{servicename}__{scheme}__{index}
```

For example, if the `api` resource has HTTP and HTTPS endpoints:

```bash
services__api__http__0=http://localhost:5123
services__api__https__0=https://localhost:7123
```

For resources with replicas, multiple indexed entries are created:

```bash
services__api__http__0=http://localhost:5201
services__api__http__1=http://localhost:5202
services__api__http__2=http://localhost:5203
```

These environment variables are the foundation of service discovery — they map a logical service name to physical addresses.

## Step 2: AddServiceDiscovery in ServiceDefaults

In each service project, the `ServiceDefaults` extension method typically calls `AddServiceDiscovery()`:

```csharp
// In ServiceDefaults/Extensions.cs
public static IHostApplicationBuilder AddServiceDefaults(
    this IHostApplicationBuilder builder)
{
    // ... other configuration ...

    builder.Services.AddServiceDiscovery();

    builder.Services.ConfigureHttpClientDefaults(http =>
    {
        http.AddServiceDiscovery();
    });

    return builder;
}
```

This registers:
- The service discovery infrastructure in the DI container
- A delegating handler on all `HttpClient` instances that intercepts requests and resolves service names

## Step 3: Making a Request to a Service Name

In your application code, you use the service name as the hostname:

```csharp
public class WeatherService(HttpClient client)
{
    public async Task<Weather[]> GetWeatherAsync()
    {
        // "api" is the resource name, not a real hostname
        var response = await client.GetAsync("http://api/weather");
        return await response.Content.ReadFromJsonAsync<Weather[]>();
    }
}
```

The URL `http://api/weather` uses `api` as the hostname — which doesn't resolve in DNS. This is where the service discovery handler takes over.

## Step 4: ServiceDiscoveryHttpClientHandler Interception

The `AddServiceDiscovery()` call on `HttpClient` registers a `DelegatingHandler` that intercepts every outgoing HTTP request. Before the request reaches the network:

```
HttpClient.SendAsync("http://api/weather")
        │
        ▼
┌───────────────────────────────────┐
│ ServiceDiscovery DelegatingHandler│
│                                   │
│ 1. Extract hostname: "api"        │
│ 2. Query endpoint providers       │
│ 3. Resolve to actual endpoint     │
│ 4. Rewrite URL                    │
└───────────────────────────────────┘
        │
        ▼
HttpClient.SendAsync("http://localhost:5123/weather")
```

The handler extracts the hostname from the request URI, resolves it using the registered endpoint providers, and rewrites the URL before the request is sent.

## Step 5: ConfigurationServiceEndpointProvider

The default endpoint provider in Aspire is `ConfigurationServiceEndpointProvider`. It reads the `services__*` environment variables (which are loaded into `IConfiguration` by the .NET configuration system) and builds a map of service names to endpoints:

```
Configuration key                          Value
─────────────────                          ─────
services:api:http:0                        http://localhost:5123
services:api:https:0                       https://localhost:7123
services:cache:tcp:0                       localhost:6379
```

Note that the `__` double-underscore in environment variable names is automatically translated to `:` (colon) by the .NET configuration system, which is why the configuration key uses colons.

When a service name like `api` is queried:
1. The provider looks up `services:api:https:*` entries first (preferring HTTPS)
2. Falls back to `services:api:http:*` entries
3. Returns all matching endpoints as `ServiceEndpoint` instances

## Step 6: URL Rewriting

Once the endpoint is resolved, the handler rewrites the request URL:

```
Original:  http://api/weather
Resolved:  https://localhost:7123/weather
```

The scheme may also change — if the service was referenced as `http://api` but the provider resolved an HTTPS endpoint, the scheme is upgraded.

## Scheme Resolution

Service discovery handles scheme selection with a priority system:

| Request Scheme | Available Endpoints | Resolution |
|---|---|---|
| `http://api` | HTTP and HTTPS | Prefers HTTPS |
| `https://api` | HTTPS only | Uses HTTPS |
| `http://api` | HTTP only | Uses HTTP |
| `http+tcp://api` | Any TCP endpoint | Uses raw TCP |

You can be explicit about scheme preference:

```csharp
// Force HTTP (no HTTPS upgrade)
var response = await client.GetAsync("http://_http.api/weather");

// Explicitly request HTTPS
var response = await client.GetAsync("https://api/weather");
```

## DNS-Based Resolution (Fallback)

If the configuration-based provider cannot resolve a service name, Aspire falls back to **DNS-based resolution**. This is useful in Kubernetes or other environments where services are registered in DNS:

```
1. ConfigurationServiceEndpointProvider → not found
2. DNS resolution → api.default.svc.cluster.local → 10.0.0.42:80
3. Request sent to resolved IP
```

In local development, DNS fallback is rarely needed because the AppHost always injects the environment variables.

## Load Balancing with Replicas

When a resource has replicas, service discovery returns multiple endpoints:

```bash
services__api__http__0=http://localhost:5201
services__api__http__1=http://localhost:5202
services__api__http__2=http://localhost:5203
```

The service discovery system selects one endpoint per request using a load-balancing strategy:

| Strategy | Behavior |
|---|---|
| **Round-robin** | Cycles through endpoints in order (1 → 2 → 3 → 1 → ...) |
| **Random** | Selects a random endpoint for each request |
| **Power-of-two-choices** | Picks two random endpoints and selects the one with fewer active connections |

The default strategy distributes requests across all available replicas.

## ServiceEndpoint and Resolution Types

The service discovery system uses several key types:

### ServiceEndpoint

Represents a resolved endpoint with an address, port, and metadata:

```csharp
public class ServiceEndpoint
{
    public IPEndPoint EndPoint { get; }
    // Additional metadata about the endpoint
}
```

### ServiceEndpointResolver

Coordinates between multiple endpoint providers to resolve a service name:

```csharp
// The resolver queries all registered providers in priority order
// and aggregates the results
var endpoints = await resolver.GetEndpointsAsync("api", cancellationToken);
```

### Provider Pipeline

Multiple providers can be registered, and they are queried in order:

```
1. ConfigurationServiceEndpointProvider  (environment variables)
2. DnsServiceEndpointProvider            (DNS SRV records)
3. PassThroughServiceEndpointProvider    (raw hostnames)
```

The first provider to return results wins.

## How WithReference Wires It Up

When you write `.WithReference(api)` in the AppHost, several things happen at build-time:

1. An `EnvironmentCallbackAnnotation` is added to the consuming resource
2. At run-time, the callback injects `services__api__http__0` (and other endpoints) as environment variables
3. The consuming service loads these via `IConfiguration`
4. The service discovery handler reads them to resolve `http://api` requests

```csharp
// In AppHost
var api = builder.AddProject<Projects.Api>("api");
var frontend = builder.AddProject<Projects.Frontend>("frontend")
    .WithReference(api);  // Injects services__api__http__0, services__api__https__0
```

## Debugging Service Discovery

When service discovery isn't working, check these common issues:

**1. Environment variables not set:**
```bash
# Check what's injected into your service
echo $services__api__http__0
# Should output something like: http://localhost:5123
```

**2. AddServiceDiscovery not called:**
```csharp
// Ensure ServiceDefaults includes:
builder.Services.AddServiceDiscovery();
builder.Services.ConfigureHttpClientDefaults(http =>
{
    http.AddServiceDiscovery();
});
```

**3. Named HttpClient missing service discovery:**
```csharp
// If using named/typed HttpClients, ensure they have service discovery:
builder.Services.AddHttpClient<WeatherService>(client =>
{
    client.BaseAddress = new Uri("http://api");
})
.AddServiceDiscovery();
```

**4. Check the dashboard:**
The Aspire dashboard shows all environment variables injected into each resource — verify the `services__*` variables are present and have correct values.

## Summary

Aspire's service discovery replaces raw connection strings and hardcoded URLs with a clean, name-based resolution system. The AppHost injects endpoint information as structured environment variables. The consuming service's `HttpClient` interceptor resolves these names at request time. This indirection means your application code never needs to know the actual ports or addresses — it just uses logical service names, and the service discovery pipeline handles the rest.
