# RemoteHost Architecture

The RemoteHost is a .NET process that bridges polyglot AppHosts to the Aspire hosting SDK. It has four key components.

---

## RemoteHostServer

The entry point. `RemoteHostServer.RunAsync(args)` starts the server:
1. Loads hosting integration assemblies (Aspire.Hosting.Redis, Aspire.Hosting.PostgreSQL, etc.)
2. Starts the `JsonRpcServer` listening on a Unix socket
3. Registers all `[AspireExport]` capabilities with the `CapabilityDispatcher`
4. Waits for the polyglot AppHost to connect

## JsonRpcServer

Listens on a Unix domain socket (Linux/macOS) or named pipe (Windows):

```
~/.aspire/remote-app-host.sock  (or REMOTE_APP_HOST_SOCKET_PATH)
```

Accepts JSON-RPC 2.0 messages over the socket. Supports:
- `invokeCapability(capabilityId, args)` — call a .NET method
- `getCapabilities()` — list all available capabilities
- `cancelToken(tokenId)` — cancel an async operation

## CapabilityDispatcher

Routes incoming RPC calls to .NET methods. It scans loaded assemblies for types marked with `[AspireExport]` and builds a lookup table:

```
"createBuilder" → DistributedApplication.CreateBuilder()
"addRedis"      → RedisBuilderExtensions.AddRedis()
"addPostgres"   → PostgresBuilderExtensions.AddPostgres()
"withReference"  → ResourceBuilderExtensions.WithReference()
```

When a TypeScript AppHost calls `builder.addRedis("cache")`, the dispatcher:
1. Looks up the capability `"addRedis"`
2. Resolves the builder handle from the HandleRegistry
3. Calls the .NET `AddRedis` extension method
4. Registers the returned `IResourceBuilder<RedisResource>` in the HandleRegistry
5. Returns the new handle ID to TypeScript

## HandleRegistry

Manages opaque object references. .NET objects (builders, resources, annotations) are registered with string IDs like `"handle-1"`, `"handle-2"`. The polyglot AppHost only sees these IDs — never the actual .NET objects.

This is how a TypeScript `const cache` becomes a reference to a .NET `IResourceBuilder<RedisResource>` without TypeScript needing to understand .NET types.

> 🐛 **Common Gotcha:** The HandleRegistry uses simple incrementing IDs (handle-1, handle-2, ...). These are **not** stable across restarts. Never persist handle IDs or use them for anything beyond a single AppHost session.
