# 9.1.2 — DCP Resource Management

## Introduction

The Developer Control Plane (DCP) is responsible for the full lifecycle of every resource in your Aspire application — from initial creation to final teardown. This lesson explores how DCP translates the abstract app model into concrete running infrastructure.

## The Translation Pipeline

When you call `builder.Build().Run()`, a translation pipeline converts your C# app model into DCP resource objects:

```
  C# App Model                    DCP Resources               Running Infrastructure
┌──────────────┐              ┌──────────────────┐           ┌────────────────────┐
│ AddProject   │──────────▶   │ Executable       │──────▶    │ dotnet process     │
│ AddContainer │──────────▶   │ Container        │──────▶    │ Docker container   │
│ AddRedis     │──────────▶   │ Container        │──────▶    │ Redis container    │
│ WithReplicas │──────────▶   │ ExecutableRepSet  │──────▶   │ N dotnet processes │
└──────────────┘              └──────────────────┘           └────────────────────┘
```

Each Aspire resource type maps to a specific DCP resource:

| Aspire Resource | DCP Resource Type | Materialized As |
|---|---|---|
| `AddProject<T>(...)` | `Executable` | `dotnet run` process |
| `AddContainer(...)` | `Container` | Docker/Podman container |
| `AddRedis(...)` | `Container` | Redis Docker container |
| `AddPostgres(...)` | `Container` | PostgreSQL Docker container |
| `AddProject(...).WithReplicas(N)` | `ExecutableReplicaSet` | N `dotnet run` processes |
| `AddDockerfile(...)` | `Container` | Built and run Docker container |

## Container Resource Lifecycle

When DCP receives a `Container` resource definition, it executes the following steps:

### 1. Image Resolution

DCP determines the container image to use. For hosting integrations, the image is preconfigured (e.g., `docker.io/library/redis:7.4` for Redis). DCP checks if the image exists locally:

- **Image cached**: Proceed to container creation
- **Image not cached**: Pull the image from the registry

### 2. Container Creation

DCP creates the container with the full configuration derived from the app model:

- **Environment variables**: Connection strings, service discovery URLs, configuration values
- **Port bindings**: Mapped from `WithEndpoint` or default integration ports
- **Volume mounts**: From `WithBindMount` or `WithVolume` calls
- **Entrypoint/command overrides**: From `WithEntrypoint` or `WithArgs`
- **Network configuration**: Container-to-container networking via Docker networks

### 3. Container Start

DCP starts the container and begins monitoring its state. The container transitions through states:

```
Created → Starting → Running → (Stopping → Stopped | Exited)
                                       ↑
                                  FailedToStart
```

### 4. Health Monitoring

If the resource has health checks registered (via `WithHealthCheck` or built-in integration health checks), DCP coordinates with the AppHost's health check polling system to track readiness.

### 5. Restart Handling

If a container exits unexpectedly, DCP can restart it based on the configured behavior. The `WaitBehavior` enum controls how dependent resources react:

```csharp
var api = builder.AddProject<Projects.Api>("api")
    .WaitFor(redis);
```

| WaitBehavior | Effect |
|---|---|
| `WaitOnResourceUnavailable` | Dependent resource waits and retries when the dependency becomes unavailable |
| `StopOnResourceUnavailable` | Dependent resource stops if the dependency becomes unavailable |

## Executable Resource Lifecycle

Project resources follow a similar but distinct lifecycle:

### 1. Build Resolution

DCP determines the executable path, working directory, and launch arguments. For .NET projects, this typically means invoking `dotnet run` with the appropriate project path.

### 2. Environment Variable Injection

Before starting the process, DCP injects all computed environment variables. These include:

```bash
# Connection strings
ConnectionStrings__cache=localhost:63791
ConnectionStrings__db=Host=localhost;Port=54321;...

# Service discovery
services__api__https__0=https://localhost:7123
services__api__http__0=http://localhost:5123

# OTEL configuration
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:18889
DOTNET_DASHBOARD_OTLP_ENDPOINT_URL=http://localhost:18889

# Aspire-specific
DOTNET_LAUNCH_PROFILE=https
```

### 3. Process Start

DCP launches the process and captures stdout/stderr for relay to the dashboard's console logs.

### 4. Port Allocation

DCP allocates ephemeral ports for the executable's endpoints. When you define an endpoint:

```csharp
var api = builder.AddProject<Projects.Api>("api")
    .WithHttpsEndpoint(port: 5001);          // fixed port
    .WithHttpEndpoint();                      // DCP allocates a random port
```

If no port is specified, DCP assigns an available port from the ephemeral range.

## ExecutableReplicaSet: Scaling Locally

When `.WithReplicas(N)` is called, DCP creates an `ExecutableReplicaSet` instead of a single `Executable`. This manages N identical process instances:

```csharp
var api = builder.AddProject<Projects.WeatherApi>("api")
    .WithReplicas(3);
```

DCP handles:
- **Unique port allocation** for each replica — every instance gets its own HTTP and HTTPS ports
- **Independent process management** — each replica is started, monitored, and can be restarted independently
- **Service discovery coordination** — all replica endpoints are published so load balancing can distribute traffic

In the dashboard, you see each replica as a separate entry:

```
api-0   Running   https://localhost:7101
api-1   Running   https://localhost:7102
api-2   Running   https://localhost:7103
```

## The Communication Flow

The full flow from AppHost to running container:

```
┌──────────┐     gRPC      ┌───────┐    Docker API    ┌─────────────────┐
│ AppHost  │ ──────────▶   │  DCP  │ ──────────────▶  │ Container       │
│          │               │       │                   │ Runtime         │
│ Submits  │               │ Pulls │                   │ (Docker/Podman) │
│ resource │  State updates│ image,│                   │                 │
│ defs     │ ◀──────────── │starts │  Container events │ Creates and     │
│          │               │ ctnr  │ ◀──────────────── │ runs containers │
└──────────┘               └───────┘                   └─────────────────┘
```

1. **AppHost → DCP** (gRPC): Submit resource definitions (Container, Executable, etc.)
2. **DCP → Container Runtime** (Docker/Podman API): Pull images, create containers, start processes
3. **Container Runtime → DCP** (Events): Container state changes (started, stopped, crashed)
4. **DCP → AppHost** (gRPC): Resource state updates flow back to `ResourceNotificationService`
5. **AppHost → Dashboard** (OTLP/gRPC): State is relayed to the dashboard UI

## Port Allocation Strategy

DCP uses an intelligent port allocation strategy:

1. **Explicit ports**: If you specify a port (e.g., `.WithHttpEndpoint(port: 5001)`), DCP uses that port. A conflict results in a startup error.
2. **Random allocation**: If no port is specified, DCP selects an available ephemeral port. This is the default and recommended approach to avoid conflicts.
3. **Proxy ports**: For containers, DCP may allocate a proxy port on the host that maps to the container's internal port.

```csharp
// Container exposes port 6379 internally; DCP allocates a random host port
var redis = builder.AddRedis("cache");

// The actual host port is computed at runtime and injected
// into dependent resources via connection strings
```

## Resource Ordering and Dependencies

DCP respects the dependency graph when starting resources. If resource B depends on resource A via `WaitFor`:

```csharp
var db = builder.AddPostgres("db");
var api = builder.AddProject<Projects.Api>("api")
    .WaitFor(db);
```

DCP ensures:
1. The PostgreSQL container starts first
2. The health check (if any) passes
3. Only then does the API executable start

This ordering prevents services from crashing because their dependencies are not yet available.

## Shutdown and Cleanup

When the AppHost process exits (Ctrl+C or programmatic shutdown):

1. DCP receives a shutdown signal
2. **Session-scoped** containers (`ContainerLifetime.Session`) are stopped and removed
3. **Persistent** containers (`ContainerLifetime.Persistent`) are left running
4. All executables are terminated
5. DCP itself exits
6. Port allocations are released

This ensures clean resource management with no leaked containers or orphaned processes in the default configuration.

## Summary

DCP is the execution engine that bridges the gap between your declarative C# app model and the actual running infrastructure. It handles every detail — image pulling, container creation, process launching, port allocation, health monitoring, and clean shutdown — so that you can focus on defining *what* your distributed application looks like rather than *how* to run it.
