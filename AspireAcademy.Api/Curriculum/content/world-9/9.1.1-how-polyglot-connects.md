# How Polyglot AppHosts Connect

You have been writing AppHosts in TypeScript and Python since World 1. They felt native — `createBuilder()`, `addRedis()`, `withReference()`. But under the hood, every polyglot AppHost call is a remote procedure call to the .NET hosting SDK.

---

## The Architecture

```
┌──────────────────────┐         JSON-RPC          ┌────────────────────────┐
│  Your AppHost         │    over Unix Socket       │  RemoteHost Process    │
│  (TypeScript/Python/  │ ═══════════════════════▶  │  (.NET)                │
│   Go/Java/Rust)       │                           │                        │
│                       │ ◀═══════════════════════  │  CapabilityDispatcher  │
│  .modules/aspire.js   │      Responses            │  HandleRegistry        │
│  aspire_app.py        │                           │  AtsMarshaller         │
└──────────────────────┘                            └───────────┬────────────┘
                                                                │
                                                    ┌───────────▼────────────┐
                                                    │  Aspire.Hosting SDK    │
                                                    │  (DistributedApp       │
                                                    │   Builder, DCP, etc.)  │
                                                    └────────────────────────┘
```

When you write `const builder = await createBuilder()` in TypeScript:

1. The generated `.modules/aspire.js` module connects to the **RemoteHost** process via a Unix domain socket
2. It sends a JSON-RPC message: `invokeCapability("createBuilder", {})`
3. The RemoteHost's **CapabilityDispatcher** routes the call to `DistributedApplication.CreateBuilder()`
4. A `DistributedApplicationBuilder` is created on the .NET side
5. The builder is registered in the **HandleRegistry** with an opaque ID (e.g., `"handle-1"`)
6. The handle ID is returned to TypeScript
7. `.modules/aspire.js` wraps it in a TypeScript class with typed methods
8. Subsequent calls like `builder.addRedis("cache")` send `invokeCapability("addRedis", { handle: "handle-1", name: "cache" })`

Every method call is a network round-trip. But the latency is negligible because it is a local Unix socket — microseconds per call.

## Why This Architecture?

The alternative would be to reimplement the entire Aspire hosting SDK in every language. That would mean:
- Duplicating the resource model in TypeScript, Python, Go, Java, and Rust
- Keeping all six implementations in sync with every Aspire release
- Reimplementing DCP communication, health checks, annotations, and publishing

Instead, the RemoteHost bridge lets every language use the **same** .NET hosting SDK. New features added to the SDK are immediately available in all languages through code generation.
