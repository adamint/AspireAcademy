# How the TypeScript AppHost Connects

You have been writing AppHosts in TypeScript since World 1. It felt native — `createBuilder()`, `addRedis()`, `withReference()`. But under the hood, every TypeScript AppHost call is a remote procedure call to the .NET hosting SDK.

When you write `const builder = await createBuilder()` in TypeScript:

1. The generated `.modules/aspire.js` module connects to the **RemoteHost** process via a Unix domain socket
2. It sends a JSON-RPC message: `invokeCapability("createBuilder", {})`
3. The RemoteHost's **CapabilityDispatcher** routes the call to `DistributedApplication.CreateBuilder()`
4. A `DistributedApplicationBuilder` is created on the .NET side
5. The builder is registered in the **HandleRegistry** with an opaque ID (e.g., `"handle-1"`)
6. The handle ID is returned to TypeScript
7. `.modules/aspire.js` wraps it in a TypeScript class with typed methods
8. Subsequent calls like `builder.addRedis("cache")` send `invokeCapability("addRedis", { handle: "handle-1", name: "cache" })`

Every method call is a network round-trip. But the latency is negligible because it is a local Unix socket — microseconds per call.

```reveal
{"question": "Why does the RemoteHost use opaque handle IDs instead of serializing full .NET objects to TypeScript?", "answer": "Serializing full .NET objects would require TypeScript to understand .NET's type system — generics, interfaces, inheritance, extension methods. Handle IDs keep the boundary clean: TypeScript only sees string references, and all the complex type logic stays on the .NET side. This also means .NET objects can hold state (open connections, running tasks) that wouldn't survive serialization."}
```

> 🎲 **Fun Fact:** The Unix socket communication adds less than 10ms total overhead for a typical AppHost with 10 resources (~50 RPC calls). That's less than the time it takes to pull a single container layer.

## Why This Architecture?

The alternative would be to reimplement the entire Aspire hosting SDK in TypeScript. That would mean:
- Duplicating the resource model
- Keeping the TypeScript implementation in sync with every Aspire release
- Reimplementing DCP communication, health checks, annotations, and publishing

Instead, the RemoteHost bridge lets TypeScript use the **same** .NET hosting SDK. New features added to the SDK are immediately available through code generation.
