# 9.2.2 — IDistributedApplicationLifecycleHook

## Introduction

`IDistributedApplicationLifecycleHook` is an interface that lets you inject custom logic at key points in the Aspire application lifecycle. Lifecycle hooks execute **globally** — they fire once for the entire application, not per-resource — and give you access to the full `DistributedApplicationModel` so you can inspect, modify, or react to the state of all resources.

## The Interface

The interface defines three methods, each corresponding to a distinct lifecycle phase:

```csharp
public interface IDistributedApplicationLifecycleHook
{
    Task BeforeStartAsync(
        DistributedApplicationModel appModel,
        CancellationToken cancellationToken = default);

    Task AfterEndpointsAllocatedAsync(
        DistributedApplicationModel appModel,
        CancellationToken cancellationToken = default);

    Task AfterResourcesCreatedAsync(
        DistributedApplicationModel appModel,
        CancellationToken cancellationToken = default);
}
```

All methods have default implementations that return `Task.CompletedTask`, so you only need to override the phases you care about.

## The Three Lifecycle Phases

### Phase 1: BeforeStartAsync

**When it fires:** After the app model is fully built but before DCP begins creating any resources.

**What you have access to:** The complete `DistributedApplicationModel` with all resources and their annotations. No ports are allocated, no containers are running, no processes are started.

**Use cases:**
- **Validation** — verify that required resources exist, naming conventions are followed, or configurations are valid
- **Dynamic resource addition** — programmatically add resources based on configuration or environment
- **App model modification** — add annotations, modify environment variables, or adjust endpoints
- **Logging** — record the planned resource graph before execution begins

```csharp
public Task BeforeStartAsync(
    DistributedApplicationModel appModel,
    CancellationToken cancellationToken = default)
{
    // Validate that every project resource has a health check
    foreach (var resource in appModel.Resources.OfType<ProjectResource>())
    {
        var hasHealthCheck = resource.Annotations
            .OfType<HealthCheckAnnotation>()
            .Any();

        if (!hasHealthCheck)
        {
            throw new InvalidOperationException(
                $"Resource '{resource.Name}' is missing a health check annotation.");
        }
    }

    return Task.CompletedTask;
}
```

### Phase 2: AfterEndpointsAllocatedAsync

**When it fires:** After DCP has allocated ports for all endpoints but before processes and containers are started.

**What you have access to:** The app model with **resolved endpoints** — you now know the actual ports and URLs each resource will use.

**Use cases:**
- **Endpoint-dependent configuration** — configure services that need to know their own or other services' endpoints before starting
- **External registration** — register endpoints with an external service registry or reverse proxy
- **Dynamic environment variables** — inject environment variables that depend on allocated ports

```csharp
public Task AfterEndpointsAllocatedAsync(
    DistributedApplicationModel appModel,
    CancellationToken cancellationToken = default)
{
    // Log all allocated endpoints
    foreach (var resource in appModel.Resources)
    {
        var endpoints = resource.Annotations
            .OfType<EndpointAnnotation>();

        foreach (var endpoint in endpoints)
        {
            Console.WriteLine(
                $"Resource '{resource.Name}' endpoint '{endpoint.Name}': " +
                $"port {endpoint.AllocatedEndpoint?.Port}");
        }
    }

    return Task.CompletedTask;
}
```

### Phase 3: AfterResourcesCreatedAsync

**When it fires:** After all resources have been created and started by DCP.

**What you have access to:** The fully materialized app model — containers are running, processes are started, and endpoints are live.

**Use cases:**
- **Database seeding** — populate databases with initial data after they're running
- **Post-startup initialization** — call APIs to configure services that are now live
- **Integration verification** — run smoke tests to verify the system is correctly wired
- **Notification** — alert external systems that the dev environment is ready

```csharp
public async Task AfterResourcesCreatedAsync(
    DistributedApplicationModel appModel,
    CancellationToken cancellationToken = default)
{
    // Seed a database after it's running
    var dbResource = appModel.Resources
        .OfType<PostgresDatabaseResource>()
        .FirstOrDefault(r => r.Name == "appdb");

    if (dbResource is not null)
    {
        var connectionString = await dbResource
            .GetConnectionStringAsync(cancellationToken);

        await SeedDatabaseAsync(connectionString!, cancellationToken);
    }
}
```

## Lifecycle Phase Timeline

The phases execute in a strict order within the application startup:

```
builder.Build().Run()
        │
        ▼
┌──────────────────────────┐
│   Build App Model        │  ◀── C# code defines resources
└──────────┬───────────────┘
           ▼
┌──────────────────────────┐
│  BeforeStartAsync        │  ◀── Hook Phase 1
│  (validate, modify model)│
└──────────┬───────────────┘
           ▼
┌──────────────────────────┐
│  DCP: Allocate Endpoints │  ◀── Ports assigned
└──────────┬───────────────┘
           ▼
┌──────────────────────────┐
│ AfterEndpointsAllocated  │  ◀── Hook Phase 2
│  (endpoints now known)   │
└──────────┬───────────────┘
           ▼
┌──────────────────────────┐
│  DCP: Create Resources   │  ◀── Containers/processes start
└──────────┬───────────────┘
           ▼
┌──────────────────────────┐
│ AfterResourcesCreated    │  ◀── Hook Phase 3
│  (everything is running) │
└──────────┬───────────────┘
           ▼
┌──────────────────────────┐
│   Application Running    │  ◀── Dashboard active, resources live
└──────────────────────────┘
```

## Registering a Lifecycle Hook

Register your hook using the `AddLifecycleHook<T>` extension method on the builder's service collection:

```csharp
var builder = DistributedApplication.CreateBuilder(args);

builder.Services.AddLifecycleHook<MyCustomHook>();

// ... add resources ...

builder.Build().Run();
```

Multiple hooks can be registered and they execute in registration order within each phase.

## Complete Example: Validation and Seeding Hook

Here's a full example that validates the app model before startup and seeds a database after resources are created:

```csharp
public class AppModelValidationAndSeedingHook : IDistributedApplicationLifecycleHook
{
    private readonly ILogger<AppModelValidationAndSeedingHook> _logger;

    public AppModelValidationAndSeedingHook(
        ILogger<AppModelValidationAndSeedingHook> logger)
    {
        _logger = logger;
    }

    public Task BeforeStartAsync(
        DistributedApplicationModel appModel,
        CancellationToken cancellationToken = default)
    {
        _logger.LogInformation(
            "App model has {Count} resources",
            appModel.Resources.Count());

        // Ensure no duplicate resource names
        var names = appModel.Resources.Select(r => r.Name).ToList();
        var duplicates = names.GroupBy(n => n)
            .Where(g => g.Count() > 1)
            .Select(g => g.Key);

        foreach (var dup in duplicates)
        {
            throw new InvalidOperationException(
                $"Duplicate resource name: '{dup}'");
        }

        return Task.CompletedTask;
    }

    public Task AfterEndpointsAllocatedAsync(
        DistributedApplicationModel appModel,
        CancellationToken cancellationToken = default)
    {
        foreach (var resource in appModel.Resources)
        {
            var endpoints = resource.Annotations
                .OfType<EndpointAnnotation>()
                .Where(e => e.AllocatedEndpoint is not null);

            foreach (var ep in endpoints)
            {
                _logger.LogInformation(
                    "{Resource}/{Endpoint} → {Url}",
                    resource.Name,
                    ep.Name,
                    ep.AllocatedEndpoint!.UriString);
            }
        }

        return Task.CompletedTask;
    }

    public async Task AfterResourcesCreatedAsync(
        DistributedApplicationModel appModel,
        CancellationToken cancellationToken = default)
    {
        _logger.LogInformation("All resources created. Running post-startup tasks...");

        // Example: seed data or verify connectivity
        await Task.Delay(1000, cancellationToken);

        _logger.LogInformation("Post-startup tasks complete.");
    }
}
```

Register it:

```csharp
builder.Services.AddLifecycleHook<AppModelValidationAndSeedingHook>();
```

## Dependency Injection in Hooks

Lifecycle hooks are resolved from the DI container, so you can inject any registered service:

```csharp
public class MyHook(
    ILogger<MyHook> logger,
    IConfiguration configuration,
    ResourceNotificationService notificationService)
    : IDistributedApplicationLifecycleHook
{
    public Task BeforeStartAsync(
        DistributedApplicationModel appModel,
        CancellationToken cancellationToken = default)
    {
        var env = configuration["ASPNETCORE_ENVIRONMENT"];
        logger.LogInformation("Starting in {Environment} mode", env);
        return Task.CompletedTask;
    }
}
```

## Hooks vs Eventing System

Lifecycle hooks and the eventing system (covered in the next lesson) serve different purposes:

| Aspect | Lifecycle Hooks | Eventing System |
|---|---|---|
| Scope | Global — fires once for the entire app | Per-resource — fires for specific resources |
| Granularity | Three fixed phases | Fine-grained events (e.g., before each resource starts) |
| Use case | App-wide validation, seeding, logging | Per-resource customization, conditional startup |
| Registration | `builder.Services.AddLifecycleHook<T>()` | `builder.Eventing.Subscribe<TEvent>(...)` |

Choose lifecycle hooks when you need to act on the entire application model. Choose events when you need per-resource control.

## Summary

`IDistributedApplicationLifecycleHook` provides three well-defined extension points in the Aspire application startup pipeline: before anything starts, after ports are allocated, and after everything is running. By implementing this interface and registering your hook, you can validate configurations, log the resource graph, seed databases, and perform any other cross-cutting startup logic — all with full access to the app model and the DI container.
