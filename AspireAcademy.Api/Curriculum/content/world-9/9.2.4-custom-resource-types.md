# 9.2.4 — Building Custom Resource Types

## Introduction

Aspire ships with hosting integrations for dozens of technologies — Redis, PostgreSQL, RabbitMQ, and more. But what if you need to integrate a service that Aspire doesn't support yet? You can build your own **custom resource type** from scratch. This lesson walks through every step: defining the resource, creating extension methods, exposing connection strings, adding health checks, and wiring it all together.

## The IResource Interface

Every resource in Aspire implements `IResource`:

```csharp
public interface IResource
{
    string Name { get; }
    ResourceAnnotationCollection Annotations { get; }
}
```

- **`Name`** — the unique identifier for the resource within the app model. This is also used as the connection string key and service discovery name.
- **`Annotations`** — a collection of metadata objects that describe the resource's behavior (endpoints, environment variables, health checks, lifecycle settings, etc.)

In practice, you rarely implement `IResource` directly. Instead, you extend the `Resource` base class or one of its specialized subclasses.

## Step 1: Define the Resource Class

Start by creating a class that inherits from `Resource` (or a more specific base class like `ContainerResource`):

```csharp
public class EmailServiceResource : ContainerResource, IResourceWithConnectionString
{
    internal const string SmtpEndpointName = "smtp";

    public EmailServiceResource(string name)
        : base(name)
    {
    }

    /// <summary>
    /// Gets the SMTP endpoint for the email service.
    /// </summary>
    public EndpointReference SmtpEndpoint =>
        new(this, SmtpEndpointName);

    /// <summary>
    /// Gets the connection string expression for the email service.
    /// </summary>
    public ReferenceExpression ConnectionStringExpression =>
        ReferenceExpression.Create(
            $"smtp://{SmtpEndpoint.Property(EndpointProperty.Host)}:{SmtpEndpoint.Property(EndpointProperty.Port)}");
}
```

### Choosing a Base Class

| Base Class | Use When |
|---|---|
| `Resource` | Your resource is abstract or doesn't map directly to a container or executable |
| `ContainerResource` | Your resource is backed by a Docker container |
| `ExecutableResource` | Your resource is a local executable or process |
| `ProjectResource` | Your resource is a .NET project (rarely extended directly) |

### Implementing IResourceWithConnectionString

If your resource exposes a connection string that other services consume, implement `IResourceWithConnectionString`:

```csharp
public interface IResourceWithConnectionString : IResource
{
    ReferenceExpression ConnectionStringExpression { get; }
}
```

The `ConnectionStringExpression` uses `ReferenceExpression` to define the connection string template with placeholders that are resolved at run-time when ports are allocated.

## Step 2: Create Extension Methods

Extension methods provide the fluent API that AppHost authors use to add and configure your resource:

```csharp
public static class EmailServiceResourceExtensions
{
    /// <summary>
    /// Adds an email service resource backed by a MailHog container.
    /// </summary>
    /// <param name="builder">The distributed application builder.</param>
    /// <param name="name">The name of the resource.</param>
    /// <param name="smtpPort">Optional fixed SMTP port. If not specified, a random port is allocated.</param>
    /// <returns>A reference to the resource builder for chaining.</returns>
    public static IResourceBuilder<EmailServiceResource> AddEmailService(
        this IDistributedApplicationBuilder builder,
        string name,
        int? smtpPort = null)
    {
        var resource = new EmailServiceResource(name);

        return builder
            .AddResource(resource)
            .WithImage("mailhog/mailhog", "latest")
            .WithEndpoint(
                port: smtpPort,
                targetPort: 1025,
                name: EmailServiceResource.SmtpEndpointName,
                scheme: "tcp")
            .WithHttpEndpoint(targetPort: 8025, name: "http");
    }

    /// <summary>
    /// Adds a web UI endpoint for viewing captured emails.
    /// </summary>
    public static IResourceBuilder<EmailServiceResource> WithMailUI(
        this IResourceBuilder<EmailServiceResource> builder)
    {
        // The HTTP endpoint is already added — this could add
        // additional configuration or annotations
        return builder;
    }
}
```

### The AddResource Pattern

Every `Add*` extension method follows a consistent pattern:

1. **Create the resource instance** — `new EmailServiceResource(name)`
2. **Register with the builder** — `builder.AddResource(resource)`
3. **Configure the container image** — `.WithImage("image", "tag")`
4. **Define endpoints** — `.WithEndpoint(...)` for each port
5. **Return `IResourceBuilder<T>`** — for method chaining

## Step 3: Add Custom Properties and Annotations

Annotations are the primary extensibility mechanism. You can define custom annotations and attach them to your resource:

```csharp
/// <summary>
/// Annotation that configures the email service's SMTP relay.
/// </summary>
public class SmtpRelayAnnotation : IResourceAnnotation
{
    public string RelayHost { get; set; } = string.Empty;
    public int RelayPort { get; set; } = 587;
}
```

Add it via an extension method:

```csharp
public static IResourceBuilder<EmailServiceResource> WithSmtpRelay(
    this IResourceBuilder<EmailServiceResource> builder,
    string relayHost,
    int relayPort = 587)
{
    builder.Resource.Annotations.Add(new SmtpRelayAnnotation
    {
        RelayHost = relayHost,
        RelayPort = relayPort
    });

    return builder.WithEnvironment(context =>
    {
        context.EnvironmentVariables["MH_SMTP_RELAY_HOST"] = relayHost;
        context.EnvironmentVariables["MH_SMTP_RELAY_PORT"] = relayPort.ToString();
    });
}
```

## Step 4: Add Health Checks

Health checks let Aspire know when your resource is ready. Register a health check using `WithHealthCheck`:

```csharp
public static IResourceBuilder<EmailServiceResource> AddEmailService(
    this IDistributedApplicationBuilder builder,
    string name,
    int? smtpPort = null)
{
    var resource = new EmailServiceResource(name);

    return builder
        .AddResource(resource)
        .WithImage("mailhog/mailhog", "latest")
        .WithEndpoint(
            port: smtpPort,
            targetPort: 1025,
            name: EmailServiceResource.SmtpEndpointName,
            scheme: "tcp")
        .WithHttpEndpoint(targetPort: 8025, name: "http")
        .WithHealthCheck(
            // Health check name (unique within the resource)
            $"{name}-smtp-health",
            // Register the health check
            (builder, context) =>
            {
                builder.AddTcpHealthCheck(options =>
                {
                    // Check that the SMTP port is accepting connections
                    options.AddHost(
                        context.Resource.SmtpEndpoint
                            .Property(EndpointProperty.Host).ToString(),
                        context.Resource.SmtpEndpoint
                            .Property(EndpointProperty.Port));
                }, name: $"{name}-smtp-health");
            });
}
```

With health checks in place, `WaitFor` will gate dependent resources until the email service's SMTP port is accepting connections.

## Step 5: Implement Lifecycle Behavior

If your resource needs custom startup or shutdown logic, use the eventing system:

```csharp
public static IResourceBuilder<EmailServiceResource> AddEmailService(
    this IDistributedApplicationBuilder builder,
    string name,
    int? smtpPort = null)
{
    var resource = new EmailServiceResource(name);

    var resourceBuilder = builder
        .AddResource(resource)
        .WithImage("mailhog/mailhog", "latest")
        .WithEndpoint(
            port: smtpPort,
            targetPort: 1025,
            name: EmailServiceResource.SmtpEndpointName,
            scheme: "tcp")
        .WithHttpEndpoint(targetPort: 8025, name: "http");

    // Register a lifecycle event for post-start initialization
    builder.Eventing.Subscribe<BeforeResourceStartedEvent>(
        resource,
        async (evt, cancellationToken) =>
        {
            // Custom initialization before the container starts
            Console.WriteLine($"Preparing email service '{name}'...");
        });

    return resourceBuilder;
}
```

## Complete Example: EmailService Resource

Putting it all together, here's the complete custom resource:

### Resource class

```csharp
public class EmailServiceResource : ContainerResource, IResourceWithConnectionString
{
    internal const string SmtpEndpointName = "smtp";
    internal const string HttpEndpointName = "http";

    public EmailServiceResource(string name)
        : base(name)
    {
    }

    public EndpointReference SmtpEndpoint =>
        new(this, SmtpEndpointName);

    public EndpointReference WebUIEndpoint =>
        new(this, HttpEndpointName);

    public ReferenceExpression ConnectionStringExpression =>
        ReferenceExpression.Create(
            $"smtp://{SmtpEndpoint.Property(EndpointProperty.Host)}:{SmtpEndpoint.Property(EndpointProperty.Port)}");
}
```

### Extension methods

```csharp
public static class EmailServiceResourceExtensions
{
    public static IResourceBuilder<EmailServiceResource> AddEmailService(
        this IDistributedApplicationBuilder builder,
        string name,
        int? smtpPort = null)
    {
        var resource = new EmailServiceResource(name);

        return builder
            .AddResource(resource)
            .WithImage("mailhog/mailhog", "latest")
            .WithEndpoint(
                port: smtpPort,
                targetPort: 1025,
                name: EmailServiceResource.SmtpEndpointName,
                scheme: "tcp")
            .WithHttpEndpoint(
                targetPort: 8025,
                name: EmailServiceResource.HttpEndpointName);
    }
}
```

### Usage in AppHost

```csharp
var builder = DistributedApplication.CreateBuilder(args);

var email = builder.AddEmailService("email");

var api = builder.AddProject<Projects.Api>("api")
    .WithReference(email)       // Injects ConnectionStrings__email
    .WaitFor(email);            // Waits for SMTP port to be ready

builder.Build().Run();
```

### Consuming in a service project

```csharp
// In the API project's Program.cs
var smtpConnectionString = builder.Configuration
    .GetConnectionString("email");
// smtpConnectionString = "smtp://localhost:52341" (example)
```

## Annotation Reference

Common annotations you might add to custom resources:

| Annotation | Purpose |
|---|---|
| `EndpointAnnotation` | Defines a network endpoint (added via `WithEndpoint`) |
| `EnvironmentCallbackAnnotation` | Injects environment variables dynamically |
| `HealthCheckAnnotation` | Associates a health check with the resource |
| `ContainerImageAnnotation` | Specifies the Docker image (added via `WithImage`) |
| `ContainerMountAnnotation` | Defines volume mounts (added via `WithBindMount`/`WithVolume`) |
| `CommandLineArgsCallbackAnnotation` | Adds command-line arguments to the resource |

## Testing Your Custom Resource

Test that your resource integrates correctly:

```csharp
[Fact]
public async Task EmailServiceResource_SetsCorrectImage()
{
    var builder = DistributedApplication.CreateBuilder();

    var email = builder.AddEmailService("test-email");

    var resource = email.Resource;

    var imageAnnotation = resource.Annotations
        .OfType<ContainerImageAnnotation>()
        .Single();

    Assert.Equal("mailhog/mailhog", imageAnnotation.Image);
    Assert.Equal("latest", imageAnnotation.Tag);
}
```

## Summary

Building a custom resource type in Aspire follows a structured pattern: define a resource class (extending `ContainerResource` or `Resource`), implement `IResourceWithConnectionString` if it exposes a connection string, create fluent extension methods for the builder API, add health checks for readiness gating, and wire up lifecycle events for custom behavior. This extensibility model ensures that any service — whether it's a container, executable, or cloud resource — can be a first-class citizen in the Aspire app model.
