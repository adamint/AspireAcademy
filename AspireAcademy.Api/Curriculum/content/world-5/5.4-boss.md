# Boss Challenge: Observable Order Service

## The Mission

Instrument an order processing service with full observability—custom distributed tracing, business metrics, and a health check for an external payment dependency. Your service must be fully observable through the Aspire dashboard.

## Requirements

### Custom Tracing

1. Create an `ActivitySource` named `"OrderService"`.
2. Every `POST /orders` request must start an activity named `"ProcessOrder"`.
3. Set tags on the activity: `order.id`, `order.total`, and `order.region`.
4. Add an `ActivityEvent` named `"PaymentProcessed"` after the payment step.

### Custom Metrics

1. Create a `Meter` named `"OrderService"`.
2. Create a `Counter<long>` named `"orders.placed"` — increment on each successful order.
3. Create a `Histogram<double>` named `"orders.duration_ms"` — record the processing duration in milliseconds.
4. Include a `region` tag on both instruments.

### Health Check

1. Implement a custom `IHealthCheck` named `"payment-provider"`.
2. The check should verify the payment provider is reachable (simulate with an HTTP call or simple logic).
3. Return `Healthy`, `Degraded`, or `Unhealthy` based on the result.

### OpenTelemetry Registration

Register your custom `ActivitySource` and `Meter` with the OpenTelemetry pipeline:

```csharp
builder.Services.AddOpenTelemetry()
    .WithTracing(t => t.AddSource("OrderService"))
    .WithMetrics(m => m.AddMeter("OrderService"));
```

## Success Criteria

- The `"OrderService"` ActivitySource appears in the dashboard traces view.
- Each `POST /orders` creates a span with `order.id` and `order.total` tags.
- The `orders.placed` counter increments with each request.
- The `orders.duration_ms` histogram records timing data.
- The `/health` endpoint returns `Healthy` and the dashboard shows green status.
