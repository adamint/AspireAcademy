# Distributed Tracing

Distributed tracing tracks requests as they flow across service boundaries. In .NET, the `System.Diagnostics.Activity` class is the span primitive—each Activity represents a unit of work within a trace.

## Creating an ActivitySource

Define one `ActivitySource` per logical component. This is typically a static field shared across the service.

```csharp
public static class Telemetry
{
    public static readonly ActivitySource Source = new("OrderService");
}
```

## Starting Spans

Wrap operations in activities to create spans in the trace.

```csharp
public async Task<Order> ProcessOrderAsync(OrderRequest request)
{
    using var activity = Telemetry.Source.StartActivity("ProcessOrder");

    activity?.SetTag("order.id", request.Id);
    activity?.SetTag("order.total", request.Total);

    var result = await ValidateOrder(request);
    activity?.AddEvent(new ActivityEvent("OrderValidated"));

    await ChargePayment(request);
    activity?.AddEvent(new ActivityEvent("PaymentProcessed"));

    return result;
}
```

## Context Propagation

W3C TraceContext headers (`traceparent`, `tracestate`) propagate automatically through `HttpClient`. When Service A calls Service B, the trace ID and parent span ID are forwarded so the dashboard can reconstruct the full distributed trace.

```
Service A (span 1) → HttpClient → Service B (span 2, parent=span 1)
```

No manual header management is needed—the OpenTelemetry SDK instruments `HttpClient` automatically.

## Registering with OpenTelemetry

Your custom `ActivitySource` must be registered so the OTLP exporter knows to collect its spans.

```csharp
builder.Services.AddOpenTelemetry()
    .WithTracing(tracing =>
    {
        tracing.AddSource("OrderService");
    });
```

Without this registration, activities are still created but never exported to the dashboard. The `AddServiceDefaults()` method registers common sources like `System.Net.Http` and ASP.NET Core—your custom sources are additive.
