# OTLP Telemetry Flow

OpenTelemetry Protocol (OTLP) is the standard wire format that carries all observability data—traces, metrics, and logs—from your application to the Aspire dashboard. Understanding this data flow is key to effective debugging and monitoring.

## The Three Pillars

Every Aspire application emits three types of telemetry:

- **Traces**: Distributed request flows across services, showing timing and causality.
- **Metrics**: Numerical measurements like request counts, durations, and queue depths.
- **Logs**: Structured log entries with severity, message templates, and properties.

## How Data Flows

When you call `AddServiceDefaults()`, the OpenTelemetry SDK is configured with an OTLP exporter that sends all three pillars over gRPC.

```
App → OTLP gRPC Exporter → Dashboard OTLP Endpoint → In-Memory Buffer → Blazor UI
```

The destination is set via the `OTEL_EXPORTER_OTLP_ENDPOINT` environment variable, which the AppHost automatically injects into every project resource. You never need to configure this manually during local development.

## Dashboard Endpoints

The dashboard exposes two distinct gRPC endpoints:

1. **OTLP Collector** — Receives traces, metrics, and logs from applications. This is where the OTLP exporter sends data. The services are `OtlpGrpcTraceService`, `OtlpGrpcLogsService`, and `OtlpGrpcMetricsService`.

2. **Resource Service** — Receives resource lifecycle information (start, stop, health status) from the AppHost. DCP (the Developer Control Plane) reports container and project state to the AppHost, which forwards it to the dashboard.

## Storage Model

The dashboard stores telemetry in **circular in-memory buffers** via `TelemetryRepository`. This means data is fast to query but not persisted across restarts. Each buffer has a fixed capacity—once full, the oldest entries are evicted. This design keeps the dashboard lightweight and responsive without requiring external storage.
