# Settings Classes and Configuration Hierarchy

How settings flow from connection strings, configuration providers, and inline delegates.

Every client integration has a **settings class** (e.g., `NpgsqlSettings`, `StackExchangeRedisSettings`) that controls connection behavior, health checks, and telemetry. Settings are resolved through three layers, applied in order.

## Layer 1: Connection Strings

The connection name passed to `AddXxxClient` resolves from the `ConnectionStrings` section:

```json
{
  "ConnectionStrings": {
    "cache": "localhost:6379",
    "postgresdb": "Host=localhost;Database=mydb"
  }
}
```

These are typically injected automatically by the AppHost via `WithReference`.

## Layer 2: Configuration Providers

Settings classes bind from structured configuration under the `Aspire:{Technology}:{Component}` key:

```json
{
  "Aspire": {
    "StackExchange": {
      "Redis": {
        "DisableHealthChecks": false,
        "DisableTracing": false
      }
    },
    "Npgsql": {
      "DisableHealthChecks": true,
      "CommandTimeout": 60
    }
  }
}
```

This works with any configuration provider — `appsettings.json`, environment variables, Azure App Configuration, or user secrets.

## Layer 3: Inline Delegates

The `configureSettings` delegate runs last and overrides everything:

```csharp
builder.AddNpgsqlDataSource("postgresdb", settings =>
{
    settings.DisableHealthChecks = true;
    settings.DisableTracing = false;
});
```

## Resolution Order

Settings resolve bottom-up: connection string → configuration binding → inline delegate. Each layer can override the previous one. This means an inline delegate always wins:

```csharp
// Even if appsettings.json sets DisableHealthChecks = false,
// this delegate overrides it to true
builder.AddRedisClient("cache", settings =>
    settings.DisableHealthChecks = true);
```

Common settings shared across all integrations include `DisableHealthChecks`, `DisableTracing`, and `DisableMetrics`, giving you fine-grained control over observability per client.
