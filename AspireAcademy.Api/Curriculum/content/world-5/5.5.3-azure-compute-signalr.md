# Azure Container Apps, App Service, and SignalR

You need to decide where your services actually run in Azure, and whether they should scale to zero when idle. You also have a live auction feature that needs sub-second updates pushed to thousands of connected browsers. These decisions — compute platform and real-time infrastructure — are tightly coupled to your cost model and architecture.

---

## Azure Container Apps

```csharp
builder.AddProject<Projects.Api>("api")
    .PublishAsAzureContainerApp((module, app) =>
    {
        app.Template.Scale.MinReplicas = 1;
        app.Template.Scale.MaxReplicas = 10;
    });
```

When publishing to Azure, Container Apps is the default compute target. You can customize scaling rules, ingress settings, and environment configuration.

> 💡 **Key insight:** Azure Container Apps can scale to zero replicas, meaning you pay nothing when there's no traffic. But scale-to-zero introduces cold start latency (~1-3 seconds for the first request). Set `MinReplicas = 1` for user-facing APIs where latency matters, and `MinReplicas = 0` for background workers and internal services where a brief cold start is acceptable. The `PublishAsAzureContainerApp` callback is where you make this architectural decision.

## Azure App Service

```csharp
builder.AddProject<Projects.Api>("api")
    .PublishAsAzureAppServiceWebApp();
```

For apps that benefit from App Service features — deployment slots, custom domains, managed certificates.

## Azure SignalR Service

```csharp
var signalr = builder.AddAzureSignalR("signalr");

builder.AddProject<Projects.Api>("api")
    .WithReference(signalr);
```

Azure SignalR Service offloads real-time WebSocket connections from your app. Use it for chat, live dashboards, and notifications.

In development, your app uses local SignalR. In production, Azure manages the connections.

## Azure Web PubSub

```csharp
var pubsub = builder.AddAzureWebPubSub("pubsub");
```

For pure WebSocket workloads without the SignalR protocol.
