# Console Logs & Resource Graph

The dashboard includes two additional views that complement the structured telemetry pages: Console Logs for raw output inspection and the Resource Graph for visualizing application topology.

## Console Logs

The Console Logs page shows raw `stdout` and `stderr` output from each resource—exactly what you would see if running the process in a terminal window.

### When to Use Console Logs

- **Startup diagnostics** — See framework initialization messages, configuration loading, and binding confirmations before structured logging is active.
- **Crash output** — Unhandled exceptions and stack traces often appear in stderr before the process exits.
- **Container output** — Infrastructure containers (Redis, PostgreSQL, RabbitMQ) emit their native log formats to stdout.

### Features

- **Resource selector** — Switch between resources to view their individual output streams.
- **ANSI color rendering** — Console logs preserve ANSI escape codes, so colored output from frameworks and tools renders correctly in the browser.
- **Auto-scroll** — The view auto-scrolls to show the latest output, with the option to pause and inspect earlier messages.

Unlike structured logs, console logs are plain text without searchable properties or trace correlation. They are a low-level debugging tool for situations where structured telemetry is unavailable.

## Resource Graph

The Resource Graph provides a directed graph visualization of your application's topology.

- **Nodes** — Each resource (project, container, executable) appears as a node with its name and type.
- **Edges** — Directed arrows represent `WithReference()` relationships from the AppHost. An edge from `apiservice` to `cache` means the API service has a reference to the cache resource.

The graph makes dependency chains immediately visible. You can see at a glance which services depend on which databases, caches, and message brokers—and spot circular or unexpected dependencies in complex application models.
