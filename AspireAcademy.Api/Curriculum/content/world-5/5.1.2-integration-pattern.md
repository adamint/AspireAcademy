# The Hosting Integration Pattern

Every hosting integration follows the same pattern, regardless of what infrastructure it wraps. Understanding this pattern is the key to using — and building — integrations.

---

## The Pattern

1. **NuGet package** — `Aspire.Hosting.{Service}` (e.g., `Aspire.Hosting.Redis`)
2. **Extension method** — `Add{Service}()` on `IDistributedApplicationBuilder`
3. **Resource class** — inherits from `ContainerResource` or `Resource`
4. **Health check** — registered via `HealthCheckAnnotation`
5. **Connection string** — implements `IResourceWithConnectionString`
6. **Default configuration** — image, tag, ports, environment variables

## Every Integration Follows This

| Integration | Image | Port | Health Check | Connection String |
|---|---|---|---|---|
| Redis | `redis:7.4` | 6379 | PING | `host:port` |
| PostgreSQL | `postgres:17` | 5432 | SQL connection | Full Npgsql format |
| RabbitMQ | `rabbitmq:4-management` | 5672 | Management API | AMQP URL |
| MongoDB | `mongo:8` | 27017 | Admin command | MongoDB URL |
| SQL Server | `mcr.microsoft.com/mssql/server` | 1433 | SQL connection | SqlClient format |

## The Power of Consistency

Because every integration follows the same pattern:
- `WithReference()` works with every integration
- `WaitFor()` uses each integration's health check
- The Dashboard displays every integration consistently
- The publishing pipeline generates artifacts for every integration

> 🎲 **Fun Fact:** The consistency of the hosting integration pattern means you can predict the API for any new integration before you even read the docs. If Aspire adds a Meilisearch integration tomorrow, you already know it'll be `builder.AddMeilisearch("search")` with `.WithReference()` and `.WaitFor()` support.
