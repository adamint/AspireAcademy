# The Add/AddKeyed Pattern

How client integrations register clients in DI — Add for singleton, AddKeyed for named instances.

Every Aspire client integration follows a consistent two-method pattern for registering services in the dependency injection container: `AddXxxClient` and `AddKeyedXxxClient`.

## The Add Pattern

The standard `Add` method registers a single client for a given connection name. It reads the connection string from `ConnectionStrings` configuration and registers the client as a singleton in DI:

```csharp
// Program.cs of your service project
builder.AddRedisClient("cache");
builder.AddNpgsqlDataSource("postgresdb");
```

The connection name (e.g., `"cache"`) must match what the AppHost provides via `WithReference`. Behind the scenes, the method registers health checks, configures tracing, and wires up metrics automatically.

## The AddKeyed Pattern

When your service connects to **multiple instances** of the same technology, use `AddKeyed` to register named clients:

```csharp
builder.AddKeyedRedisClient("sessions");
builder.AddKeyedRedisClient("output-cache");
```

Retrieve keyed services using the `[FromKeyedServices]` attribute:

```csharp
public class CartService(
    [FromKeyedServices("sessions")] IConnectionMultiplexer sessions,
    [FromKeyedServices("output-cache")] IConnectionMultiplexer outputCache)
{
    // Each parameter resolves to a different Redis instance
}
```

## Method Signature Convention

All client integration methods follow the same signature pattern:

```csharp
public static void AddXxxClient(
    this IHostApplicationBuilder builder,
    string connectionName,
    Action<XxxSettings>? configureSettings = null);
```

The optional `configureSettings` delegate lets you customize behavior inline:

```csharp
builder.AddRedisClient("cache", settings =>
    settings.DisableHealthChecks = true);
```

This consistency across all integrations — databases, caches, message brokers — means that once you learn the pattern, every new integration feels familiar.
