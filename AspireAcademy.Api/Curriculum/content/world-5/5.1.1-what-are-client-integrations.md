# What Are Client Integrations?

Client integrations are NuGet packages consumed by **service projects** — not the AppHost. While hosting integrations define and orchestrate resources in the AppHost, client integrations live in the projects that actually *connect* to those resources at runtime.

## What They Do

When you install a client integration package, it provides an `Add{Component}` extension method on `IHostApplicationBuilder` that:

1. **Registers a configured SDK client** into the dependency injection container
2. **Wires up health checks** so the Aspire dashboard can monitor connectivity
3. **Enables OpenTelemetry tracing and metrics** for automatic observability
4. **Applies resilience patterns** like retries and timeouts where appropriate

## The Connection Name Contract

The connection name you pass to `Add{Component}("name")` must match the resource name established by `WithReference()` in the AppHost. This is how Aspire connects your service project to the right resource:

```csharp
// In AppHost
var db = builder.AddPostgres("pg").AddDatabase("catalogdb");
builder.AddProject<Projects.CatalogApi>().WithReference(db);

// In CatalogApi's Program.cs
builder.AddNpgsqlDataSource("catalogdb"); // name must match!
```

## Hosting vs. Client Integrations

| Aspect | Hosting Integration | Client Integration |
|---|---|---|
| **Installed in** | AppHost project | Service/worker project |
| **Purpose** | Define and orchestrate resources | Connect to resources |
| **Package prefix** | `Aspire.Hosting.*` | `Aspire.*` (no Hosting) |

## Common Examples

- **`Aspire.Npgsql`** registers `NpgsqlDataSource` for PostgreSQL access
- **`Aspire.StackExchange.Redis`** registers `IConnectionMultiplexer` for Redis
- **`Aspire.RabbitMQ.Client`** registers `IConnection` for RabbitMQ messaging

Every client integration follows the same pattern: one method call, fully configured client, automatic observability. This consistency makes adopting new technologies straightforward — learn the pattern once, apply it everywhere.
