# The Azure.Provisioning Pattern

When you use `Aspire.Hosting.Azure.*` packages, Aspire does not just run containers locally — it generates Azure Bicep templates for production deployment.

---

## How It Works

```csharp
var storage = builder.AddAzureStorage("storage")
    .AddBlobs("blobs")
    .AddQueues("queues");
```

**In run mode**: Aspire starts the Azurite storage emulator container locally.

**In publish mode**: Aspire generates `main.bicep` that provisions:
- An Azure Storage account
- Blob container configuration
- Queue configuration
- Role assignments for managed identity

## The AzureBicepResource

Azure hosting integrations create `AzureBicepResource` objects that wrap Azure.Provisioning infrastructure:

```csharp
var cosmos = builder.AddAzureCosmosDB("cosmos");
// Creates AzureBicepResource with:
// - Azure.Provisioning.CosmosDB.CosmosDBAccount
// - Default configuration (serverless, consistent prefix)
// - Outputs: connectionString
```

## Customizing Azure Resources

```csharp
builder.AddAzureStorage("storage")
    .ConfigureInfrastructure(infra =>
    {
        var account = infra.GetResources().OfType<StorageAccount>().Single();
        account.Sku = new StorageSku { Name = StorageSkuName.StandardGrs };
        account.Kind = StorageKind.StorageV2;
    });
```

`ConfigureInfrastructure` gives you access to the Azure.Provisioning objects before Bicep is generated.

## Local Emulators

Most Azure integrations support local emulators:

```csharp
builder.AddAzureCosmosDB("cosmos").RunAsEmulator();
builder.AddAzureServiceBus("bus").RunAsEmulator();
builder.AddAzureStorage("storage").RunAsEmulator();
```

Develop locally, deploy to Azure. Same code, different infrastructure.

> 🐛 **Common Gotcha:** The emulators don't support all features of the real Azure services. For example, the Azurite storage emulator doesn't support Azure AD authentication — it only uses the well-known development storage account key. Always test your authentication flow against real Azure services before deploying.
