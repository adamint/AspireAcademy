# Anatomy of AddRedis()

To understand how hosting integrations work, let us trace through what `AddRedis("cache")` actually does. This is where Aspire's magic stops being magic and starts being engineering.

---

## The Call

```csharp
var cache = builder.AddRedis("cache");
```

One line. But behind it, a *lot* happens. Let's trace every step.

## Step 1: Create the Resource

`AddRedis` creates a `RedisResource` (which inherits from `ContainerResource`):

```csharp
var resource = new RedisResource(name); // name = "cache"
```

## Step 2: Configure the Container

The extension method configures the container image and ports:

- **Image**: `docker.io/library/redis`
- **Tag**: `7.4` (default, can be overridden)
- **Port**: `6379` (the standard Redis port)

## Step 3: Add Health Check

A health check annotation is added so that `WaitFor` can verify Redis is ready:

```csharp
resource.Annotations.Add(new HealthCheckAnnotation("redis_ping"));
```

The health check sends a `PING` command and expects `PONG` back.

```deepdive
{"title": "Why health checks matter more than you think", "content": "Without the health check, here's what happens: DCP starts the Redis container. Docker reports the container is 'running' within milliseconds. DCP marks the resource as healthy. Your API starts and tries to connect to Redis... but Redis is still loading its data file from disk (especially with persistent volumes). Connection refused.\n\nThe PING health check prevents this. DCP creates the container, then polls it with PING commands every 500ms. Only when Redis responds with PONG does DCP mark it as Running. Any resource using WaitFor(cache) waits for this healthy signal before starting.\n\nThis is why Aspire's startup ordering is more reliable than Docker Compose's depends_on — it checks actual service readiness, not just container existence."}
```

## Step 4: Register Connection String

Redis implements `IResourceWithConnectionString`. When another resource calls `.WithReference(cache)`, the connection string is resolved as:

```
{cache.bindings.tcp.host}:{cache.bindings.tcp.port}
```

At runtime, this becomes something like `localhost:63211` (DCP allocates a random host port).

## Step 5: Return the Builder

The method returns `IResourceBuilder<RedisResource>`, enabling fluent chaining:

```csharp
builder.AddRedis("cache")
    .WithLifetime(ContainerLifetime.Persistent)
    .WithDataVolume("redis-data");
```

## The Pattern

Every hosting integration follows this pattern:

1. **Create** a resource class (usually inheriting from `ContainerResource`)
2. **Configure** image, ports, and environment variables
3. **Add health checks** via annotations
4. **Implement** `IResourceWithConnectionString` for connection string injection
5. **Return** an `IResourceBuilder<T>` for fluent configuration
