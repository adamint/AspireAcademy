# DashboardClient and Resource Monitoring

How DashboardClient connects to DCP and streams resource state updates to the UI.

The Dashboard needs real-time updates about every resource — when a container starts, when a project crashes, when endpoints become available. This is handled by a gRPC streaming connection between the AppHost and the Dashboard.

## The Resource Update Flow

When the AppHost starts, it launches the Dashboard and establishes a gRPC connection. The `DashboardService` in the AppHost implements a streaming RPC that pushes resource snapshots whenever state changes:

```
AppHost (DashboardService)
    │
    ├── Resource created → snapshot sent
    ├── State changed   → snapshot sent
    ├── Endpoint ready  → snapshot sent
    └── Resource stopped → snapshot sent
          │
          ▼
    Dashboard (DashboardClient)
          │
          ▼
    Blazor UI updates in real-time
```

## Resource Snapshots

Each snapshot contains the complete state of a resource:

- **ResourceType**: `Project`, `Container`, or `Executable`
- **State**: `Starting`, `Running`, `Finished`, `FailedToStart`
- **Properties**: Name, creation time, process ID
- **Endpoints**: Allocated URLs and ports
- **Environment**: All configured environment variables
- **Health status**: Aggregated health check results

## Real-Time UI Updates

The Dashboard uses Blazor Server's `SignalR` connection to push UI updates instantly. When the `DashboardClient` receives a new resource snapshot:

1. The resource state store is updated
2. Blazor components are notified via state change events
3. The UI re-renders the affected resource row

This means you see state changes within milliseconds — a container starting, a health check failing, or a new endpoint becoming available.

## Console Logs

Beyond resource state, the Dashboard also streams **console logs** (stdout/stderr) from each resource. These are captured by the AppHost's process management layer and forwarded to the Dashboard, where they appear in the Console Logs tab with real-time tailing.

## Structured Logs vs Console Logs

The Dashboard shows two types of logs:

- **Console Logs**: Raw stdout/stderr from the process — useful for startup diagnostics
- **Structured Logs**: OpenTelemetry log records received via OTLP — searchable, filterable, with trace correlation
