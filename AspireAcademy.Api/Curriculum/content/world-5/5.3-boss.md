# Boss Challenge: Cached API with Message Bus

## The Mission

You are building a product inventory API that combines Redis distributed caching with RabbitMQ event publishing. Your API must serve product data efficiently using cache-first reads and broadcast inventory changes as events for downstream consumers.

## Requirements

### GET `/products/{id}` — Cache-First Read

1. Check Redis distributed cache for key `product:{id}`.
2. On **cache hit**, return the cached JSON directly—no database call.
3. On **cache miss**, fetch from the data source, store the result in cache with a 5-minute expiration, and return it.

### POST `/products/{id}/restock` — Restock with Event

1. Accept a JSON body with a `quantity` field.
2. Update the product's stock (invalidate or update cache).
3. Publish an event to the RabbitMQ exchange `inventory` with routing key `inventory.restocked`.
4. The message body should include the product ID, new quantity, and a timestamp.

## Integration Setup

Register two client integrations in `Program.cs`:

```csharp
builder.AddRedisDistributedCache("cache");
builder.AddRabbitMQClient("messaging");
```

These correspond to resources defined in the AppHost:

```csharp
var cache = builder.AddRedis("cache");
var messaging = builder.AddRabbitMQ("messaging");

builder.AddProject<Projects.InventoryApi>()
       .WithReference(cache)
       .WithReference(messaging);
```

## Success Criteria

- `IDistributedCache` and `IConnection` resolve from DI without errors.
- Cache miss correctly falls through to the data source and populates the cache.
- Cache hit returns data without touching the data source.
- Restock endpoint publishes a properly serialized message to the `inventory.restocked` routing key.

## Tips

- Use `JsonSerializer` to serialize/deserialize cache entries.
- Create a channel from `IConnection` for each publish operation.
- Set `DeliveryMode = 2` for persistent messages.
