# Boss Challenge: Multi-Database Service

## The Mission

You've mastered individual database integrations. Now it's time to combine them. In this challenge, you'll build a service that connects to **both PostgreSQL and MongoDB** simultaneously — a common real-world pattern where relational and document databases serve different purposes.

## Scenario

Your catalog service stores structured product data in PostgreSQL using EF Core. A separate analytics system needs denormalized product snapshots in MongoDB for fast reads. Your job: build a `/sync` endpoint that reads products from PostgreSQL and writes them to a MongoDB collection.

## Requirements

1. **Register EF Core PostgreSQL** using `AddNpgsqlDbContext<CatalogContext>("catalogdb")`
2. **Register MongoDB** using `AddMongoDBClient("mongodb")`
3. **Create a `/sync` endpoint** that reads all products from PostgreSQL and inserts them into MongoDB's `"products"` collection
4. **Health checks must be active** for both integrations (default behavior — don't disable them)

## What You'll Demonstrate

- Registering multiple client integrations in a single service
- Using both relational and document database clients together
- Understanding that each integration independently contributes health checks and tracing
- Navigating MongoDB's client → database → collection pattern alongside EF Core's DbContext pattern

## Success Criteria

- `CatalogContext` resolves from DI without errors
- `IMongoClient` resolves from DI without errors
- The `/sync` endpoint returns `200 OK`
- Both PostgreSQL and MongoDB health checks are registered and passing

Good luck — this is your gateway to building polyglot-persistence services with Aspire!
