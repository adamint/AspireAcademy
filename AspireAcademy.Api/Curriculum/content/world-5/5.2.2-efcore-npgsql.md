# Entity Framework Core with PostgreSQL

The `Aspire.Npgsql.EntityFrameworkCore.PostgreSQL` package bridges EF Core and Aspire, registering your `DbContext` with connection pooling, automatic retries, and full observability.

## Registration

```csharp
builder.AddNpgsqlDbContext<CatalogContext>("catalogdb");
```

This registers `CatalogContext` using `DbContextPool` for efficient context reuse. The connection name `"catalogdb"` maps to your AppHost resource name.

## What You Get Automatically

Beyond basic DI registration, this integration configures:

- **Connection pooling** via `AddDbContextPool` (not `AddDbContext`)
- **Retry strategy** for transient failures using Npgsql's built-in execution strategy
- **Health check** of type `DbContextHealthCheck<TContext>` — ensures migrations are applied and the database is reachable
- **OpenTelemetry tracing and metrics** for all EF Core operations

## Settings and Configuration

The settings class is `NpgsqlEntityFrameworkCorePostgreSQLSettings`, configured under `Aspire:Npgsql:EntityFrameworkCore:PostgreSQL`:

```json
{
  "Aspire": {
    "Npgsql": {
      "EntityFrameworkCore": {
        "PostgreSQL": {
          "DisableRetry": false,
          "CommandTimeout": 30,
          "DisableHealthChecks": false
        }
      }
    }
  }
}
```

```csharp
builder.AddNpgsqlDbContext<CatalogContext>("catalogdb", settings =>
{
    settings.DisableRetry = true;  // Disable for local development
    settings.CommandTimeout = 60;
});
```

## When to Use This vs. Raw Npgsql

| Choose | When |
|---|---|
| `Aspire.Npgsql` | You want raw SQL, micro-ORM, or Dapper |
| `Aspire.Npgsql.EntityFrameworkCore.PostgreSQL` | You use EF Core for data access |

Both packages can coexist in the same project. The EF Core integration builds on top of the Npgsql provider, so you benefit from the same connection pooling and observability — with the added convenience of EF Core's change tracking, LINQ queries, and migrations.
