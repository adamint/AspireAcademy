# Dashboard Architecture Overview

How the Dashboard is built with Blazor, receives telemetry via gRPC, and stores data in TelemetryRepository.

The Aspire Dashboard is a **Blazor Server** application that provides real-time monitoring of your distributed application. It displays resources, logs, traces, and metrics — all collected during local development.

## Core Components

The Dashboard consists of several key layers:

1. **Blazor UI**: Interactive server-side Blazor pages for resources, logs, traces, and metrics
2. **gRPC Resource Service**: Receives resource state updates from the AppHost via `DashboardService`
3. **OTLP Ingestion**: Accepts traces, logs, and metrics from services via OpenTelemetry Protocol
4. **In-Memory Storage**: `TelemetryRepository` stores telemetry data for the session

## How Data Flows

```
Services ──(OTLP)──► Dashboard OTLP Endpoints
                         │
                         ▼
                   TelemetryRepository
                         │
                         ▼
                   Blazor UI Pages

AppHost ──(gRPC)──► DashboardService
                         │
                         ▼
                   Resource State Store
```

## DashboardOptions

The Dashboard is configured via `DashboardOptions`:

- **Frontend endpoint**: The browser URL (default `http://localhost:15888`)
- **OTLP gRPC endpoint**: Where services send telemetry (default port `18889`)
- **OTLP HTTP endpoint**: Alternative HTTP-based OTLP ingestion (default port `18890`)
- **Resource service endpoint**: gRPC endpoint for AppHost resource updates

## Resource State

The AppHost streams resource state changes to the Dashboard via gRPC. Each resource has:

- **Name and type** (project, container, executable)
- **State** (Starting, Running, Finished, FailedToStart)
- **Endpoints** (URLs where the resource is accessible)
- **Environment variables** (visible in resource details)

## Telemetry Storage

`TelemetryRepository` stores telemetry in circular buffers with configurable limits. This means data is kept in memory only — when the Dashboard restarts, telemetry history is lost. For production monitoring, export telemetry to a persistent backend like Application Insights or Prometheus.
