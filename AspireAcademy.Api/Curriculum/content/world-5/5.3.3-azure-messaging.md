# Azure Service Bus and Event Hubs

When you need managed cloud messaging instead of self-hosted brokers.

---

## Azure Service Bus

```csharp
var serviceBus = builder.AddAzureServiceBus("servicebus");
```

In run mode, this provisions a Service Bus namespace using the Azure provisioning system. In publish mode, it generates Bicep to create the namespace.

Use Service Bus for:
- Enterprise messaging with transactions
- Dead-letter queues and sessions
- Integration with Azure Functions triggers

## Azure Event Hubs

```csharp
var eventHubs = builder.AddAzureEventHubs("eventhubs")
    .AddHub("telemetry");
```

Event Hubs is Azure's Kafka-compatible event streaming service. Use for:
- High-throughput event ingestion
- Real-time analytics pipelines
- IoT telemetry

## Local Development

Both services support emulators for local development:

```csharp
var serviceBus = builder.AddAzureServiceBus("servicebus")
    .RunAsEmulator();  // uses the Azure Service Bus emulator container
```

This lets you develop against Azure messaging APIs without an Azure subscription. In publish mode, the real Azure service is provisioned.

> 🐛 **Common Gotcha:** The Azure Service Bus emulator has some limitations compared to the real service — features like message sessions, scheduled delivery beyond 24 hours, and geo-disaster recovery are not fully emulated. Always test critical messaging flows against the real Azure service before deploying to production.

## When to Use Cloud vs Self-Hosted

| Concern | Self-Hosted (RabbitMQ/Kafka) | Cloud (Service Bus/Event Hubs) |
|---|---|---|
| Cost | Server costs | Pay-per-message |
| Operations | You manage it | Azure manages it |
| Dev experience | Identical local/prod | Emulator for local |
| Features | Standard broker features | Azure-specific (Functions triggers, managed identity) |
