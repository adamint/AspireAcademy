# Azure Storage, Key Vault, and Functions

Your app needs to store user-uploaded files, process background jobs from a queue, and keep secrets out of environment variables. These are the bread-and-butter Azure services you'll reach for in almost every production app — and Aspire integrates all three with local emulators so you can develop without an Azure subscription.

---

## Azure Storage

```csharp
var storage = builder.AddAzureStorage("storage")
    .RunAsEmulator();  // Azurite locally

var blobs = storage.AddBlobs("blobs");
var queues = storage.AddQueues("queues");
var tables = storage.AddTables("tables");

builder.AddProject<Projects.Api>("api")
    .WithReference(blobs)
    .WithReference(queues);
```

Each sub-resource (blobs, queues, tables) gets its own connection string.

> 💡 **Key insight:** Azure Storage's blob, queue, and table sub-resources share a single storage account but get separate connection strings via `WithReference()`. This matters for managed identity: in publish mode, Aspire generates role assignments for each sub-resource independently. A service that only references `blobs` gets `Storage Blob Data Contributor`, not full storage account access — principle of least privilege, enforced by the integration pattern.

## Azure Key Vault

```csharp
var vault = builder.AddAzureKeyVault("secrets");

builder.AddProject<Projects.Api>("api")
    .WithReference(vault);
```

In run mode, secrets are read from local configuration. In publish mode, Key Vault is provisioned and the app uses managed identity to access secrets.

## Azure Functions

```csharp
var functions = builder.AddAzureFunctionsProject<Projects.OrderProcessor>("processor")
    .WithReference(queues)
    .WithReference(cosmos);
```

Aspire can orchestrate Azure Functions projects alongside other services. Functions receive the same connection strings and service discovery as any other resource.

var acr = builder.AddAzureContainerRegistry("registry");
