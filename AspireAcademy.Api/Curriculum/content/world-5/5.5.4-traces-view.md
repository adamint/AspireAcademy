# Traces View

The Traces view displays distributed traces as interactive waterfall diagrams. Each trace represents a complete request flow across one or more services, broken down into individual spans showing timing, hierarchy, and status.

## Trace List

The top-level list shows recent traces with summary information:

- **Trace ID** — The unique W3C trace identifier.
- **Resources** — Icons for each service that participated in the trace.
- **Duration** — Total end-to-end time for the entire trace.
- **Span count** — How many spans make up the trace.
- **Error indicator** — A red marker if any span in the trace has an error status.

Filter traces by resource to focus on a specific service, or filter by minimum duration to find slow requests.

## Waterfall Diagram

Click a trace to open the waterfall view. Each row represents a span:

- **Service name** — Which resource owns the span, color-coded for visual grouping.
- **Operation name** — The span's operation (e.g., `GET /api/products`, `ProcessOrder`, `SELECT products`).
- **Duration bar** — A horizontal bar showing relative timing. The bar's position and width indicate when the span started and how long it took.
- **Nesting** — Child spans are indented under their parent, showing causality across service boundaries.

The waterfall makes it immediately clear where time is spent. A long database span nested under an API call reveals the bottleneck visually.

## Span Details

Click any span to view its details:

- **Tags** — Key-value attributes like `http.method`, `db.statement`, or custom tags you set via `SetTag()`.
- **Events** — Timestamped annotations added via `AddEvent()`.
- **Status** — Ok or Error, with an error message if applicable.

Error spans appear in red throughout the waterfall, drawing attention to failures across the entire distributed call chain.
