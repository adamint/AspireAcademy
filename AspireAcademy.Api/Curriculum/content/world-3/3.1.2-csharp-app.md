# AddCSharpApp — Path-Based Projects

`AddCSharpApp` is for .NET projects that are not part of your solution — when you have a path to a `.csproj` but no project reference.

---

## When to Use AddCSharpApp vs AddProject

| Method | Reference Type | Best For |
|---|---|---|
| `AddProject<T>()` | Project reference in `.csproj` | Projects in the same solution |
| `AddCSharpApp()` | File system path | External projects, separate repos, downloaded samples |

## Usage

```csharp
builder.AddCSharpApp("external-api", "../other-repo/Api")
    .WithHttpEndpoint(port: 5100);
```

The first parameter is the resource name. The second is the path to the directory containing the `.csproj` file.

## How It Works

Under the hood, `AddCSharpApp` creates an `ExecutableResource` that runs `dotnet run` in the specified directory. It:

1. Resolves the `.csproj` path
2. Registers the resource with the app model
3. Configures DCP to start it with `dotnet run --project <path>`
4. Injects environment variables just like any other resource

## Limitations

Because there is no project reference, you lose:
- The generated `Projects.MyApi` type (no generic `AddProject<T>` syntax)
- Compile-time validation that the project exists
- Automatic rebuilds when the referenced project changes

Use `AddProject<T>()` when possible. Reserve `AddCSharpApp` for cross-repo scenarios.
