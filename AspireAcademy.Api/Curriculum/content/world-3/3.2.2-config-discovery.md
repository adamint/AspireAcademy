# Configuration-Based Service Discovery

How service discovery reads endpoint URIs from configuration and environment variables.

## The Configuration Source

Service discovery reads endpoint information from `IConfiguration`. The AppHost injects environment variables that .NET configuration flattens into a hierarchy:

```
services__api__https__0 = https://localhost:5001
```

Becomes this configuration path:
```json
{
  "services": {
    "api": {
      "https": {
        "0": "https://localhost:5001"
      }
    }
  }
}
```

## Multiple Endpoints

A service can have multiple endpoints of the same scheme:

```
services__api__https__0 = https://localhost:5001
services__api__https__1 = https://localhost:5002
```

Service discovery can load-balance across them.

## Manual Configuration

You can configure service discovery without the AppHost by adding entries to `appsettings.json`:

```json
{
  "services": {
    "catalog": {
      "https": {
        "0": "https://catalog.example.com:443"
      }
    }
  }
}
```

This is useful for connecting to external services not managed by Aspire.

## Service Discovery Options

Configure allowed schemes:

```csharp
builder.Services.Configure<ServiceDiscoveryOptions>(options =>
{
    options.AllowedSchemes = ["https"];
});
```

This restricts service discovery to only resolve HTTPS endpoints.

## Resolution Priority

Service discovery checks multiple sources in order:
1. **Configuration** — environment variables and appsettings.json
2. **DNS** — SRV records (if configured)
3. **Pass-through** — if the URI is already a valid hostname, use it directly

## Debugging Service Discovery

If a service can't be found, check:
1. The environment variables in the Dashboard's resource detail view
2. The service name matches between `WithReference` and `HttpClient.BaseAddress`
3. The scheme (http/https) matches the endpoint scheme
