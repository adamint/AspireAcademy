# How Services Consume Injected Values

The AppHost injects environment variables; the consuming service reads them. Here's how to access injected connection strings and endpoint URLs in your service code.

## Consuming Connection Strings

For resources with connection strings (databases, caches, brokers):

```csharp
// Program.cs of the consuming service
var connectionString = builder.Configuration.GetConnectionString("inventorydb");
// Returns: "Host=localhost;Port=5432;Username=postgres;Password=xyz;Database=inventorydb"
```

`GetConnectionString("name")` reads from `ConnectionStrings:name` in the configuration hierarchy, which maps to the `ConnectionStrings__name` environment variable injected by the AppHost.

## Using Client Integration Packages

In practice, you rarely call `GetConnectionString()` directly. Aspire's client integration packages handle it:

```csharp
// Reads GetConnectionString("inventorydb") internally
builder.AddNpgsqlDataSource("inventorydb");

// Reads GetConnectionString("cache") internally
builder.AddRedisClient("cache");

// Reads GetConnectionString("messaging") internally
builder.AddRabbitMQClient("messaging");
```

These packages register the appropriate client in DI, configure health checks, and set up telemetry — all from the connection name.

## Consuming Endpoint URLs

For project-to-project references:

```csharp
// Direct configuration access (rarely needed)
var apiUrl = builder.Configuration["services:apiservice:http:0"];
// Returns: "http://localhost:5123"
```

## Service Discovery (The Preferred Pattern)

Rather than reading raw endpoint URLs, use Aspire's service discovery:

```csharp
builder.AddServiceDefaults(); // Registers ServiceDiscoveryHttpMessageHandler

builder.Services.AddHttpClient<CatalogClient>(client =>
    client.BaseAddress = new Uri("http://apiservice"));
```

The `ServiceDiscoveryHttpMessageHandler` resolves `http://apiservice` to `http://localhost:5123` using the injected `services__apiservice__http__0` variable. Your code never sees raw URLs.

## The Abstraction Layers

```
AppHost: .WithReference(postgres)
  → Environment: ConnectionStrings__inventorydb=Host=...;Port=5432;...
    → IConfiguration: GetConnectionString("inventorydb")
      → Client Package: builder.AddNpgsqlDataSource("inventorydb")
        → DI: NpgsqlDataSource registered and ready
```

## Key Points

- Connection strings: `GetConnectionString("name")` or client integration packages
- Endpoints: Service discovery with `http://resourcename` is preferred over raw config access
- Client integration packages abstract away configuration reading, health checks, and telemetry
- Most service code should be clean of infrastructure wiring logic
