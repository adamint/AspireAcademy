# WaitBehavior and Dependency Graphs

`WaitBehavior` controls what happens when a dependency fails or becomes unavailable. Understanding dependency graphs and circular detection helps you design robust startup ordering.

## WaitBehavior Enum

```csharp
public enum WaitBehavior
{
    WaitOnResourceUnavailable,   // Keep waiting if dependency becomes unavailable
    StopOnResourceUnavailable    // Stop the waiting resource if dependency fails
}
```

## Using WaitBehavior

```csharp
var api = builder.AddProject<Projects.Api>("api")
    .WaitFor(postgres, WaitBehavior.StopOnResourceUnavailable);
```

- **`WaitOnResourceUnavailable`** (default): If the dependency crashes and restarts, the waiting resource continues waiting for it to become healthy again
- **`StopOnResourceUnavailable`**: If the dependency becomes unavailable, the waiting resource is stopped — it doesn't endlessly wait for recovery

## When to Use Each

| Behavior | Use When |
|----------|----------|
| `WaitOnResourceUnavailable` | Dependencies that may restart (containers with persistent lifetime) |
| `StopOnResourceUnavailable` | Hard dependencies that should fail fast if unavailable |

## Dependency Graphs

`WaitFor()` and `WaitForCompletion()` create a directed graph of dependencies. Aspire resolves this graph to determine startup order:

```csharp
var postgres = builder.AddPostgres("pg").AddDatabase("db");
var migrator = builder.AddProject<Projects.Migrator>("migrator")
    .WaitFor(postgres);
var api = builder.AddProject<Projects.Api>("api")
    .WaitFor(postgres)
    .WaitForCompletion(migrator);
var web = builder.AddProject<Projects.Web>("web")
    .WaitFor(api);
```

Startup order: `postgres` → `migrator` → `api` → `web`

## Circular Dependency Detection

Aspire detects circular dependencies at startup:

```csharp
// This would cause a circular dependency error:
var a = builder.AddProject<Projects.A>("a").WaitFor(b);
var b = builder.AddProject<Projects.B>("b").WaitFor(a);
```

Aspire throws an `InvalidOperationException` describing the cycle. If you encounter this, rethink your dependency structure — usually one service should be the "root" that doesn't wait for the other.

## Key Points

- `WaitBehavior.StopOnResourceUnavailable` fails fast when dependencies are down
- Default behavior keeps waiting for recovery
- Aspire builds a dependency graph from `WaitFor()` and `WaitForCompletion()` calls
- Circular dependencies are detected and throw at startup
- Design dependency graphs as DAGs (directed acyclic graphs)
