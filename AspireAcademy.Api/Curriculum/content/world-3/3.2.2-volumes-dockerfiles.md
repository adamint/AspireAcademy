# Volumes, Bind Mounts, and Dockerfiles

Containers are ephemeral by default — data is lost when they stop. Volumes and bind mounts solve this. Dockerfiles let you build custom images.

---

## Named Volumes

Named volumes persist data in a Docker-managed location:

```csharp
builder.AddPostgres("pg")
    .WithDataVolume("pg-data");
```

The volume `pg-data` survives container restarts and AppHost restarts (when combined with `ContainerLifetime.Persistent`).

## Bind Mounts

Bind mounts map a host directory into the container:

```csharp
builder.AddContainer("nginx", "nginx")
    .WithBindMount("./nginx.conf", "/etc/nginx/nginx.conf");
```

Changes to `./nginx.conf` on your host are immediately visible inside the container (no rebuild needed).

## Building from Dockerfiles

You can build a custom image from a Dockerfile:

```csharp
builder.AddContainer("custom-api", "custom-api")
    .WithDockerfile("../custom-api")
    .WithBuildArg("ENV", "development");
```

The path points to the build context directory containing the `Dockerfile`. Aspire calls `docker build` before starting the container.

## Combining Techniques

```csharp
builder.AddPostgres("pg")
    .WithLifetime(ContainerLifetime.Persistent)
    .WithDataVolume("pg-data")
    .WithBindMount("./init.sql", "/docker-entrypoint-initdb.d/init.sql");
```

This creates a PostgreSQL container with:
- Persistent lifetime (survives AppHost restarts)
- Named volume for data persistence
- An init script mounted from the host

> 🐛 **Common Gotcha:** Bind mount paths are relative to the AppHost project directory. If you write `./init.sql`, the file must be next to your AppHost `.csproj` — not in the solution root. A missing file silently creates an empty directory in the container instead of mounting the file.
