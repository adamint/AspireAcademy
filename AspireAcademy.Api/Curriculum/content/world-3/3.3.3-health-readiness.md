# Health-Based Readiness

Aspire's `ResourceHealthCheckService` is the background service that monitors resource health and determines when resources are "ready." Understanding this system explains how `WaitFor()` works under the hood.

## How It Works

1. **Resource starts** → enters `Running` state
2. **Health check polling begins** → `ResourceHealthCheckService` checks registered health checks
3. **Health report is Healthy** → `ResourceReadyEvent` is fired
4. **Waiting resources proceed** → Resources blocked by `WaitFor()` can now start

## HealthCheckAnnotation

Resources register health checks via `HealthCheckAnnotation`:

```csharp
// Hosting integrations add health checks automatically
// For example, AddPostgres registers a TCP connection health check
```

Most built-in hosting integrations (PostgreSQL, Redis, SQL Server, RabbitMQ) register health checks automatically. Generic `AddContainer()` resources do **not** — they're considered ready immediately upon entering `Running` state.

## The Polling Loop

`ResourceHealthCheckService` runs as a `BackgroundService`:

1. It watches `ResourceNotificationService` for resources entering `Running` state
2. For each running resource with `HealthCheckAnnotation`, it polls `HealthCheckService.CheckHealthAsync()`
3. When the health report status is `Healthy`, it fires `ResourceReadyEvent`
4. If no health checks exist, `ResourceReadyEvent` fires immediately

## ResourceReadyEvent

`ResourceReadyEvent` is part of Aspire's eventing system:

```csharp
// Aspire fires this internally — you can subscribe to it
builder.Eventing.Subscribe<ResourceReadyEvent>(resource, (evt, ct) =>
{
    // Resource is healthy and ready
    return Task.CompletedTask;
});
```

## Adding Custom Health Checks

For custom resources, register health checks:

```csharp
var custom = builder.AddContainer("myservice", "myimage:latest")
    .WithHttpEndpoint(targetPort: 8080, name: "http")
    .WithHttpHealthCheck("/health");
```

## Resources Without Health Checks

When a resource has no health checks:
- `WaitFor()` still works — it just considers the resource ready once it's `Running`
- There's no health polling overhead
- This is fine for resources that are ready as soon as their process starts

## Key Points

- `ResourceHealthCheckService` polls health checks on running resources
- Resources are "ready" when health checks pass or when no health checks exist
- Built-in hosting integrations register health checks automatically
- `ResourceReadyEvent` signals that a resource is healthy and ready for consumers
- `WaitFor()` blocks until `ResourceReadyEvent` fires for the dependency
