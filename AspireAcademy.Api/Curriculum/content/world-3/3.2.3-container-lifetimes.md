# Container Lifetimes: Session vs Persistent

Every container resource has a **lifetime** that controls whether it survives AppHost restarts.

---

## Two Modes

| Mode | Behavior |
|---|---|
| `ContainerLifetime.Session` | Container created on startup, destroyed on shutdown (default) |
| `ContainerLifetime.Persistent` | Container survives AppHost restarts, reuses existing state |

## Setting the Lifetime

```csharp
builder.AddRedis("cache")
    .WithLifetime(ContainerLifetime.Persistent);
```

In TypeScript:
```typescript
await cache.withLifetime(ContainerLifetime.Persistent);
```

In Python:
```python
cache.with_lifetime(ContainerLifetime.PERSISTENT)
```

## When to Use Each

**Session (default)** — Use for stateless services or when you want a clean slate every time you restart development. Good for message brokers (RabbitMQ) where leftover messages might cause confusion.

**Persistent** — Use for databases and caches where losing data between restarts is painful. Combined with a data volume, your database retains schema, data, and migrations across development sessions.

## How DCP Handles Each

- **Session**: DCP creates the container on startup and sends `docker rm -f` on shutdown
- **Persistent**: DCP checks if the container already exists. If so, it starts it. If not, it creates a new one. On shutdown, the container is stopped but not removed.

The Dashboard shows both types with the same resource state indicators — the difference is only visible in what happens when you stop and restart.

> 🌍 **Real World:** Most teams settle on persistent databases + session everything-else. But during CI, everything should be session-lifetime for test isolation. You can use `if (builder.ExecutionContext.IsRunMode)` to conditionally set lifetimes.
