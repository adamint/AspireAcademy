# AddProject<T> — .NET Services

`AddProject<T>()` registers a .NET project as an Aspire resource. This is the most common resource type you will use — it is how your C# services become part of the Aspire app model.

---

## Basic Usage

```csharp
var api = builder.AddProject<Projects.BazaarApi>("api");
```

This tells Aspire: "there is a .NET project called BazaarApi — start it, give it a name, and make it discoverable."

But there is more happening than you might expect.

## How Project References Work

In a .csproj AppHost, you add a `<ProjectReference>`:

```xml
<ProjectReference Include="../Bazaar.Api/Bazaar.Api.csproj" />
```

The Aspire SDK generates a `Projects.BazaarApi` type that you use as the generic parameter. In a single-file AppHost, you use a directive:

```csharp
#:project ../Bazaar.Api/Bazaar.Api.csproj
```

In a TypeScript AppHost, you pass the path directly:

```typescript
const api = await builder.addProject("api", "../Bazaar.Api");
```

```reveal
{"question": "In the TypeScript version, there's no generic type parameter — you just pass the path as a string. What do you lose compared to the C# version?", "answer": "You lose compile-time verification. In C#, if you rename or delete the Bazaar.Api project, Projects.BazaarApi disappears and you get a build error immediately. In TypeScript, the string path \"../Bazaar.Api\" isn't checked until runtime — the AppHost starts, tries to find the project at that path, and fails.\n\nThis is a deliberate trade-off: TypeScript AppHosts are simpler to write (no project references, no code generation), but they catch path errors later. For teams that prefer TypeScript, this is usually acceptable because 'aspire run' fails fast with a clear error message."}
```

## Replicas

Run multiple instances of the same service for load testing:

```csharp
builder.AddProject<Projects.BazaarApi>("api")
    .WithReplicas(3);
```

DCP creates three instances, each with its own port. Service discovery load-balances across them.

## External Endpoints

Expose a service publicly (outside the Aspire network):

```csharp
builder.AddProject<Projects.BazaarApi>("api")
    .WithExternalHttpEndpoints();
```

## Launch Profiles

By default, Aspire uses the `http` launch profile from `launchSettings.json`. Override with:

```csharp
builder.AddProject<Projects.BazaarApi>("api")
    .WithLaunchProfile("https");
```

> ⚠️ **Common gotcha:** If your `launchSettings.json` doesn't have the profile you specify, Aspire throws at startup. Double-check your profile names if you see a "launch profile not found" error.
