# Deployment Commands

The Aspire CLI provides two pipeline-driven commands for deploying applications: `publish` generates deployment artifacts, while `deploy` goes further and applies them to a target environment.

## aspire publish

Runs the pipeline up to the `publish` step — building images, generating manifests, compose files, or Helm charts.

```bash
aspire publish                                    # publish to default target
aspire publish --project ./MyApp.AppHost           # specify AppHost
```

| Option | Description |
|--------|-------------|
| `--apphost/--project` | AppHost project path |
| `--log-level` | Pipeline log verbosity |
| `--include-exception-details` | Show full exception info |
| `--environment` | Target environment name |
| `--format` | Output format |

`PublishCommand` extends `PipelineCommandBase` with `--operation publish --step publish`. It launches the AppHost in publish mode, connects via backchannel, and watches the pipeline execute. The pipeline runs steps up to and including `WellKnownPipelineSteps.Publish` — building container images, pushing to registries, and writing deployment artifacts.

## aspire deploy

Runs the full pipeline through `deploy` — everything `publish` does plus actual deployment (e.g., `docker compose up` or Helm install).

```bash
aspire deploy                                     # deploy to target
aspire deploy --clear-cache                        # force full redeploy
```

| Option | Description |
|--------|-------------|
| `--clear-cache` | Clear deployment state cache for full redeploy |
| `--apphost/--project` | AppHost project path |
| `--log-level` | Pipeline log verbosity |
| `--include-exception-details` | Show full exception info |
| `--environment` | Target environment name |

`DeployCommand` extends `PipelineCommandBase` with `--operation publish --step deploy`. The `--clear-cache` flag clears the deployment state cache, forcing a complete rebuild and redeploy rather than an incremental update.

## Pipeline Execution Flow

Both commands share the same execution model via `PipelineCommandBase`:

1. **Locate AppHost** — find and validate the AppHost project
2. **Launch in publish mode** — start the AppHost with the pipeline operation
3. **Connect backchannel** — establish communication with the running pipeline
4. **Monitor progress** — stream step execution status and reporting
5. **Collect summary** — display the `PipelineSummary` key-value pairs

## aspire export

Exports deployment artifacts without running the pipeline:

```bash
aspire export --output ./deploy
```

## Deployment State Cache

The pipeline caches deployment state between runs. On subsequent deploys, unchanged resources are skipped. The `deploy-prereq` step manages this cache via `FileDeploymentStateManager` or `UserSecretsDeploymentStateManager`. Use `--clear-cache` to force a full redeploy when the cache is stale.

## End-to-End Example

```bash
# Scaffold a new app
aspire new aspire-starter --name MyApp

# Add Docker Compose publishing to AppHost
# (in Program.cs: builder.AddDockerComposeEnvironment("compose"))

# Publish artifacts
aspire publish --project ./MyApp/MyApp.AppHost

# Deploy (build, push, docker compose up)
aspire deploy --project ./MyApp/MyApp.AppHost
```
