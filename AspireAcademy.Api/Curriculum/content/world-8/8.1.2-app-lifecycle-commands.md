# App Lifecycle Commands

The Aspire CLI provides five commands that cover the full application lifecycle — from creating a new project to stopping a running one.

## aspire new

Scaffolds a new Aspire application from a template.

```bash
aspire new aspire-starter --name MyApp --output ./MyApp
```

| Option | Description |
|--------|-------------|
| `--name/-n` | Project name |
| `--output/-o` | Output directory |
| `--source/-s` | NuGet package source |
| `--version` | Template package version |
| `--channel` | Release channel (stable, preview, etc.) |

`NewCommand` dynamically registers template subcommands from `ITemplateProvider.GetTemplates()`. After scaffolding, it offers to run `aspire agent init` to set up AI agent integration. Template-specific options (like `--use-redis-cache` or `--test-framework`) are added per-template.

## aspire init

Adds Aspire orchestration to an **existing** .NET solution.

```bash
aspire init --language csharp
```

| Option | Description |
|--------|-------------|
| `--source/-s` | NuGet package source |
| `--version` | Template package version |
| `--channel` | Release channel |
| `--language` | Target language (`csharp`, `typescript`, `java`) |

For C#, `InitCommand` finds the solution file, installs `Aspire.ProjectTemplates` via `dotnet new`, creates AppHost and ServiceDefaults projects in a temp directory, copies them into the solution, adds project references, and trusts dev certificates. For non-C# languages, it creates a polyglot AppHost via CLI scaffolding.

## aspire run

Builds and runs the AppHost, streaming logs and resource state to the terminal.

```bash
aspire run --project ./MyApp.AppHost
aspire run --detach  # runs in background
```

| Option | Description |
|--------|-------------|
| `--detach` | Run in background, print summary |
| `--no-build` | Skip building before running |
| `--apphost/--project` | Path to AppHost project |
| `--format` | Output format (`table`/`json`) |
| `--isolated` | Run without backchannel connection |

In non-detach mode, `RunCommand` connects to the AppHost's backchannel, streams resource state changes, and opens the Dashboard URL. In detach mode, it spawns a child CLI process and waits for the backchannel to report readiness.

## aspire start / aspire stop

`start` is a thin wrapper around `AppHostLauncher.LaunchDetachedAsync()` — it always runs the AppHost in background:

```bash
aspire start --no-build
```

`stop` gracefully terminates running AppHosts:

```bash
aspire stop                    # stop the current AppHost
aspire stop --all              # stop all running AppHosts
aspire stop --project ./path   # stop a specific AppHost
```

`StopCommand` uses a three-tier shutdown strategy: CLI PID kill → RPC shutdown → AppHost PID fallback. With `--all`, it discovers all running instances and stops them.
