# DistributedApplicationTestingBuilder

`DistributedApplicationTestingBuilder` builds your AppHost's resource model for integration tests — real Docker containers, real DCP orchestration, no Dashboard.

---

## Basic Setup

```csharp
public class BazaarApiTests : IAsyncLifetime
{
    private DistributedApplication _app = null!;

    public async Task InitializeAsync()
    {
        var builder = await DistributedApplicationTestingBuilder
            .CreateAsync<Projects.BazaarAppHost>();

        _app = await builder.BuildAsync();
        await _app.StartAsync();
    }

    public async Task DisposeAsync()
    {
        await _app.DisposeAsync();
    }

    [Fact]
    public async Task ApiHealthEndpointReturns200()
    {
        var client = _app.CreateHttpClient("api");
        var response = await client.GetAsync("/health");
        response.StatusCode.Should().Be(HttpStatusCode.OK);
    }
}
```

## What It Provides

- **`CreateAsync<T>()`** — loads your AppHost project and builds the app model
- **`BuildAsync()`** — creates the `DistributedApplication` in-memory
- **`StartAsync()`** — starts all resources (using real Docker containers)
- **`CreateHttpClient("name")`** — returns an `HttpClient` configured to talk to the named resource

## DCP in Tests

Unlike unit tests, integration tests with `DistributedApplicationTestingBuilder` **do** start real containers via DCP. PostgreSQL, Redis, and RabbitMQ run as actual Docker containers. This gives you high-fidelity tests at the cost of slower execution.

> 🐛 **Common Gotcha:** The first test run pulls container images, which can take 30+ seconds. Subsequent runs reuse cached images. If your CI has no Docker cache, consider using `ContainerLifetime.Persistent` or pre-pulling images in your pipeline.

## Test Isolation

Each test class gets its own AppHost instance. Resources are created fresh and destroyed after the test. This ensures tests do not leak state between each other.

```deepdive

DCP spins up fresh containers with randomized port assignments, so two test classes running in parallel never collide.\n\nDisposeAsync() sends SIGTERM to all containers and waits for them to stop. If a test crashes without calling DisposeAsync(), orphaned containers may linger — xUnit's IAsyncLifetime guarantees cleanup even on test failure, which is why it's the recommended pattern over manual try/finally blocks."}
```

> 🎲 **Fun Fact:** Aspire's own test suite uses this exact pattern — over 1,200 integration tests run against real containers in the Aspire GitHub repo's CI pipeline.
