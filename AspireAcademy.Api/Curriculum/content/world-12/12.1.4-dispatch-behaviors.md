# Dispatch Behaviors

When publishing an event, you choose how handlers execute. The four behaviors trade off between blocking/non-blocking and sequential/concurrent execution.

---

## The Enum

```csharp
// src/Aspire.Hosting/Eventing/EventDispatchBehavior.cs

public enum EventDispatchBehavior
{
    /// <summary>
    /// Fires events sequentially and blocks until they are all processed.
    /// </summary>
    BlockingSequential,

    /// <summary>
    /// Fires events concurrently and blocks until they are all processed.
    /// </summary>
    BlockingConcurrent,

    /// <summary>
    /// Fires events sequentially but does not block.
    /// </summary>
    NonBlockingSequential,

    /// <summary>
    /// Fires events concurrently but does not block.
    /// </summary>
    NonBlockingConcurrent
}
```

## How Each Behavior Executes (from the source)

The actual dispatch logic in `DistributedApplicationEventing.PublishAsync`:

```csharp
public async Task PublishAsync<T>(T @event,
    EventDispatchBehavior dispatchBehavior, CancellationToken ct)
{
    if (_eventSubscriptionListLookup.TryGetValue(typeof(T), out var subscriptions))
    {
        if (dispatchBehavior is BlockingConcurrent or NonBlockingConcurrent)
        {
            // Start all handlers simultaneously
            var pending = new List<Task>(subscriptions.Count);
            foreach (var sub in subscriptions.ToArray())
            {
                pending.Add(sub.Callback(@event, ct));
            }

            if (dispatchBehavior == NonBlockingConcurrent)
            {
                // Fire and forget via Task.Run
                _ = Task.Run(async () =>
                    await Task.WhenAll(pending).ConfigureAwait(false));
            }
            else
            {
                // Block until all complete
                await Task.WhenAll(pending).ConfigureAwait(false);
            }
        }
        else // Sequential
        {
            if (dispatchBehavior == NonBlockingSequential)
            {
                _ = Task.Run(async () =>
                {
                    foreach (var sub in subscriptions.ToArray())
                        await sub.Callback(@event, ct).ConfigureAwait(false);
                });
            }
            else // BlockingSequential (default)
            {
                foreach (var sub in subscriptions.ToArray())
                    await sub.Callback(@event, ct).ConfigureAwait(false);
            }
        }
    }
}
```

Key observations:
- **`subscriptions.ToArray()`** — creates a snapshot to avoid issues if a handler adds/removes subscriptions
- **Non-blocking variants use `Task.Run`** — the fire-and-forget pattern
- **Default is `BlockingSequential`** — calling `PublishAsync(@event)` without a behavior uses this

## When to Use Each

| Behavior | Use Case |
|---|---|
| **BlockingSequential** | Validation, state modification, must complete before lifecycle continues |
| **BlockingConcurrent** | Independent setup tasks that must finish but can run in parallel |
| **NonBlockingSequential** | Notifications, logging — ordered but lifecycle should not wait |
| **NonBlockingConcurrent** | Independent background tasks — fan-out notifications |

Most framework events use `BlockingSequential` because handlers may modify the app model before the next lifecycle phase.
