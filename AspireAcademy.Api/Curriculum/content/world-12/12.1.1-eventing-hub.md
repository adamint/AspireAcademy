# IDistributedApplicationEventing

The eventing system is Aspire's pub/sub mechanism for lifecycle events. Every significant moment in the AppHost lifecycle — resource starting, endpoints allocated, health check passing — publishes an event that your code can subscribe to.

---

## The Interface (from the source)

```csharp
// src/Aspire.Hosting/Eventing/IDistributedApplicationEventing.cs

[AspireExport(ExposeMethods = true)]
public interface IDistributedApplicationEventing
{
    // Subscribe to application-wide events
    DistributedApplicationEventSubscription Subscribe<T>(
        Func<T, CancellationToken, Task> callback)
        where T : IDistributedApplicationEvent;

    // Subscribe to events for a specific resource
    DistributedApplicationEventSubscription Subscribe<T>(
        IResource resource,
        Func<T, CancellationToken, Task> callback)
        where T : IDistributedApplicationResourceEvent;

    // Remove a subscription
    void Unsubscribe(DistributedApplicationEventSubscription subscription);

    // Publish with default behavior (BlockingSequential)
    Task PublishAsync<T>(T @event, CancellationToken cancellationToken = default)
        where T : IDistributedApplicationEvent;

    // Publish with explicit dispatch behavior
    Task PublishAsync<T>(T @event, EventDispatchBehavior dispatchBehavior,
        CancellationToken cancellationToken = default)
        where T : IDistributedApplicationEvent;
}
```

Note the `[AspireExport(ExposeMethods = true)]` attribute — this is what makes the eventing system available to polyglot AppHosts via the RemoteHost bridge.

> 🎲 **Fun Fact:** The `[AspireExport]` attribute powers the entire polyglot story. When a TypeScript or Python AppHost calls `builder.eventing.subscribe(...)`, that call crosses a Unix socket into the .NET AppHost and hits this exact interface via the RemoteHost bridge.

## Two Flavors of Subscribe

**Application-wide**: Fires for every resource.

```csharp
builder.Eventing.Subscribe<BeforeStartEvent>(async (evt, ct) =>
{
    // Fires once, before any resource starts
    foreach (var resource in evt.Model.Resources)
    {
        Console.WriteLine($"Will start: {resource.Name}");
    }
});
```

**Resource-scoped**: Fires only for a specific resource.

```csharp
builder.Eventing.Subscribe<ResourceReadyEvent>(
    builder.GetResource("api"),
    async (evt, ct) =>
    {
        // Fires only when "api" becomes ready
        Console.WriteLine($"{evt.Resource.Name} is ready");
    });
```

The resource-scoped version works by wrapping your callback with a filter:

```csharp
// From DistributedApplicationEventing.cs
public DistributedApplicationEventSubscription Subscribe<T>(
    IResource resource, Func<T, CancellationToken, Task> callback)
    where T : IDistributedApplicationResourceEvent
{
    var resourceFilteredCallback = async (T @event, CancellationToken ct) =>
    {
        if (@event.Resource == resource)
        {
            await callback(@event, ct).ConfigureAwait(false);
        }
    };
    return Subscribe(resourceFilteredCallback);
}
```

It subscribes to ALL events of that type, but the wrapper only invokes your callback when the event's `Resource` matches. Simple and effective.

## The Implementation Internals

`DistributedApplicationEventing` uses two `ConcurrentDictionary` lookups:

```csharp
// Type → List of subscriptions for that event type
ConcurrentDictionary<Type, List<DistributedApplicationEventSubscription>>
    _eventSubscriptionListLookup;

// Subscription → Type (for reverse lookup during Unsubscribe)
ConcurrentDictionary<DistributedApplicationEventSubscription, Type>
    _subscriptionEventTypeLookup;
```

When you subscribe, your callback is wrapped in a `DistributedApplicationEventSubscription`:

```csharp
public class DistributedApplicationEventSubscription(
    Func<IDistributedApplicationEvent, CancellationToken, Task> callback)
{
    public Func<IDistributedApplicationEvent, CancellationToken, Task> Callback { get; }
        = callback;
}
```

## The Two Event Interfaces

```csharp
// Application-wide — no resource context
public interface IDistributedApplicationEvent
{
}

// Resource-specific — includes resource and services
public interface IDistributedApplicationResourceEvent : IDistributedApplicationEvent
{
    IResource Resource { get; }
    IServiceProvider Services { get; }
    ILogger Logger => Services.GetRequiredService<ResourceLoggerService>()
        .GetLogger(Resource);
}
```

Resource events automatically provide a per-resource `ILogger` via the `Logger` property — you can log directly to the resource's log stream, and it appears in the Dashboard's Console Logs.

## Accessing Eventing

In your AppHost:

```csharp
builder.Eventing.Subscribe<BeforeStartEvent>(...);
```

In a service registered with DI:

```csharp
public class MyService(IDistributedApplicationEventing eventing)
{
    public void Setup()
    {
        eventing.Subscribe<ResourceReadyEvent>(async (evt, ct) => { ... });
    }
}
```
