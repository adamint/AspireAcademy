# The Resource State Machine

Resources transition through a set of well-known states. Understanding these transitions is essential for debugging WaitFor behavior, interpreting Dashboard indicators, and writing custom event handlers.

---

## State Transitions

```
              +---------------+
              |   Starting    |
              +-------+-------+
                      |
         +------------+------------+
         v            v            v
  +-----------+ +-----------+ +----------------+
  |  Running  | |  Waiting  | | FailedToStart  |
  +-----+-----+ +-----------+ +----------------+
        |
   +----+-----+
   v          v
+----------+ +--------+
| Stopping | | Exited |
+----+-----+ +--------+
     v
+----------+
| Stopped  |
+----------+
```

## KnownResourceStates (from the source)

The framework defines these state constants:

| State | KnownResourceStateStyles | Dashboard Color |
|---|---|---|
| `Starting` | `Info` | Blue spinner |
| `Running` | `Success` | Green |
| `Waiting` | `Info` | Yellow |
| `FailedToStart` | `Error` | Red |
| `Stopping` | `Info` | Transitional |
| `Exited` | `Info` | Gray |
| `Finished` | `Success` | Gray (exit code 0) |
| `RuntimeUnhealthy` | `Warning` | Orange |
| `Hidden` | — | Not displayed |

## How WaitFor Consumes States

When you write `api.WaitFor(db)`:

1. DCP starts the database container
2. `ResourceNotificationService` publishes `Starting`
3. WaitFor subscribes and holds the API in `Waiting`
4. Health check polls the database
5. Health check passes — state becomes `Running` with `HealthStatus.Healthy`
6. `ResourceReadyEvent` fires
7. WaitFor releases — API transitions from `Waiting` to `Starting`

## Exit Codes

When a resource exits:

```csharp
snapshot => snapshot with
{
    State = new ResourceStateSnapshot(
        exitCode == 0 ? "Finished" : "Exited",
        exitCode == 0 ? KnownResourceStateStyles.Success
                      : KnownResourceStateStyles.Error),
    ExitCode = exitCode
}
```

- **Exit code 0** → `Finished` with Success style (normal completion)
- **Non-zero** → `Exited` with Error style (unexpected failure)

`WaitForCompletion` only succeeds when the resource reaches `Finished` (exit code 0).

## Health Status Integration

Resources can be `Running` but unhealthy:

```csharp
snapshot => snapshot with
{
    State = new ResourceStateSnapshot("Running",
        KnownResourceStateStyles.Success),
    HealthStatus = HealthStatus.Unhealthy,
    HealthReports = [
        new HealthReportSnapshot("pg_check", HealthStatus.Unhealthy,
            "Connection refused", null)
    ]
}
```

The Dashboard shows this as `RuntimeUnhealthy` — the resource is running but health checks are failing.

> 🐛 **Common Gotcha:** A resource can be `Running` but not `Ready`. The `Running` state means the process/container is alive. The `Ready` state (which triggers `ResourceReadyEvent`) requires `HealthStatus.Healthy`. If you use `WaitFor` without configuring health checks, the resource becomes ready immediately when it starts running — even if the database inside isn't accepting connections yet. Always add health checks to container resources!
