# Publish and Deploy Commands

Generate deployment artifacts and deploy to cloud targets.

---

## aspire publish

Generate deployment artifacts without deploying:

```bash
aspire publish                              # publish using the app model's target
aspire publish --output-path ./deploy       # specify output directory
aspire publish --environment Production     # set the environment name
aspire publish --non-interactive            # skip prompts (for CI/CD)
aspire publish --log-level debug            # verbose logging
```

There is no `--target` flag. The publish target is determined by what you add to the app model in your AppHost — for example, `AddDockerComposeEnvironment`, `AddKubernetesEnvironment`, or `AddAzureContainerAppEnvironment`.

The command:
1. Builds the AppHost in publish mode
2. Establishes a backchannel to the AppHost
3. Executes the publishing pipeline for the configured target
4. Streams progress back to the CLI
5. Writes artifacts to the output path

## aspire deploy

Publish and deploy to Azure in one step (preview):

```bash
aspire deploy                               # publish + deploy to Azure
aspire deploy --output-path ./deploy        # specify output directory
aspire deploy --clear-cache                 # force a clean rebuild
aspire deploy --non-interactive             # skip prompts (for CI/CD)
```

`aspire deploy` is specifically for Azure deployment. It runs the full pipeline: build → push → publish → deploy. For Kubernetes or Docker Compose, use `aspire publish` to generate artifacts and then deploy them with your own tooling (kubectl, docker compose, etc.).

Each line corresponds to a pipeline step executing in the DAG.

> 🌍 **Real World:** In CI/CD pipelines, use `aspire publish` to generate artifacts in one stage, then deploy them using your existing deployment tooling (Helm, Terraform, Azure CLI). This separates artifact generation from deployment, giving you full control over release gates and approvals.
