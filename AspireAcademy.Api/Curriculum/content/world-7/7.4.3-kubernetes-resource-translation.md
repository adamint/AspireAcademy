# Kubernetes Resource Translation

The `KubernetesResource` class decides how each Aspire resource maps to Kubernetes objects — choosing between Deployments and StatefulSets, wiring endpoints to Services, and handling volumes, probes, and environment variables.

## Deployment vs. StatefulSet

The choice depends on whether the resource implements `IResourceWithConnectionString`:

- **Resources with connection strings** (databases, caches) → `StatefulSet` with stable storage
- **Stateless compute** (projects, workers) → `Deployment`

This matches Kubernetes best practices: databases need stable network identities and persistent volumes, while application services can be freely scaled and replaced.

## Service Translation

Aspire endpoints map to Kubernetes `Service` ports. Each endpoint becomes a port entry with the endpoint's scheme, protocol, and target port. The default service type is `ClusterIP` (internal-only), configurable via `KubernetesEnvironmentResource.DefaultServiceType`.

Explicit target ports are preserved as literal values. Dynamically-allocated ports become Helm-backed values so operators can customize them at deploy time.

## ConfigMap and Secret

Non-sensitive environment variables go into a `ConfigMap`. Sensitive values — connection strings, parameter secrets — go into a `Secret`. The names follow conventions: `api-config` and `api-secrets`.

## Volume Handling

Aspire volumes map to Kubernetes `PersistentVolumeClaim` (PVC) objects using the environment's defaults:

- Storage size: `DefaultStorageSize` (`"1Gi"`)
- Access mode: `DefaultStorageReadWritePolicy` (`"ReadWriteOnce"`)
- Storage class: `DefaultStorageClassName` (cluster default if null)

**Bind mounts are rejected** — they don't translate to Kubernetes. If your Aspire app uses `WithBindMount()`, publish will report an error. Use `WithDataVolume()` instead.

## Health Probes

Resources with endpoint-based health checks get Kubernetes HTTP probes. The probe configuration maps Aspire health check endpoints to `httpGet` probes using the endpoint's scheme and port.

## Environment Variable Resolution

Environment variables are resolved through the same expression system as the manifest, but with Helm-specific output:

| Aspire Source | Kubernetes Output |
|--------------|-------------------|
| `EndpointReference` | Service URL (e.g., `http://cache:6379`) |
| `ParameterResource` | Helm parameter expression (`{{ .Values.parameters.name }}`) |
| `ReferenceExpression` (conditional) | Helm `if/else` template block |
| Literal string | Literal value in ConfigMap |

## Conditional Expressions as Helm Templates

When a conditional expression's condition is a parameter, the publisher emits a Helm `{{ if }}` / `{{ else }}` template:

```yaml
{{- if .Values.parameters.use-managed-db }}
  CONNECTION_STRING: {{ .Values.secrets.managed-db.connectionstring }}
{{- else }}
  CONNECTION_STRING: {{ .Values.secrets.local-db.connectionstring }}
{{- end }}
```

This powerful mapping lets operators toggle between deployment modes using a single Helm value.
