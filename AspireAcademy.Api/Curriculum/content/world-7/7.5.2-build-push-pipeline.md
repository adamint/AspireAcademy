# Build and Push Pipeline

Container image building and pushing are fully integrated into the Aspire pipeline. When you call `WithDockerfile` or register a `ProjectResource`, the pipeline automatically creates build and push steps that run at the right time.

## Pipeline Step Wiring

`EnsureBuildAndPushPipelineAnnotations()` registers two pipeline steps per resource:

| Step | Tag | Depends On | Required By |
|------|-----|-----------|-------------|
| `build-<resource>` | `build-compute` | `build-prereq` | `build` |
| `push-<resource>` | `push-container-image` | `push-prereq`, `build-<resource>` | `push` |

This ensures all images are **built** before the `build` barrier and **pushed** before the `push` barrier. Environment resources (Docker Compose, Kubernetes) wire their publish steps to depend on these barriers.

## ResourceContainerImageManager

`ResourceContainerImageManager` is the service that actually builds and pushes images. It handles two scenarios:

### Project Resources
For `ProjectResource`, building runs `dotnet publish /t:PublishContainer`, which uses the .NET SDK's built-in container publishing support. If the project has container files (via `WithContainerFiles`), the manager:
1. Builds the base image
2. Tags it with a temporary name
3. Creates a temporary Dockerfile that layers container files on top
4. Rebuilds the final image

### Dockerfile Resources
For resources with `DockerfileBuildAnnotation`, the manager delegates to `IContainerRuntime.BuildImageAsync()` — running `docker build` (or equivalent) with the configured context, Dockerfile, stage, build args, and secrets.

## Image Destinations

The `ContainerImageDestination` enum controls where images go:

| Destination | Behavior |
|-------------|----------|
| `Registry` | Push to a container registry (default) |
| `Archive` | Save as a tar archive (skips push) |

When destination is `Archive`, the `OutputPath` property specifies where to write the tar file.

## Container Image Formats and Platforms

```csharp
builder.AddDockerfile("api", "./Api")
    .WithContainerImageFormat(ContainerImageFormat.Oci)
    .WithContainerTargetPlatform(ContainerTargetPlatform.LinuxAmd64);
```

Supported platforms include `LinuxAmd64`, `LinuxArm64`, `LinuxArm`, `Linux386`, `WindowsAmd64`, `WindowsArm64`, and `AllLinux` (multi-platform).

## Push Logic

`PipelineStepHelpers.PushImageToRegistryAsync()` handles push:
- If destination is `Archive`, push is skipped entirely
- If the registry endpoint is empty, the image is tagged locally (useful for local Docker Compose)
- Otherwise, `IResourceContainerImageManager.PushImageAsync()` pushes to the configured registry

## PublishAsDockerFile

For resources that aren't Dockerfile-based by default (projects, executables), `PublishAsDockerFile` converts them:

```csharp
builder.AddProject<Projects.Api>("api")
    .PublishAsDockerFile();
```

This adds the Dockerfile build annotation and pipeline steps, making the project build as a container image during publish rather than deploying as a bare process.
