# Blazor + gRPC + OTLP

The Aspire Dashboard is built from three technology layers, each handling a different data source.

---

## Architecture

```
                          Browser
                            |
                        SignalR WebSocket
                            |
                    +-------+--------+
                    | Blazor Server  |
                    +---+----+---+---+
                        |    |   |
            +-----------+    |   +------------+
            |                |                |
    gRPC streaming      OTLP receiver    Static assets
    (resource state)    (telemetry)
            |                |
            |        +-------+--------+
            |        | Traces (memory) |
    AppHost gRPC     | Metrics (memory)|
    service          | Logs (memory)   |
            |        +----------------+
            |
    ResourceNotificationService
    ResourceLoggerService
```

## Layer 1: Blazor Server (UI)

The Dashboard is a **Blazor Server** application. Blazor renders HTML on the server and pushes DOM diffs to the browser via SignalR WebSocket.

Key components:
- `ResourcesPage.razor` — renders the resources table, subscribes to `IDashboardClient`
- `ConsoleLogs.razor` — streams log lines from `IDashboardClient.SubscribeConsoleLogs()`
- `StructuredLogs.razor` — queries the in-memory OTLP log store
- `Traces.razor` — renders the waterfall timeline from OTLP trace store
- `Metrics.razor` — plots time-series charts from OTLP metric store

Why Blazor Server:
- **Real-time push** — SignalR sends only diffs, no polling
- **Server-side data access** — direct access to gRPC data without CORS
- **Minimal client JS** — the browser renders HTML diffs, no heavy bundle

## Layer 2: gRPC Resource Service

The Dashboard connects to the AppHost via gRPC to receive resource state:

```csharp
// IDashboardClient interface (from the source)
public interface IDashboardClient : IAsyncDisposable
{
    Task<ResourceViewModelSubscription> SubscribeResourcesAsync(CancellationToken ct);
    IAsyncEnumerable<IReadOnlyList<ResourceLogLine>> SubscribeConsoleLogs(
        string resourceName, CancellationToken ct);
    Task<ResourceCommandResponseViewModel> ExecuteResourceCommandAsync(
        string resourceName, string resourceType,
        CommandViewModel command, CancellationToken ct);
    IReadOnlyList<ResourceViewModel> GetResources();
}
```

The gRPC channel uses HTTP/2 with keepalive pings:

```csharp
return GrpcChannel.ForAddress(address, new()
{
    HttpHandler = new SocketsHttpHandler
    {
        EnableMultipleHttp2Connections = true,
        KeepAlivePingDelay = TimeSpan.FromSeconds(20),
    },
    MaxReceiveMessageSize = 16 * 1024 * 1024  // 16 MB
});
```

## Layer 3: OTLP Receiver

The Dashboard runs an OTLP receiver that ingests three signal types:

| Signal | Storage | Retention |
|---|---|---|
| **Traces** | In-memory ring buffer | Recent spans only |
| **Metrics** | In-memory time-series | Rolling window |
| **Logs** | In-memory buffer | Recent entries only |

DCP injects `OTEL_EXPORTER_OTLP_ENDPOINT` into every service. The OpenTelemetry SDK in each service (regardless of language) exports to this endpoint.

**All data is ephemeral** — when the AppHost stops, everything is gone. For production, export to Seq, Application Insights, Jaeger, or Prometheus.

> 🐛 **Common Gotcha:** All Dashboard telemetry data is stored in memory and lost when the AppHost stops. If you need to preserve traces or logs across restarts, configure a persistent OTLP backend (like Seq, Jaeger, or Application Insights) alongside the Dashboard.
