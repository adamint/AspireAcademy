# Manifest Expression Syntax

Aspire manifests use a **template expression syntax** to reference values that are only known at deploy time — endpoint URLs, connection strings, and parameter inputs. These expressions are enclosed in curly braces and follow dot-notation paths.

## Expression Format

The general pattern is `{resourceName.property.subProperty}`:

| Expression | Resolves To |
|-----------|-------------|
| `{cache.bindings.tcp.host}` | The hostname of the `cache` resource's `tcp` binding |
| `{cache.bindings.tcp.port}` | The port number of the `tcp` binding |
| `{cache.bindings.tcp.url}` | The full URL (e.g., `tcp://hostname:6379`) |
| `{pg.connectionString}` | The connection string of the `pg` resource |
| `{admin-password.inputs.value}` | The runtime value of the `admin-password` parameter |
| `{admin-password.value}` | The resolved value of the parameter |

## How Expressions Are Generated

Resources that implement `IManifestExpressionProvider` expose a `ValueExpression` property that returns the template string. For example, an `EndpointReference` for a container's TCP binding generates `{resourceName.bindings.endpointName.url}`.

Connection strings are composed by combining expressions. A PostgreSQL database resource produces:

```
{pg.connectionString};Database=pgdb
```

Here `{pg.connectionString}` resolves to the server-level connection string, and `;Database=pgdb` is appended literally.

## Conditional Expressions

Some resources produce **conditional expressions** — values that depend on whether a parameter is set. These are stored internally in `_conditionalExpressions` and emitted as separate `value.v0` resources after all primary resources:

```json
"pg-connection-conditional": {
  "type": "value.v0",
  "connectionString": "{pg.connectionString}"
}
```

This pattern handles scenarios where a connection string switches between a managed service and a local emulator based on a parameter flag.

## Formatted Parameters

When a parameter uses a format specifier (like Base64 encoding), the manifest creates a synthetic resource:

```json
"api-key-base64-encoded": {
  "type": "annotated.string",
  "value": "{api-key.value}",
  "filter": "base64"
}
```

The name follows the pattern `<parameter>-<sanitized-format>-encoded`, and the `filter` field tells the deployment tool how to transform the value.

## Why Expressions Matter

Expressions are the **bridge between your Aspire app model and any deployment target**. Docker Compose, Kubernetes, and Azure all consume the same manifest — they just resolve the expressions differently. Docker Compose maps them to environment variable placeholders (`${VAR}`). Kubernetes maps them to Helm template expressions (`{{ .Values.parameters.varName }}`). This abstraction is why Aspire can publish to multiple targets from a single app model.
