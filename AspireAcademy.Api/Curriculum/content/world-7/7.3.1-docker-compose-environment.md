# Docker Compose Environment

Publishing to Docker Compose transforms your Aspire app model into a `docker-compose.yaml` file that can run your entire application with a single `docker compose up`. The entry point is `AddDockerComposeEnvironment()`.

## Setting Up Docker Compose Publishing

Register a Docker Compose environment in your AppHost:

```csharp
var builder = DistributedApplication.CreateBuilder(args);

var compose = builder.AddDockerComposeEnvironment("compose");

var cache = builder.AddRedis("cache");
var api = builder.AddProject<Projects.Api>("api")
    .WithReference(cache);

builder.Build().Run();
```

`AddDockerComposeEnvironment()` creates a `DockerComposeEnvironmentResource` — a compute environment resource that tells the pipeline to emit Docker Compose output instead of (or alongside) a raw manifest.

## DockerComposeEnvironmentResource

This resource carries configuration for the entire compose environment:

- **`DefaultNetworkName`** — the bridge network name (defaults to `"aspire"`)
- **`DashboardEnabled`** — whether to include the Aspire Dashboard as a compose service

## Extension Methods

The `DockerComposeEnvironmentExtensions` class provides configuration methods:

| Method | Purpose |
|--------|---------|
| `AddDockerComposeEnvironment(builder, name)` | Registers the compose environment |
| `WithProperties(...)` | Sets environment-level properties |
| `ConfigureComposeFile(Action<ComposeFile>)` | Directly modify the generated `ComposeFile` object |
| `ConfigureEnvFile(Action<...>)` | Customize the `.env` file content |
| `WithDashboard(bool enabled)` | Enable/disable the Aspire Dashboard service |
| `WithDashboard(Action<IResourceBuilder<DockerComposeAspireDashboardResource>>)` | Configure the Dashboard container |

## Pipeline Steps

When the pipeline runs, `DockerComposeEnvironmentResource` injects four steps:

1. **`publish-<env>`** — Writes the `docker-compose.yaml` file
2. **`prepare-<env>`** — Writes the `.env.<environment>` file with captured variables
3. **`docker-compose-up-<env>`** — Runs `docker compose up -d --remove-orphans`
4. **`docker-compose-down-<env>`** — Runs `docker compose down` for cleanup

These steps form a dependency chain: `prepare` depends on `publish` and all `build` steps; `up` depends on `prepare` and all `push` steps; summary/print steps depend on `up`.

## The Dashboard Service

When `DashboardEnabled` is true, the compose file includes a `DockerComposeAspireDashboardResource` container with two endpoints: `PrimaryEndpoint` (the web UI) and `OtlpGrpcEndpoint` (for receiving telemetry). All other services are configured to send their OTLP data to this container.
