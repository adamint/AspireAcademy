# Compose File Generation

When the `publish-<env>` pipeline step runs, `DockerComposePublishingContext` translates every resource in your app model into a strongly-typed `ComposeFile` object, then serializes it to YAML.

## The ComposeFile Model

`ComposeFile` is a C# representation of a Docker Compose file with typed properties:

- `Services` — dictionary of `Service` objects
- `Networks` — dictionary of `Network` objects
- `Volumes` — dictionary of `Volume` objects
- `Secrets`, `Configs`, `Extensions` — additional compose sections
- `ToYaml(string lineEndings = "\n")` — serializes to YAML string

The YAML serializer uses `UnderscoredNamingConvention` (matching Docker Compose's snake_case), skips empty collections, and applies custom quoting for special characters.

## Resource-to-Service Translation

Each compute resource becomes a `DockerComposeServiceResource`, which produces a compose `Service`. The translation sets:

- **`image`** — container image (or a placeholder for project/Dockerfile resources that get built)
- **`container_name`** — the resource name
- **`entrypoint`** — overridden entry point if specified
- **`environment`** — resolved environment variables
- **`ports`** / **`expose`** — endpoint mappings
- **`volumes`** — volume mounts
- **`depends_on`** — service dependencies derived from `WithReference` calls

## Value Resolution

The `ProcessValueAsync()` method resolves Aspire expressions into compose-compatible values:

| Aspire Concept | Compose Output |
|---------------|----------------|
| String literal | String literal |
| `EndpointReference` | URL, host, or port value |
| `ParameterResource` | Environment variable placeholder (`${VAR_NAME}`) |
| Connection string | Resolved connection string with placeholders |
| `ReferenceExpression` | Composed string with resolved parts |

Parameters become entries in the `.env` file, referenced as `${PARAMETER_NAME}` in the compose YAML.

## Networking

A default bridge network named `"aspire"` is added automatically. All services join this network, enabling inter-service DNS resolution by container name — the same way Aspire's service discovery works in development.

## Dockerfile Handling

For resources with `DockerfileBuildAnnotation` that use a `DockerfileFactory` (dynamically generated Dockerfiles), the publisher materializes the Dockerfile content to disk at `<output>/<resource>.Dockerfile` before referencing it in the compose build context.

## Environment Files

Captured environment variables — parameters, secrets, and runtime values that shouldn't be hardcoded — are written to `.env.<environment>`. The `CapturedEnvironmentVariable` class tracks each variable's `Name`, `Description`, `DefaultValue`, `Source`, and associated `Resource`.
