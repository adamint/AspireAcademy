# Well-Known Steps and Tags

The pipeline seeds itself with a set of **well-known steps** defined in `WellKnownPipelineSteps`. These constants form the skeleton of every publish or deploy operation — resource-specific steps attach to them via `DependsOn` and `RequiredBy`.

## WellKnownPipelineSteps

| Constant | Value | Purpose |
|----------|-------|---------|
| `Publish` | `"publish"` | Writes deployment artifacts (manifests, compose files, Helm charts) |
| `PublishPrereq` | `"publish-prereq"` | Pre-publish validation and preparation |
| `Deploy` | `"deploy"` | Executes the actual deployment (e.g., `docker compose up`) |
| `DeployPrereq` | `"deploy-prereq"` | Deployment-state cache, deploy tag assignment |
| `Build` | `"build"` | Container image build barrier — all build steps complete before this |
| `BuildPrereq` | `"build-prereq"` | Pre-build validation |
| `Push` | `"push"` | Container image push barrier — all pushes complete before this |
| `PushPrereq` | `"push-prereq"` | Registry validation, archive-vs-registry destination checks |
| `ProcessParameters` | `"process-parameters"` | Runs `ParameterProcessor.InitializeParametersAsync` to resolve parameter values |
| `Diagnostics` | `"diagnostics"` | Emits the resolved pipeline graph for debugging |

## WellKnownPipelineTags

Tags group steps by category rather than specific ordering:

| Constant | Value | Usage |
|----------|-------|-------|
| `ProvisionInfrastructure` | `"provision-infra"` | Steps that provision cloud infrastructure |
| `BuildCompute` | `"build-compute"` | Per-resource container image builds |
| `PushContainerImage` | `"push-container-image"` | Per-resource image pushes |
| `DeployCompute` | `"deploy-compute"` | Per-resource deployment steps |

## How Steps Wire Together

When you call `builder.AddProject<Projects.Api>("api")`, the `ProjectResource` constructor registers a `PipelineStepAnnotation` that creates:

- `build-api` — tagged `build-compute`, depends on `build-prereq`, required by `build`
- `push-api` — tagged `push-container-image`, required by `push`

The `deploy-prereq` step handles deployment-state caching (so repeated deploys skip unchanged resources) and assigns a generated deploy tag to build resources. The `push-prereq` step validates that a container registry is available when the destination is `Registry` rather than `Archive`.

This two-layer system — named steps for ordering and tags for grouping — lets environment resources like Docker Compose and Kubernetes query "give me all steps tagged `build-compute`" and wire their own publish steps to depend on the results.
