# The Pipeline Architecture

When you run `aspire publish` or `aspire deploy`, Aspire doesn't just dump files to disk — it executes a **pipeline**, a directed acyclic graph (DAG) of steps that build, push, and deploy your application. The engine lives in `Aspire.Hosting.Pipelines` and is orchestrated by `DistributedApplicationPipeline`.

## PipelineStep — The Building Block

Every unit of work is a `PipelineStep`. Each step has a `Name`, an `Action` (a `Func<PipelineStepContext, Task>`), and dependency edges:

```csharp
var step = new PipelineStep
{
    Name = "build-api",
    Description = "Build the API container image",
    Action = async ctx =>
    {
        var manager = ctx.Services.GetRequiredService<IResourceContainerImageManager>();
        await manager.BuildImageAsync(ctx.PipelineContext.Model.Resources.First());
    },
    DependsOnSteps = ["build-prereq"],
    Tags = ["build-compute"]
};
```

`DependsOnSteps` declares which steps must complete first. `RequiredBySteps` declares which steps depend on *this* step — at execution time, `RequiredBy` is normalized into `DependsOn` so the DAG flows in one direction.

## PipelineContext — The Shared State

`PipelineContext` is the environment shared across all steps. It carries the `DistributedApplicationModel` (all registered resources), the `ExecutionContext` (publish vs. run mode), the DI `IServiceProvider`, a `Logger`, and a `CancellationToken`. Each step receives a `PipelineStepContext` wrapper that adds a `ReportingStep` for progress tracking and a `Summary` for collecting output key-value pairs.

## DAG Execution

`DistributedApplicationPipeline.ExecuteAsync()` follows this flow:

1. **Collect** step annotations from every resource (`PipelineStepAnnotation`)
2. **Run** configuration callbacks (`PipelineConfigurationAnnotation`)
3. **Validate** for duplicate step names
4. **Normalize** `RequiredBy` into `DependsOn`
5. **Filter** steps by `PipelineOptions.Step` (e.g., only `publish` or `deploy`)
6. **Execute** as a dependency-ordered DAG — steps with satisfied dependencies run concurrently

This architecture means adding a resource like `AddPostgres("pg")` automatically injects build, push, and publish steps into the graph. The pipeline is fully extensible — your code can insert custom steps at any point in the DAG.
