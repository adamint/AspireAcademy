# Build().Run() and the Dashboard

You've assembled a resource graph with `CreateBuilder`, `AddProject`, and `WithReference`. Now it's time to start everything. That's what `builder.Build().Run()` does — but there's a lot happening under the hood.

## Build(): Freezing the Resource Graph

When you call `builder.Build()`, Aspire:

1. **Validates the resource graph** — checks for duplicate names, circular dependencies, and invalid configurations.
2. **Creates a `DistributedApplication`** — this is the runtime object that manages the lifecycle of all your resources. It's similar to how `WebApplication.CreateBuilder().Build()` creates a `WebApplication` in ASP.NET Core.
3. **Registers internal services** — the DCP client, health monitors, resource notification handlers, and the OTLP telemetry collector.

After `Build()`, the resource graph is immutable. You can't add more resources.

## Run(): Starting the World

`Run()` kicks off the orchestration sequence:

1. **DCP starts** — The **Developer Control Plane** (DCP) is Aspire's runtime engine. It's a process that manages the lifecycle of all resources — starting containers, launching .NET projects, assigning ports, and monitoring health.

2. **Resources are started** — DCP starts each resource in dependency order. If your web frontend depends on the API, the API starts first. Each .NET project is launched as a separate process with `dotnet run`, and each container is started via the container runtime (Docker or Podman).

3. **Environment variables are injected** — Aspire injects configuration into each service process. Connection strings, service URLs, and OTLP endpoints are all passed as environment variables. Your service code reads them through the standard `IConfiguration` system.

4. **The Dashboard launches** — Aspire starts a web application (the Dashboard) that displays the state of all resources. It's typically available at `https://localhost:17178` (the exact port is shown in the console output).

5. **Health monitoring begins** — DCP continuously monitors each resource's health status by polling the `/health` endpoint that ServiceDefaults configures. The Dashboard shows real-time state transitions: **Starting** → **Running** → and potentially **Exited** or **FailedToStart**.

## The Console Output

When you run your AppHost, you'll see output like:

```
info: Aspire.Hosting.DistributedApplication[0]
      Aspire version: 9.2.1
info: Aspire.Hosting.DistributedApplication[0]
      Distributed application starting.
info: Aspire.Hosting.DistributedApplication[0]
      Dashboard is available at: https://localhost:17178/login?t=abc123
```

The Dashboard URL includes a one-time login token for security. Click it, and you're inside the Dashboard — the topic of our next lesson.

## When Run() Blocks

`Run()` is a blocking call — it keeps the AppHost process alive until you press `Ctrl+C` or the process is terminated. When you stop it, DCP gracefully shuts down all managed resources.
