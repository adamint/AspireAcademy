# Reading Distributed Traces

How to follow a request across multiple services using the trace waterfall view.

## What a Trace Represents

A distributed trace tracks a single operation from start to finish across all services it touches. Every trace has:

- A **Trace ID** — a unique identifier shared by all spans in the trace
- Multiple **spans** — each representing a unit of work in a specific service
- A **parent-child relationship** — forming a tree (or waterfall) of operations

## The Waterfall View

In the Dashboard's Traces tab, click any trace to see its waterfall:

```
webfrontend  GET /weather           ██████████████████  45ms
  api        GET /api/weather       ████████████        30ms
    redis    GET cache:weather      ████                 8ms
    api      DB: SELECT forecasts   ██████              15ms
```

Each bar represents a span. The width shows duration. Indentation shows parent-child relationships.

## Reading Span Details

Click any span to see:

| Field | Example |
|-------|---------|
| **Service** | api |
| **Operation** | GET /api/weather |
| **Duration** | 30ms |
| **Status** | OK (200) or Error (500) |
| **Tags** | `http.method=GET`, `http.url=/api/weather` |
| **Events** | Exception details if an error occurred |

## Common Patterns to Look For

### Slow Database Query
```
api  GET /api/products    ████████████████████  200ms
  db SELECT * FROM ...    ████████████████████  195ms  ← bottleneck
```

### N+1 Query Problem
```
api  GET /api/orders      ████████████████████████  500ms
  db SELECT order         ██                         10ms
  db SELECT items         ██                         10ms
  db SELECT items         ██                         10ms
  ... (repeated 50 times)
```

### Failed Downstream Call
```
web  GET /checkout         ████████████████████  error
  api  POST /api/order     ████████  error (500)
    payment  charge        ████  error (timeout)  ← root cause
```

## Trace Context Propagation

Traces work because of **context propagation**. When Service A calls Service B via `HttpClient`, .NET automatically adds a `traceparent` HTTP header:

```
traceparent: 00-abc123-def456-01
```

Service B reads this header and creates a child span under the same trace. This happens automatically — no code needed when using `AddServiceDefaults()`.

## Filtering Traces

In the Dashboard, you can filter by:
- **Resource** — show only traces involving the API
- **Duration** — find slow requests
- **Status** — find errors
