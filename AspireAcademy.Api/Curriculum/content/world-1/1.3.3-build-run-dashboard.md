# Build, Run, and the Dashboard

When you run `aspire run`, a sequence of events unfolds that transforms your declarative AppHost into running services with full observability. Understanding this sequence helps you diagnose problems when things go wrong.

---

## What Happens When You Run

```bash
aspire run
```

## The Dashboard

The Dashboard is a web UI that opens automatically at `https://localhost:PORT`. It has five views:

### Resources
A table of every resource in your application. Each row shows:
- **Name** — the resource name you declared in your AppHost
- **State** — Starting, Running, Exited, FailedToStart
- **Type** — Project, Container, Executable
- **Endpoints** — clickable URLs for HTTP endpoints

### Console Logs
Real-time stdout/stderr from every resource. Select a resource from the dropdown to filter.

### Structured Logs
Queryable structured log entries with severity, category, and custom properties. Unlike console logs, structured logs have fields you can filter on.

### Traces
Distributed traces displayed as a waterfall timeline. Click a trace to see every span — which service handled which part of the request, and how long each step took.

### Metrics
Counters, histograms, and gauges from your services. View request rates, error rates, response times, and custom metrics.

## Stopping

Press `Ctrl+C` in the terminal where `aspire run` is executing. DCP tears down all session-scoped containers and stops all processes.

> 💡 **Persistent containers** (configured with `.WithLifetime(ContainerLifetime.Persistent)`) survive `Ctrl+C`. This is useful for databases — you don't want to wipe your test data every time you restart. Session-scoped containers (the default) are cleaned up.
