# HTTP Resilience with Polly

In a distributed system, HTTP calls between services can fail: the target service might be restarting, the network might blip, or the service might be overloaded. **Resilience** is the practice of handling these failures gracefully so they don't cascade into outages.

## What ServiceDefaults Configures

ServiceDefaults adds resilience to every `HttpClient` in your application with two lines:

```csharp
builder.Services.ConfigureHttpClientDefaults(http =>
{
    http.AddStandardResilienceHandler();
    http.AddServiceDiscovery();
});
```

`ConfigureHttpClientDefaults` applies configuration to **all** `HttpClient` instances created via the DI container. This means you don't need to configure resilience per client — it's a global default.

## AddStandardResilienceHandler()

This method comes from `Microsoft.Extensions.Http.Resilience` and adds a comprehensive resilience pipeline powered by **Polly** (the most popular .NET resilience library). The standard handler includes:

### 1. Retry (Exponential Backoff)

If a request fails with a transient error (HTTP 500, 502, 503, or a timeout), the handler retries the request. Each retry waits longer than the last:

- 1st retry: ~2 seconds
- 2nd retry: ~4 seconds
- 3rd retry: ~8 seconds

This is called **exponential backoff** — it gives the target service time to recover without hammering it with immediate retries.

### 2. Circuit Breaker

If a service keeps failing (e.g., 10 failures in 30 seconds), the circuit breaker **opens** — it stops sending requests to the failing service and returns errors immediately. After a cool-down period, it lets a few test requests through to see if the service recovered. This prevents a failing service from consuming resources on the caller.

### 3. Timeout

Each individual request attempt has a timeout (default: 30 seconds). If a response doesn't arrive in time, the request is cancelled. There's also a **total request timeout** that caps how long all retries together can take.

## The Full Pipeline

The standard resilience handler applies these strategies in order:

```
Request → Total Timeout → Retry → Circuit Breaker → Attempt Timeout → Send HTTP
```

Each layer wraps the next. If the attempt times out, the retry decides whether to try again. If too many retries fail, the circuit breaker opens. If everything exceeds the total timeout, the request fails.

## Customizing the Pipeline

You can tune the defaults by configuring options:

```csharp
http.AddStandardResilienceHandler(options =>
{
    options.Retry.MaxRetryAttempts = 5;
    options.CircuitBreaker.SamplingDuration = TimeSpan.FromSeconds(60);
    options.AttemptTimeout.Timeout = TimeSpan.FromSeconds(10);
});
```

For most applications, the defaults are well-tuned. But knowing they exist — and that they're active by default in every Aspire service — is crucial for understanding your app's behavior under failure conditions.
