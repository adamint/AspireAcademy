# Aspire vs Docker Compose and Kubernetes

What makes Aspire different from other orchestration tools and when to use each.

## Docker Compose: Declarative YAML

Docker Compose defines services in a `docker-compose.yml` file. Each service maps to a container image, ports, and environment variables. It's language-agnostic and widely adopted, but it has limits for .NET developers:

- **No compile-time safety** — a typo in a service name or port is only caught at runtime.
- **No service discovery** — you wire hostnames manually in environment variables.
- **No observability** — you add tracing, logging, and health checks yourself.
- **Everything is a container** — running a .NET project means building and publishing a Docker image first, slowing your inner loop.

## Kubernetes: Production Orchestration

Kubernetes manages containers at scale with deployments, services, ingresses, and config maps. It's the gold standard for production, but it's heavyweight for local development:

- **Steep learning curve** — even a two-service app needs dozens of YAML lines across multiple files.
- **Not designed for dev** — tools like Minikube or Kind simulate a cluster locally, but the feedback loop is slow.
- **Separate concerns** — you define _what to run_ in one place and _how to connect_ in another.

## Aspire: Code-First Dev-Time Orchestration

Aspire takes a different approach. Your AppHost is C# code:

```csharp
var builder = DistributedApplication.CreateBuilder(args);
var redis = builder.AddRedis("cache");
builder.AddProject<Projects.Api>("api").WithReference(redis);
builder.Build().Run();
```

**Key differences:**

- **Compile-time safety** — `AddProject<T>()` uses generated types; a broken reference is a build error.
- **Automatic service discovery** — `WithReference()` injects connection strings and URLs as environment variables.
- **Built-in observability** — OpenTelemetry tracing, structured logging, metrics, and a Dashboard come free.
- **Mixed resources** — run .NET projects directly (no container build), alongside containers, executables, and cloud resources.

## When to Use Each

| Scenario | Best Tool |
|----------|-----------|
| Local development of a .NET distributed app | **Aspire** |
| Running a stack of pre-built container images | **Docker Compose** |
| Production deployment at scale | **Kubernetes** |
| CI/CD pipeline for integration tests | **Aspire** or **Compose** |

Aspire isn't a replacement for Kubernetes in production. It's a **dev-time orchestrator** that makes building and debugging distributed apps fast. You can even generate Docker Compose or Kubernetes manifests from your Aspire app model for deployment.
