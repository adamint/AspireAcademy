# The Five Pillars of Aspire

Aspire is often described as "a dev-time orchestrator." That was true once. Today, Aspire is a **full-lifecycle platform for building distributed applications** — and it works with any language your services are written in.

> 🎯 **By the end of this lesson, you will be able to:**
> - List the five pillars of Aspire and explain what each one provides
> - Distinguish between the app model (declarative graph) and runtime orchestration
> - Recognize that AppHosts can be written in C# or TypeScript

Aspire rests on five pillars:

---

## 1. The App Model

> 📝 **App model** — a declarative graph of your application's resources and their relationships, defined in an **AppHost** file.

You define what services, databases, caches, and message brokers your application needs, and how they connect. This is the fundamental shift: instead of writing shell scripts that glue things together, you describe *what you want* and let Aspire figure out *how to make it happen*.

Here is the same Bazaar AppHost in two languages:

**C#**
```csharp
var builder = DistributedApplication.CreateBuilder(args);
var postgres = builder.AddPostgres("pg").AddDatabase("bazaar-db");
var redis = builder.AddRedis("cache");
var api = builder.AddProject<Projects.BazaarApi>("api")
    .WithReference(postgres).WithReference(redis);
var frontend = builder.AddViteApp("frontend", "../bazaar-web")
    .WithReference(api).WaitFor(api);
builder.Build().Run();
```

**TypeScript**
```typescript
import { createBuilder } from './.modules/aspire.js';
const builder = await createBuilder();
const postgres = await (await builder.addPostgres("pg")).addDatabase("bazaar-db");
const redis = await builder.addRedis("cache");
const api = await builder.addProject("api", "../Bazaar.Api");
await api.withReference(postgres);
await api.withReference(redis);
const frontend = await builder.addViteApp("frontend", "../bazaar-web");
await frontend.withReference(api);
await frontend.waitFor(api);
await builder.build().run();
```

> 💡 Same concepts, different syntax. The app model works the same way in both languages — pick whichever your team prefers.

## 2. Integrations

> 📝 **Integrations** — NuGet/npm/PyPI packages that add support for specific infrastructure (databases, caches, message brokers, AI services, etc.)

`Aspire.Hosting.Redis` adds `AddRedis()`. `Aspire.Hosting.PostgreSQL` adds `AddPostgres()`. There are 50+ hosting integrations.

Each integration handles the boilerplate: container images, ports, health checks, connection strings, and environment variable injection. You write one line — `AddRedis("cache")` — and Aspire knows how to start Redis, health-check it, and inject the connection string into every service that references it.

## 3. Tooling

Aspire provides:

| Tool | What it does |
|---|---|
| **CLI** | `aspire run`, `aspire publish`, `aspire deploy` — 40+ commands |
| **VS Code extension** | Code lenses, tree views, debugger integration |
| **MCP server** | Lets AI agents interact with your running application |

## 4. Observability

> 📝 **Aspire Dashboard** — a web-based observability UI that displays resources, console logs, structured logs, distributed traces, and metrics in real time.

The Dashboard consumes OpenTelemetry Protocol (OTLP) data, so any service that emits OTLP (which includes .NET, Python, Go, Java, and TypeScript via OpenTelemetry SDKs) appears automatically.

## 5. Deployment

Aspire is not just for development. When you are ready to ship, `aspire publish` generates deployment artifacts directly from your app model:

| Target | Output |
|---|---|
| Docker Compose | `docker-compose.yaml` |
| Kubernetes | Helm chart |
| Azure | Bicep templates |

> 💡 The same AppHost that runs locally produces production infrastructure. No separate IaC to maintain.

---

## What Aspire Is Not

- **Not a runtime.** Your services run as normal processes or containers. Aspire orchestrates their startup and wiring, but it does not host your code.
- **Not a replacement for Kubernetes.** Aspire can publish *to* Kubernetes, but it does not replace it in production. DCP (the Developer Control Plane) runs locally; K8s runs in production.
- **Not .NET-only.** You can write AppHosts in C# or TypeScript. You can orchestrate services in any language.

---

> 📋 **Key Takeaways**
> - Aspire's five pillars are: App Model, Integrations, Tooling, Observability, and Deployment
> - The app model is a declarative graph — you describe *what* your app needs, not *how* to wire it
> - Integrations eliminate infrastructure boilerplate with one-line declarations
> - The same AppHost powers local dev, CI, and production deployment
