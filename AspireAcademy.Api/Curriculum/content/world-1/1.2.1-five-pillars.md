# The Five Pillars of Aspire

Aspire is often described as "a dev-time orchestrator." That was true once. Today, Aspire is a **full-lifecycle platform for building distributed applications** — and it works with any language your services are written in.

Aspire rests on five pillars:

---

## 1. The App Model

The **app model** is a declarative graph of your application's resources and their relationships. You define it in an **AppHost** — a file that describes what services, databases, caches, and message brokers your application needs, and how they connect.

Here is the same Atlas AppHost in three languages:

**C#**
```csharp
var builder = DistributedApplication.CreateBuilder(args);
var postgres = builder.AddPostgres("pg").AddDatabase("atlas-db");
var redis = builder.AddRedis("cache");
var api = builder.AddProject<Projects.AtlasApi>("api")
    .WithReference(postgres).WithReference(redis);
var frontend = builder.AddViteApp("frontend", "../atlas-web")
    .WithReference(api).WaitFor(api);
builder.Build().Run();
```

**TypeScript**
```typescript
import { createBuilder } from './.modules/aspire.js';
const builder = await createBuilder();
const postgres = await (await builder.addPostgres("pg")).addDatabase("atlas-db");
const redis = await builder.addRedis("cache");
const api = await builder.addProject("api", "../Atlas.Api");
await api.withReference(postgres);
await api.withReference(redis);
const frontend = await builder.addViteApp("frontend", "../atlas-web");
await frontend.withReference(api);
await frontend.waitFor(api);
await builder.build().run();
```

**Python**
```python
from aspire_app import create_builder
with create_builder() as builder:
    postgres = builder.add_postgres("pg").add_database("atlas-db")
    redis = builder.add_redis("cache")
    api = builder.add_project("api", "../Atlas.Api")
    api.with_reference(postgres).with_reference(redis)
    frontend = builder.add_vite_app("frontend", "../atlas-web")
    frontend.with_reference(api).wait_for(api)
    builder.run()
```

Same concepts, different syntax. The app model is language-agnostic.

## 2. Integrations

**Integrations** are NuGet packages (for hosting) and client libraries (for services) that add support for specific infrastructure. `Aspire.Hosting.Redis` adds `AddRedis()`. `Aspire.Hosting.PostgreSQL` adds `AddPostgres()`. There are 50+ hosting integrations covering databases, caches, message brokers, AI services, identity providers, and more.

Each integration handles the boilerplate: container images, ports, health checks, connection strings, and environment variable injection. You write one line — `AddRedis("cache")` — and Aspire knows how to start Redis, health-check it, and inject the connection string into every service that references it.

## 3. Tooling

Aspire provides a **CLI** (`aspire run`, `aspire publish`, `aspire deploy`, 40+ commands), a **VS Code extension** (code lenses, tree views, debugger integration), and an **MCP server** that lets AI agents interact with your running application.

## 4. Observability

The **Aspire Dashboard** is a web-based observability UI that shows your application's resources, console logs, structured logs, distributed traces, and metrics — all in real time. It consumes OpenTelemetry Protocol (OTLP) data, so any service that emits OTLP (which includes .NET, Python, Go, Java, and TypeScript via OpenTelemetry SDKs) appears automatically.

## 5. Deployment

Aspire is not just for development. When you are ready to ship, `aspire publish` generates deployment artifacts — Docker Compose files, Kubernetes Helm charts, or Azure Bicep templates — directly from your app model. The same AppHost that runs locally produces production infrastructure.

---

## What Aspire Is Not

- **Not a runtime.** Your services run as normal processes or containers. Aspire orchestrates their startup and wiring, but it does not host your code.
- **Not a replacement for Kubernetes.** Aspire can publish *to* Kubernetes, but it does not replace it in production. DCP (the Developer Control Plane) runs locally; K8s runs in production.
- **Not .NET-only.** You can write AppHosts in C#, TypeScript, Python, Go, Java, or Rust. You can orchestrate services in any language.
