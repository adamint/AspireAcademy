# Exploring the Aspire Dashboard

The Aspire Dashboard is your command center for understanding what's happening across your distributed app. It launches automatically when you run the AppHost and provides five key views.

## Resources Page

The **Resources** page is the landing page. It shows every resource in your app — projects, containers, executables — with:

- **State**: Current lifecycle state (Starting, Running, Exited, FailedToStart)
- **Health**: Whether the resource's health checks are passing (Healthy, Degraded, Unhealthy)
- **Endpoints**: Clickable URLs for each exposed endpoint
- **Environment Variables**: The configuration injected by the AppHost (expand a row to see these)
- **Source**: The project path or container image
- **Commands**: Start, Stop, and Restart buttons for each resource

This page is where you quickly verify that everything is running. If a resource shows **FailedToStart**, expand it to see the error details and check its console logs.

## Console Logs Page

The **Console Logs** page shows raw standard output and standard error from each resource. Select a resource from the dropdown to view its output.

- Project logs are color-coded by severity (green for information, red for errors)
- Container logs show the raw container output
- You can download logs to your local machine for offline analysis

This is equivalent to running `docker logs` or watching a terminal window, but for every resource in one place.

## Structured Logs Page

**Structured Logs** go beyond raw text. Aspire configures OpenTelemetry to capture **structured log entries** — log messages with typed properties that can be filtered and searched.

Each entry shows:
- **Resource**: Which service emitted the log
- **Level**: Information, Warning, Error, Critical
- **Timestamp**: When the log was recorded
- **Message**: The formatted log message
- **Trace**: A link to the associated distributed trace (if the log was emitted during a traced request)

You can filter by log level, resource, or search for specific text. The filter dialog lets you build complex queries on any log property.

## Traces Page

The **Traces** page shows **distributed traces** — end-to-end views of requests as they flow across services. Each trace consists of **spans** (individual units of work, like an HTTP call or a database query).

Key columns:
- **Name**: The HTTP method and route (e.g., `GET /weatherforecast`)
- **Spans**: Icons showing which resources participated
- **Duration**: How long the request took, with a visual bar for comparison

Click a trace to see the **waterfall view** — a timeline showing each span's start time and duration. This is how you diagnose latency: you can see exactly which service or database call is the bottleneck.

If a span has an error, it's marked with a red error icon. Select it to see the exception details, and use the **View Logs** button to jump directly to the structured logs for that trace.

## Metrics Page

The **Metrics** page displays time-series charts for OpenTelemetry metrics. Select a resource, then browse its available **meters** (groups of related metrics) and **instruments** (individual metrics).

Common built-in metrics include:
- `http.server.request.duration` — how long HTTP requests take
- `http.client.request.duration` — how long outgoing HTTP calls take
- Runtime metrics — GC heap size, thread pool queue length, etc.

You can switch between chart and table views, filter by dimensions (like HTTP method or status code), and compare durations visually.

## Putting It Together

The Dashboard's real power is how these views connect. From a slow trace, you can jump to the structured logs that were emitted during that request. From an error log, you can navigate to the trace that triggered it. From the metrics page, you can spot anomalies and then drill into individual traces to understand the cause.

This interconnected observability — with zero configuration beyond `AddServiceDefaults()` — is what makes Aspire's Dashboard uniquely valuable for .NET developers.
