# Creating a Project with `aspire new`

*Time to bring Constellation to life. In this lesson, you'll install the Aspire CLI, scaffold a project, and see what gets generated — in your choice of language.*

---

The fastest way to start an Aspire application is with the **Aspire CLI**. The `aspire new` command scaffolds a complete solution with all the project types wired together.

## Installing the Aspire CLI

:::tabs
::tab{label="macOS / Linux"}
```bash
curl -sSL https://aspire.dev/install.sh | bash
```
::tab{label="Windows (PowerShell)"}
```powershell
powershell -ExecutionPolicy ByPass -NoProfile -Command "iex ((New-Object System.Net.WebClient).DownloadString('https://aspire.dev/install.ps1'))"
```
:::

The CLI is a standalone tool — it bundles everything it needs, so you don't need to install the .NET SDK separately just to run an AppHost (though you'll need it for .NET service projects).

## Scaffolding Constellation

```bash
aspire new aspire-starter --name Constellation
```

The CLI launches an interactive wizard where you choose your **AppHost language**. Aspire currently supports:

| Language | AppHost file | Status |
|----------|-------------|--------|
| **C#** | `Program.cs` or `apphost.cs` | Stable |
| **TypeScript (Node.js)** | `apphost.ts` | Stable |
| **Python** | `apphost.py` | Experimental |
| **Java** | `AppHost.java` | Experimental |
| **Go** | `main.go` | Experimental |

All languages produce the **same app model** at runtime. The difference is just which language you write your topology in.

## What Gets Scaffolded

The generated structure depends on your AppHost language. Here's what you'll see:

:::tabs
::tab{label="C# AppHost"}
```
Constellation/
├── Constellation.sln
├── Constellation.AppHost/
│   ├── Program.cs                    ← Resource graph definition
│   ├── Constellation.AppHost.csproj  ← References Aspire.Hosting.AppHost
│   └── Properties/
│       └── launchSettings.json
├── Constellation.ServiceDefaults/
│   ├── Extensions.cs                 ← AddServiceDefaults(), OTel, health checks
│   └── Constellation.ServiceDefaults.csproj
├── Constellation.ApiService/
│   ├── Program.cs                    ← API with weather endpoint
│   └── Constellation.ApiService.csproj
└── Constellation.Web/
    ├── Program.cs                    ← Blazor frontend
    └── Constellation.Web.csproj
```
::tab{label="TypeScript AppHost"}
```
Constellation/
├── apphost.ts                        ← Resource graph definition
├── package.json                      ← Dependencies (tsx, nodemon)
├── tsconfig.json
├── .modules/                         ← Generated Aspire SDK (auto-managed)
│   ├── aspire.ts
│   ├── base.ts
│   └── transport.ts
├── Constellation.ApiService/
│   └── ...                           ← .NET API project
└── Constellation.Web/
    └── ...                           ← Web frontend
```
::tab{label="Python AppHost"}
```
Constellation/
├── apphost.py                        ← Resource graph definition
├── .modules/                         ← Generated Aspire SDK (auto-managed)
│   ├── aspire.py
│   └── ...
├── Constellation.ApiService/
│   └── ...                           ← .NET API project
└── Constellation.Web/
    └── ...                           ← Web frontend
```
:::

> **What is `.modules/`?** For non-C# AppHosts, the Aspire CLI generates a local SDK in the `.modules/` directory. This SDK provides the `createBuilder()`, `addProject()`, `withReference()`, and other functions in your language. It's auto-generated from the Aspire hosting packages you reference — you don't edit it directly.

## The `aspire.config.json` File

When you use the Aspire CLI, an `aspire.config.json` file is created at the project root. This tells the CLI which file is the AppHost:

```json
{
  "appHost": {
    "path": "Constellation.AppHost/Constellation.AppHost.csproj"
  }
}
```

For TypeScript AppHosts, this points to `apphost.ts` instead. This allows you to run `aspire run` from the project root without specifying the path.

## The Generated AppHost

Here's the starter AppHost that the template generates:

:::tabs
::tab{label="C#"}
```csharp
var builder = DistributedApplication.CreateBuilder(args);

var api = builder.AddProject<Projects.Constellation_ApiService>("api");

builder.AddProject<Projects.Constellation_Web>("webfrontend")
       .WithExternalHttpEndpoints()
       .WithReference(api);

builder.Build().Run();
```
::tab{label="TypeScript"}
```typescript
import { createBuilder } from "./.modules/aspire.js";

const builder = createBuilder();

const api = builder.addProject("api", "../Constellation.ApiService");

builder.addProject("webfrontend", "../Constellation.Web")
       .withExternalHttpEndpoints()
       .withReference(api);

await builder.buildAndRun();
```
::tab{label="Python"}
```python
import aspire

builder = aspire.create_builder()

api = builder.add_project("api", "../Constellation.ApiService")

builder.add_project("webfrontend", "../Constellation.Web") \
       .with_external_http_endpoints() \
       .with_reference(api)

builder.build_and_run()
```
:::

This is the starting point you'll build on throughout this course. Every `addProject`, `withReference`, and `build().run()` will become second nature.
