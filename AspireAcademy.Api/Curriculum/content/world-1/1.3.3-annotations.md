# Annotations: Behavior Without Inheritance

Annotations are Aspire's mechanism for attaching capabilities to resources without requiring deep class hierarchies. Instead of encoding every feature into the resource class itself, Aspire uses the **composition over inheritance** pattern.

## The IResourceAnnotation Interface

```csharp
public interface IResourceAnnotation
{
}
```

It's an empty **marker interface** — a design pattern where the interface exists solely to tag types as belonging to a category. Any class that implements `IResourceAnnotation` can be added to a resource's `Annotations` collection.

## How Annotations Work

When you call an extension method like `WithEnvironment("KEY", "value")`, it doesn't modify the resource class. Instead, it adds an `EnvironmentCallbackAnnotation` to the resource's `Annotations` collection. Later, when DCP starts the resource, it reads all annotations and uses them to configure the process or container.

```csharp
// This call:
builder.AddContainer("redis", "redis").WithEnvironment("REDIS_PASSWORD", "secret");

// Internally adds an EnvironmentCallbackAnnotation to the container's Annotations collection.
// DCP reads it at startup and injects REDIS_PASSWORD=secret into the container.
```

## Key Annotation Types

### EndpointAnnotation

Defines a network endpoint (port, protocol, scheme) for a resource.

```csharp
public sealed class EndpointAnnotation : IResourceAnnotation
{
    public string Name { get; set; }
    public int? Port { get; set; }
    public int? TargetPort { get; set; }
    public string UriScheme { get; set; }    // "http", "https", "tcp"
    public ProtocolType Protocol { get; set; } // TCP, UDP
    public bool IsExternal { get; set; }
}
```

Added by methods like `WithHttpEndpoint()` and `WithEndpoint()`.

### ContainerImageAnnotation

Specifies the Docker image for a container resource.

```csharp
public sealed class ContainerImageAnnotation : IResourceAnnotation
{
    public string? Registry { get; set; }  // e.g., "docker.io"
    public required string Image { get; set; }  // e.g., "redis"
    public string? Tag { get; set; }       // e.g., "7.4"
}
```

Automatically added when you call `AddContainer("name", "redis:7.4")` or `WithImage()`.

### EnvironmentCallbackAnnotation

Registers a callback that provides environment variables at startup time.

```csharp
public class EnvironmentCallbackAnnotation : IResourceAnnotation
{
    public Func<EnvironmentCallbackContext, Task> Callback { get; }
}
```

Added by `WithEnvironment()`. The callback is invoked when DCP starts the resource.

### ContainerMountAnnotation

Defines a volume or bind mount for a container.

```csharp
public sealed class ContainerMountAnnotation : IResourceAnnotation
{
    public string? Source { get; }     // volume name or host path
    public string Target { get; }      // path inside container
    public ContainerMountType Type { get; }  // Volume or BindMount
    public bool IsReadOnly { get; }
}
```

Added by `WithVolume()` or `WithBindMount()`.

### HealthCheckAnnotation

Associates a health check registration key with the resource. DCP uses this to monitor the resource's health status.

```csharp
public class HealthCheckAnnotation(string key) : IResourceAnnotation
{
    public string Key => key;
}
```

## Why Annotations Over Inheritance?

If every capability required a subclass, you'd end up with an explosion of types: `ContainerWithVolumesAndEndpointsResource`, `ProjectWithHealthChecksResource`, etc. Annotations let you mix and match capabilities freely. A container can have endpoints, volumes, environment variables, and health checks — all by composing annotations, not by inheriting from increasingly specific base classes.
