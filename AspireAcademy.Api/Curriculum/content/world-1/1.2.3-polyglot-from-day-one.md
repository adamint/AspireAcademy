# Polyglot from Day One

Aspire's polyglot story works at two levels: the **services** you orchestrate and the **AppHost** you write them in.

---

## Polyglot Services

Aspire can orchestrate services written in any language:

| Language | Method | Example |
|---|---|---|
| C# / .NET | `AddProject<T>()` | ASP.NET API, Blazor app, gRPC service |
| TypeScript / JavaScript | `AddNodeApp()`, `AddViteApp()` | Express.js API, React/Vue/Svelte frontend |
| Python | `AddPythonApp()`, `AddUvicornApp()` | FastAPI, Flask, Django, ML pipelines |
| Go | Community Toolkit `AddGoApp()` | HTTP servers, gRPC, CLI tools |
| Java | Community Toolkit `AddJavaApp()` | Spring Boot, Quarkus, Micronaut |
| Rust | Community Toolkit `AddRustApp()` | Actix-web, Axum, Rocket |
| Any Docker image | `AddContainer()` | Anything that runs in a container |

Each service receives its connection strings and endpoint URLs as environment variables — the universal configuration mechanism that every language understands.

```reveal
{"question": "Why does Aspire use environment variables to pass connection strings and URLs to services, instead of a shared config file?", "answer": "Environment variables are the one configuration mechanism that every language and runtime understands natively — C# reads them with Environment.GetEnvironmentVariable(), Python with os.environ, Go with os.Getenv(), Node with process.env. ## Polyglot AppHosts
```

You do not have to write your AppHost in C#. Aspire also supports AppHosts in **TypeScript**:

| Language | File | Entry Point |
|---|---|---|
| C# (single-file) | `apphost.cs` | `DistributedApplication.CreateBuilder(args)` |
| C# (.csproj) | `Program.cs` | Same, with project references |
| TypeScript | `apphost.ts` | `import { createBuilder } from './.modules/aspire.js'` |

Under the hood, the TypeScript AppHost uses a **RemoteHost bridge** — a JSON-RPC connection over Unix sockets that routes calls to the .NET hosting SDK. You will learn how this works in World 9. For now, just know that TypeScript gets the same capabilities as C#.

> 💡 Aspire can **orchestrate** services in any language (Python, Go, Java, Rust, etc.), but the **AppHost itself** — the file that defines your app model — must be written in C# or TypeScript.

> 🎲 **Fun Fact:** Aspire's Community Toolkit already supports Go, Java, Rust, Ollama, Meilisearch, and more — each using the same `AddXxxApp()` pattern. The hosting integration API is public, so anyone can write support for a new language or service in a few hundred lines of code.

## Bazaar Is Polyglot

The story app, Bazaar, uses this polyglot capability directly:

- **AppHost**: C# (you will also rewrite it in TypeScript later)
- **API**: C# / ASP.NET (the backend team's choice)
- **Frontend**: TypeScript / React / Vite (the UI team's choice)
- **ML Service**: Python / FastAPI (the data science team's choice)
- **Worker**: Go (high-throughput background processing)
- **Database**: PostgreSQL (via container)
- **Cache**: Redis (via container)
- **Message Queue**: RabbitMQ (via container)

This is a realistic modern team. Aspire does not force everyone into one language — it meets them where they are.
