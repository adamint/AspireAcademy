# HTTP Resilience with Polly

AddStandardResilienceHandler: retry, circuit breaker, timeout — how ConfigureHttpClientDefaults works.

## The Problem

In a distributed system, HTTP calls fail. Services restart, networks blip, dependencies get overloaded. Without resilience, one failing service cascades into a full outage.

## What AddServiceDefaults() Configures

```csharp
builder.Services.ConfigureHttpClientDefaults(http =>
{
    http.AddStandardResilienceHandler();
});
```

This applies Polly resilience policies to **every** `HttpClient` in the application — no per-client setup needed.

## The Standard Resilience Pipeline

`AddStandardResilienceHandler()` from `Microsoft.Extensions.Http.Resilience` creates a pipeline of five strategies:

### 1. Total Request Timeout
Caps the total time for the request (including retries) at 30 seconds.

### 2. Retry
Retries on transient failures with exponential backoff:
- Max 3 attempts
- Backoff: ~2s, ~4s, ~8s (with jitter)
- Retries on `HttpRequestException`, 5xx, and 408 responses

### 3. Circuit Breaker
After too many failures, the circuit "opens" and fails fast for 30 seconds instead of hammering a broken service:
- Failure threshold: 10% of requests
- Sampling duration: 30 seconds
- Break duration: 5 seconds

### 4. Attempt Timeout
Each individual attempt times out after 10 seconds (separate from the total timeout).

### 5. Rate Limiter
Limits concurrent outbound requests to prevent overwhelming downstream services.

## How It Works in Practice

```
Request → [Total Timeout: 30s]
           → [Retry: 3 attempts]
               → [Circuit Breaker]
                   → [Attempt Timeout: 10s]
                       → [Rate Limiter]
                           → HTTP call
```

If the first attempt fails with a 503, Polly waits ~2 seconds, then retries. If the second attempt also fails, it waits ~4 seconds and retries once more. If all attempts fail, the caller gets an exception.

## ConfigureHttpClientDefaults

The key to making this work globally is `ConfigureHttpClientDefaults` — it applies configuration to every `HttpClient` created by the DI container's `IHttpClientFactory`, including named and typed clients. You don't need to opt each client in.
