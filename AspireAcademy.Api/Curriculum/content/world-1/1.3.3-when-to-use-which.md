# When to Use Which Style

Both single-file and `.csproj` AppHosts produce the same result — a running Aspire application with a Dashboard, service discovery, and observability. The difference is in developer experience and tooling.

## Single-File AppHost

**Pros:**
- Zero ceremony — one file, no solution, no `.csproj`
- Fast iteration for prototypes and learning
- Easy to share as a gist or snippet

**Cons:**
- No `Projects.*` generated types (you pass project paths as strings)
- Limited IDE support — no F5 debugging in Visual Studio
- No MSBuild customization (custom targets, analyzers, etc.)
- `#:package` directives lack version ranges and conditional inclusion

**Best for:** prototyping, demos, learning, simple apps.

```csharp
#:sdk Aspire.AppHost.Sdk@9.3.0
#:package Aspire.Hosting.Redis@9.3.0

var builder = DistributedApplication.CreateBuilder(args);
var cache = builder.AddRedis("cache");
builder.AddProject("api", "./src/Api/Api.csproj")
       .WithReference(cache);
builder.Build().Run();
```

## Traditional .csproj AppHost

**Pros:**
- Full Visual Studio / VS Code support (F5, breakpoints, IntelliSense)
- `Projects.*` generated types for compile-time safety
- MSBuild integration (analyzers, custom targets, CI pipelines)
- NuGet package management with `Directory.Packages.props`
- Works with all Aspire CLI commands (`aspire run`, `aspire publish`)

**Cons:**
- More files to manage (`.csproj`, `launchSettings.json`, solution file)
- More ceremony for simple scenarios

**Best for:** production apps, team projects, CI/CD pipelines.

```csharp
var builder = DistributedApplication.CreateBuilder(args);
var cache = builder.AddRedis("cache");
builder.AddProject<Projects.Api>("api")
       .WithReference(cache);
builder.Build().Run();
```

## Decision Matrix

| Factor | Single-File | .csproj |
|--------|-------------|---------|
| Setup time | Seconds | Minutes |
| Compile-time safety | ❌ String paths | ✅ Generated types |
| IDE debugging | Limited | Full |
| CI/CD | Basic | Full MSBuild |
| Polyglot resources | ✅ | ✅ |
| Production apps | Not recommended | ✅ |

## Migration Path

You can start with a single-file AppHost and migrate to `.csproj` later. The resource model code (`AddRedis`, `AddProject`, `WithReference`) is identical — you just move from `#:` directives to MSBuild XML and gain generated types.
