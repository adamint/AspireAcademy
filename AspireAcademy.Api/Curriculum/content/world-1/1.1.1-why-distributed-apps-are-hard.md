# Why Distributed Apps Are Hard

Imagine you're building an e-commerce app. You start with a single project — a monolith — and everything lives in one process. Database calls, business logic, and the web UI all share the same memory, the same config file, and the same deployment unit. Life is simple.

Now your team grows. You split into an **API service**, a **web frontend**, and a **background worker** for order processing. Congratulations — you've built a distributed application. And you've inherited a whole category of problems that monoliths never face.

## Service Discovery

**Service discovery** is how one service finds the network address of another. In a monolith, you call a method. In a distributed app, your frontend needs to know the API's URL — and that URL changes between your laptop, staging, and production. Hardcoding `localhost:5001` works until it doesn't.

## Configuration Management

Each service needs **connection strings** (the address and credentials for a database or message broker), ports, API keys, and feature flags. Managing these across three services in four environments (dev, CI, staging, prod) means dozens of configuration values that must stay in sync. One typo in a connection string and an entire service fails silently at startup.

## Observability Gaps

When a user reports a slow checkout, which service is the bottleneck? **Observability** — the ability to understand what's happening inside your system through logs, traces, and metrics — is trivial in a monolith (one log file, one debugger). In a distributed app, a single user request fans out across multiple services. Without **distributed tracing** (a technique that assigns a unique ID to a request and follows it across service boundaries), you're reading three separate log files and guessing.

## Resilience

What happens when the order-processing worker goes down for 30 seconds? Does the API hang? Does the frontend show an error? **Resilience** — the ability of a system to handle partial failures gracefully — requires retry logic, circuit breakers, and timeouts. Without them, one failing service cascades into a full outage.

## The Inner Loop Tax

All of these concerns add friction to your **inner loop** — the cycle of code → build → run → test that you repeat dozens of times per day. You end up maintaining Docker Compose files, shell scripts, and environment-specific configs just to run your app locally. The result: more time fighting infrastructure, less time writing features.

These problems aren't new, but for .NET developers, there hasn't been a purpose-built tool to solve them — until Aspire.
