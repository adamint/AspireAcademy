# Why Distributed Apps Are Hard

*Welcome to Aspire Academy. Over this course, you'll build a distributed application called **Constellation** — a collaborative knowledge-sharing platform with an API, a web frontend, a recommendation engine, and backing infrastructure. Each lesson adds a new capability to Constellation. By the end, you'll understand every layer of Aspire, from "hello world" to its internal source code.*

*But first — why do you need Aspire at all?*

---

Imagine you're starting to build Constellation. You begin with a single project — a monolith — where the API, web UI, and database access all live in one process. Life is simple.

Then requirements grow. You split into an **API service**, a **web frontend**, and a **recommendation engine** that runs in Python. You've built a distributed application. And you've inherited a whole category of problems that monoliths never face.

## Service Discovery

**Service discovery** is how one service finds the network address of another. In a monolith, you call a method. In Constellation, the web frontend needs to know the API's URL — and that URL changes between your laptop, staging, and production. Hardcoding `localhost:5001` works until it doesn't.

## Configuration Management

Each service needs **connection strings** (addresses and credentials for databases or message brokers), ports, API keys, and feature flags. Managing these across three services in four environments (dev, CI, staging, prod) means dozens of values that must stay in sync. One typo in a connection string and an entire service fails silently at startup.

## Observability Gaps

When a user reports that search is slow, which service is the bottleneck? **Observability** — understanding what's happening inside your system through logs, traces, and metrics — is trivial in a monolith (one log file, one debugger). In Constellation, a single search request fans out across the API, the database, and the recommendation engine. Without **distributed tracing** (assigning a unique ID to each request and following it across service boundaries), you're reading three separate log files and guessing.

## Resilience

What happens when the recommendation engine goes down for 30 seconds? Does the API hang? Does the frontend show an error? **Resilience** — handling partial failures gracefully — requires retry logic, circuit breakers, and timeouts. Without them, one failing service cascades into a full outage.

## The Inner Loop Tax

All of these concerns add friction to your **inner loop** — the cycle of code → build → run → test that you repeat dozens of times per day. You end up maintaining Docker Compose files, shell scripts, and environment-specific configs just to run your app locally. The result: more time fighting infrastructure, less time writing features.
