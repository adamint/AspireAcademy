# Aspire's Solution: Orchestration + Observability

Aspire is an opinionated stack for building distributed applications. It tackles the problems from the previous lesson with two core pillars: **orchestration** and **observability**.

## Pillar 1: Orchestration

Instead of writing YAML files or shell scripts to wire your services together, Aspire lets you define your entire application topology in C#. The **AppHost** project is where you declare every resource — services, databases, caches, message brokers — and how they connect.

Here's what a minimal AppHost looks like:

```csharp
var builder = DistributedApplication.CreateBuilder(args);

var api = builder.AddProject<Projects.ApiService>("api");

builder.AddProject<Projects.WebFrontend>("web")
       .WithReference(api);

builder.Build().Run();
```

This code does more than it looks:

- `AddProject<T>()` registers a .NET project as a **resource** (a unit of your app that Aspire manages — a service, container, or database).
- `WithReference(api)` tells Aspire that the web frontend depends on the API. Aspire automatically injects the API's URL into the frontend as configuration, handling **service discovery** for you.
- `Build().Run()` starts everything — launching each service, assigning ports, and opening the Dashboard.

Because this is C# code, you get **compile-time safety**. Rename a project and the compiler catches it. Pass the wrong type to `WithReference` and you get a red squiggle, not a runtime error in production.

## Pillar 2: Observability

Aspire ships with a **Dashboard** — a web UI that launches automatically when you run your AppHost. The Dashboard provides real-time:

- **Structured Logs**: Filterable, searchable log entries from every service
- **Distributed Traces**: A waterfall view showing how a single request flows across services, with timing for each span
- **Metrics**: Time-series charts for request rates, error rates, and latency

This works because the **ServiceDefaults** project (a shared library referenced by all your services) automatically configures **OpenTelemetry** — an open standard for collecting telemetry data — and exports it to the Dashboard via the **OTLP** (OpenTelemetry Protocol) exporter.

You don't install extra tools. You don't configure Prometheus or Jaeger for local dev. You run `dotnet run` and observability is there.

## Code-First, Not Config-First

The key insight is that Aspire is **code-first**. Your app topology is a C# program, not a collection of YAML files. This means IDE support (IntelliSense, refactoring, debugging), type safety, and the ability to use loops, conditionals, and any .NET library to define your infrastructure.
