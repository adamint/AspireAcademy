# Aspire's Answer: Topology as Code

*Constellation has the classic distributed-app problems: service discovery, configuration sprawl, and observability gaps. Let's see how Aspire solves all of them — in the language of your choice.*

---

Aspire is a stack for building distributed applications. It tackles the problems from the previous lesson with two core pillars: **orchestration** and **observability**. And it works across languages — you can write your AppHost in C#, TypeScript, Python, and more.

## Pillar 1: Orchestration — The AppHost

Instead of writing YAML files or shell scripts to wire your services together, Aspire lets you define your entire application topology **in code**. The **AppHost** is where you declare every resource — services, databases, caches, message brokers — and how they connect.

Here's what a minimal AppHost looks like for Constellation's API and web frontend:

:::tabs
::tab{label="C#"}
```csharp
var builder = DistributedApplication.CreateBuilder(args);

var api = builder.AddProject<Projects.ApiService>("api");

builder.AddProject<Projects.WebFrontend>("web")
       .WithReference(api);

builder.Build().Run();
```
::tab{label="TypeScript"}
```typescript
import { createBuilder } from "./.modules/aspire.js";

const builder = createBuilder();

const api = builder.addProject("api", "../ApiService");

builder.addProject("web", "../WebFrontend")
       .withReference(api);

await builder.buildAndRun();
```
::tab{label="Python"}
```python
import aspire

builder = aspire.create_builder()

api = builder.add_project("api", "../ApiService")

builder.add_project("web", "../WebFrontend") \
       .with_reference(api)

builder.build_and_run()
```
:::

Regardless of which language you use, this code does the same thing:

- **Adds resources** — each `addProject` call registers a service as a **resource** (a unit of your app that Aspire manages).
- **Wires them together** — `withReference(api)` tells Aspire that the web frontend depends on the API. Aspire automatically injects the API's URL into the frontend's configuration, handling **service discovery** for you.
- **Starts everything** — `build().run()` launches each service, assigns ports, configures networking, and opens the Dashboard.

> **How does polyglot work?** The AppHost language is just the "control plane" — the file that describes your topology. The services themselves can be written in any language. A TypeScript AppHost can orchestrate a .NET API, a Python ML service, and a React frontend. Under the hood, Aspire generates a local SDK (in `.modules/`) that communicates with the Aspire runtime via JSON-RPC. The runtime handles the actual orchestration regardless of your AppHost language.

## Pillar 2: Observability — The Dashboard

Aspire ships with a **Dashboard** — a web UI that launches automatically when you run your AppHost. It provides real-time:

- **Structured Logs**: Filterable, searchable log entries from every service
- **Distributed Traces**: A waterfall view showing how a single request flows across services, with timing for each span
- **Metrics**: Time-series charts for request rates, error rates, and latency

This works because Aspire automatically configures **OpenTelemetry** — an open standard for collecting telemetry data — and exports it to the Dashboard via **OTLP** (OpenTelemetry Protocol). For .NET services, this is handled by the **ServiceDefaults** project. For non-.NET services, you configure your service's OpenTelemetry SDK to point at the same OTLP endpoint.

You don't install extra tools. You don't configure Prometheus or Jaeger for local dev. You run your AppHost and observability is there.

## Code-First, Not Config-First

The key insight from the [Aspire architecture docs](https://aspire.dev/architecture/overview/) is that Aspire is a **"compiler for application topology"**. Your AppHost is a high-level description of your app's architecture, and Aspire "lowers" that description into:

- **Run mode**: Real processes, containers, and networking for local development
- **Publish mode**: Deployment-ready artifacts (Docker Compose files, Kubernetes manifests, Bicep templates) for production

This means you define your architecture once — in code, with IDE support — and Aspire handles the rest.
