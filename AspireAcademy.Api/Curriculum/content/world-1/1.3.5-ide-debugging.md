# Debugging: F5 Across Every Service

With Aspire's IDE integration, you press F5 and the debugger attaches to every service in your app. Set a breakpoint in the API, trigger a request from the frontend, and the breakpoint hits — across process boundaries, across languages, as if it were one application.

---

## The Problem Without Aspire

In a multi-service app without Aspire, debugging means:

1. Start PostgreSQL and Redis manually (or via Docker Compose)
2. Open the API project, configure its launch profile, hit F5
3. Open a second IDE window for the Python service, configure it, start it
4. Open a third window for the frontend, run `npm run dev`
5. Hope you got all the environment variables right
6. When a request fails, check each service's debugger independently

You're managing three debugger sessions, four terminal windows, and correlating behavior by eye. Most developers give up and add `console.log` everywhere.

## F5 with Aspire

The Aspire VS Code extension and Visual Studio integration change this completely:

### VS Code

1. Install the **Aspire extension** for VS Code
2. Open your AppHost project
3. Press **F5** (or click the play button the extension adds to your AppHost)
4. Aspire starts all resources — containers, processes, everything
5. The debugger attaches to every .NET project in your app model automatically
6. The Dashboard opens in your browser

Now set a breakpoint in your API controller. Send a request from the frontend. The breakpoint hits. Step through the code. Inspect variables. Continue. The request completes and the response appears in the browser.

### Visual Studio

Visual Studio has built-in Aspire support:

1. Set your AppHost as the startup project
2. Press **F5**
3. All .NET projects launch with debuggers attached
4. Containers and non-.NET services start via DCP
5. The Dashboard opens

Visual Studio also adds **Aspire-specific tooling** in the Solution Explorer — you can see resource status, health checks, and environment variables directly in the IDE.

## Debugging Tips

**Hot reload works.** Change a line of C# code while debugging. Aspire picks up the change without restarting. For frontend code (Vite), hot module replacement works as usual.

**Use the Dashboard traces to find what to debug.** If a request is slow, open the Dashboard traces view first. Find the slow span. Note which service and method it points to. Then set your breakpoint there. The Dashboard tells you *where* to look; the debugger tells you *why*.

> 🐛 **Common Gotcha:** If your breakpoint isn't being hit, check that the service is actually receiving the request. Open the Dashboard traces view — if there's no trace spanning your service, the request isn't reaching it. Common causes: wrong port, service not healthy yet, WaitFor dependency still pending.

> 📋 **Key Takeaways**
> - F5 starts your entire distributed app and attaches debuggers to all .NET services automatically
> - Set breakpoints across service boundaries — the debugger works as if it's one application
> - Use Dashboard traces to identify *where* to debug, then use the IDE debugger to understand *why*
> - Non-.NET services (Python, Go, Node) can be debugged by attaching language-specific debuggers alongside Aspire
