## The Scenario

Imagine you are building Bazaar. You start with a single project — a monolith — where the API, web UI, and recommendation logic all live in one process. Life is simple.

Then requirements grow. You split into separate services. The API stays in C# because your backend team knows .NET. The frontend moves to TypeScript and React because your UI engineers prefer it. The recommendation engine uses Python because the data science team needs scikit-learn. A background worker is written in Go for raw throughput.

You have built a distributed application. And you have inherited five problems that monoliths never face.

## 1. Service Discovery

> 📝 **Service discovery** — how one service finds the network address of another at runtime.

In a monolith, you call a method. In Bazaar, the TypeScript frontend needs to know the C# API's URL — and that URL changes between your laptop, CI, staging, and production. Hardcoding `localhost:5001` works until it does not.

Every language has its own way of reading environment variables or config files. The Go worker reads `os.Getenv("API_URL")`. The Python service uses `os.environ["API_URL"]`. The TypeScript frontend reads `process.env.API_URL`. Keeping all of these in sync across four services is your first headache.

## 2. Configuration Management

> 📝 **Connection strings** — addresses and credentials for databases, caches, or message brokers that a service needs at startup.

Each service needs connection strings, ports, API keys, and feature flags. Managing these across four services written in four languages, deployed to four environments (dev, CI, staging, prod), means dozens of values that must stay in sync. One typo in a connection string and an entire service fails silently at startup.

## 3. Observability Gaps

When a user reports that search is slow, which service is the bottleneck? Is it the TypeScript frontend? The C# API? The Python recommendation engine? The PostgreSQL database?

> 📝 **Observability** — understanding what is happening inside your system through logs, traces, and metrics.

Observability is trivial in a monolith (one log file, one debugger). In Bazaar, a single search request fans out across four services. Without **distributed tracing** (assigning a unique ID to each request and following it across service boundaries), you are reading four separate log files in four different formats and guessing.

## 4. Resilience

What happens when the Python recommendation engine goes down for 30 seconds? Does the C# API hang waiting for a response? Does the TypeScript frontend show a spinner forever? Does the Go worker crash?

> 📝 **Resilience** — handling partial failures gracefully using retry logic, circuit breakers, and timeouts so one failing service does not cascade into a full outage.

## 5. The Inner-Loop Tax

> 📝 **Inner loop** — the code → build → run → test cycle that developers repeat dozens of times per day.

All of these concerns add friction to your inner loop. You end up maintaining Docker Compose files with service definitions for each language, shell scripts to set environment variables, and separate terminal windows for each service's logs. The result: more time fighting infrastructure, less time writing features.

> 🧠 **Think about it:** If you have 4 services across 4 environments, how many individual configuration values might you need to keep in sync? What happens when one of them drifts?

## Quick Comparison

| Concern | Monolith | Distributed (Bazaar) |
|---|---|---|
| Service discovery | Method call | Network address per environment |
| Configuration | One config file | Dozens of values × N services × N envs |
| Observability | One log file | N log streams, no automatic correlation |
| Resilience | Process crash = total, but simple | Partial failures cascade unpredictably |
| Inner loop | `dotnet run` | Docker Compose + shell scripts + N terminals |

---

> 📋 **Key Takeaways**
> - Distributed applications inherit five problems monoliths never face: service discovery, configuration management, observability, resilience, and inner-loop friction
> - These problems compound — bad config leads to failed discovery, which leads to observability gaps
> - No single existing tool solved all five across all languages — until Aspire
