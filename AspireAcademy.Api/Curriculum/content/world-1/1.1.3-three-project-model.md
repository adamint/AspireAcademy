# The Three-Project Model

Every Aspire solution follows a consistent structure built around three types of projects. Understanding their roles is essential before writing any code.

## 1. AppHost (Orchestrator)

The **AppHost** is the entry point of your Aspire application. It's a console app that defines your application's resource graph — which services exist, what databases they use, and how they connect. Think of it as the blueprint for your distributed app.

```
MyApp.AppHost/
├── Program.cs          ← defines the resource graph
├── MyApp.AppHost.csproj
└── Properties/
    └── launchSettings.json
```

The AppHost references every other project in the solution. When you run it, Aspire's **DCP** (Developer Control Plane — the runtime engine that starts, stops, and monitors resources) launches all your services, assigns ports, injects configuration, and opens the Dashboard.

**Critical point:** The AppHost is **never deployed to production**. It's a dev-time orchestrator only. In production, each service runs independently, deployed through your normal CI/CD pipeline.

## 2. ServiceDefaults (Shared Library)

The **ServiceDefaults** project is a class library that configures **cross-cutting concerns** — functionality that every service in your app needs, such as:

- **OpenTelemetry**: Tracing, metrics, and structured logging
- **Health checks**: `/health` and `/alive` endpoints
- **Service discovery**: Automatic resolution of service URLs
- **HTTP resilience**: Retry policies, circuit breakers, and timeouts

```
MyApp.ServiceDefaults/
├── Extensions.cs       ← AddServiceDefaults() and related methods
└── MyApp.ServiceDefaults.csproj
```

Every service project references ServiceDefaults and calls `builder.AddServiceDefaults()` in its `Program.cs`. This single line wires up all the above — no copy-pasting config across projects.

## 3. Service Projects (Your Actual Apps)

These are your real applications — Web APIs, Blazor frontends, background workers, gRPC services. They're standard .NET projects with one addition: they reference ServiceDefaults.

```
MyApp.ApiService/
├── Program.cs          ← calls AddServiceDefaults()
└── MyApp.ApiService.csproj

MyApp.Web/
├── Program.cs          ← calls AddServiceDefaults()
└── MyApp.Web.csproj
```

## How Configuration Flows

The dependency chain is:

```
AppHost → references → ServiceDefaults
AppHost → references → ApiService, Web (via AddProject<T>)
ApiService → references → ServiceDefaults
Web → references → ServiceDefaults
```

At dev time, the AppHost injects environment variables into each service (connection strings, URLs, telemetry endpoints). ServiceDefaults reads those variables and configures each service automatically. Your service code stays clean — it doesn't know about Aspire, just about the standard .NET configuration system.
