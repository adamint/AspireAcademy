# OpenTelemetry: Tracing with ActivitySource

Distributed tracing lets you follow a single request as it flows across multiple services. In .NET, tracing is built on the `System.Diagnostics` namespace, and Aspire configures it automatically.

## The Core Concepts

### Activity and ActivitySource

An **`ActivitySource`** is a named producer of trace data. You create one per logical component:

```csharp
private static readonly ActivitySource s_activitySource = new("MyApp.OrderService");
```

An **`Activity`** represents a single unit of work — called a **span** in OpenTelemetry terminology. You create activities from an `ActivitySource`:

```csharp
using var activity = s_activitySource.StartActivity("ProcessOrder");
activity?.SetTag("order.id", orderId);
// ... do work ...
```

Each activity has a start time, duration, status, and key-value **tags** (attributes) that provide context.

### Spans and Traces

A **trace** is a tree of spans representing an end-to-end operation. For example, a user clicking "Place Order" might produce:

```
[WebFrontend] POST /orders  (200ms)
  └── [ApiService] POST /api/orders  (150ms)
        ├── [ApiService] INSERT INTO orders  (30ms)
        └── [ApiService] Publish OrderCreated  (10ms)
```

Each box is a span. The whole tree is a trace. Spans are linked by parent-child relationships.

## W3C TraceContext Propagation

When one service calls another over HTTP, how does the child service know it's part of the same trace? Through **W3C TraceContext** — a standard that defines HTTP headers (`traceparent` and `tracestate`) carrying trace and span IDs.

```
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
```

.NET's `HttpClient` automatically propagates these headers when OpenTelemetry instrumentation is configured. On the receiving side, ASP.NET Core reads the headers and creates a child activity linked to the parent trace.

ServiceDefaults configures this with:

```csharp
.WithTracing(tracing =>
{
    tracing.AddAspNetCoreInstrumentation()   // incoming requests
        .AddHttpClientInstrumentation();      // outgoing HTTP calls
});
```

**`AddAspNetCoreInstrumentation()`** creates spans for incoming HTTP requests. **`AddHttpClientInstrumentation()`** creates spans for outgoing `HttpClient` calls and ensures `traceparent` headers are sent.

## OTLP Export to the Dashboard

The final piece is getting trace data to the Dashboard. ServiceDefaults configures the **OTLP exporter**:

```csharp
if (!string.IsNullOrWhiteSpace(builder.Configuration["OTEL_EXPORTER_OTLP_ENDPOINT"]))
{
    builder.Services.AddOpenTelemetry().UseOtlpExporter();
}
```

The AppHost sets `OTEL_EXPORTER_OTLP_ENDPOINT` as an environment variable pointing to the Dashboard's OTLP receiver. Every span your service produces is sent there, and the Dashboard renders the waterfall view you saw in the Dashboard lesson.

## Creating Custom Spans

You can add your own spans for business operations:

```csharp
public class OrderService
{
    private static readonly ActivitySource s_source = new("MyApp.OrderService");

    public async Task<Order> PlaceOrderAsync(Cart cart)
    {
        using var activity = s_source.StartActivity("PlaceOrder");
        activity?.SetTag("cart.item_count", cart.Items.Count);

    }
}
```

Register the source name so OpenTelemetry collects it:

```csharp
.WithTracing(tracing =>
{
    tracing.AddSource("MyApp.OrderService");
});
```

The custom span will appear in the Dashboard's trace waterfall alongside the automatic HTTP spans.
