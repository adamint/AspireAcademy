# The Builder: Your App's Entry Point

*Constellation is scaffolded. Now let's understand the single most important line in every AppHost — the builder that creates your app model.*

---

Every Aspire AppHost starts the same way: create a builder, add resources, build, and run. The syntax differs by language, but the concept is identical.

## Creating the Builder

:::tabs
::tab{label="C#"}
```csharp
var builder = DistributedApplication.CreateBuilder(args);
```
::tab{label="TypeScript"}
```typescript
import { createBuilder } from "./.modules/aspire.js";
const builder = createBuilder();
```
::tab{label="Python"}
```python
import aspire
builder = aspire.create_builder()
```
:::

This single line sets up:

1. **Configuration** — loads settings from environment variables, command-line args, and config files
2. **Resource collection** — an empty collection where `addProject`, `addRedis`, etc. register resources
3. **Pipeline** — ordered steps that execute on `build().run()`
4. **Eventing** — lifecycle hooks for custom behavior

## The Fluent Builder Pattern

Every `add*` method returns a resource builder for fluent chaining. This pattern is the same across all languages:

:::tabs
::tab{label="C#"}
```csharp
var builder = DistributedApplication.CreateBuilder(args);

var redis = builder.AddRedis("cache");
var api = builder.AddProject<Projects.Api>("api")
                 .WithReference(redis)
                 .WithEnvironment("MY_VAR", "value");

builder.Build().Run();
```
::tab{label="TypeScript"}
```typescript
const builder = createBuilder();

const redis = builder.addRedis("cache");
const api = builder.addProject("api", "../Api")
                   .withReference(redis)
                   .withEnvironment("MY_VAR", "value");

await builder.buildAndRun();
```
::tab{label="Python"}
```python
builder = aspire.create_builder()

redis = builder.add_redis("cache")
api = builder.add_project("api", "../Api") \
             .with_reference(redis) \
             .with_environment("MY_VAR", "value")

builder.build_and_run()
```
:::

The pattern is always: **create builder → add resources → configure with `with*` methods → build and run**.

## Build() and Run()

`build()` finalizes the app model and returns a `DistributedApplication`. Calling `run()` starts the orchestrator:

1. Delegates to **DCP** (Developer Control Plane) — a Kubernetes-compatible API server that manages resource lifecycles
2. Starts all resources (processes, containers) concurrently, respecting dependency order
3. Launches the **Dashboard** for real-time monitoring
4. Blocks until shutdown (Ctrl+C)

> This is what the [Aspire architecture docs](https://aspire.dev/architecture/overview/) call **run mode** — the AppHost orchestrates your entire app locally. Later in the course, you'll learn about **publish mode**, where the same AppHost generates deployment artifacts instead.
