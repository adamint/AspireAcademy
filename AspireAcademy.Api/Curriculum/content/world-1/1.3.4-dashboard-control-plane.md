# The Dashboard: Your Control Plane

The Dashboard is not a monitoring page. It is a live control plane for your entire application — you manage, observe, and operate every service from one place.

---

## Beyond Monitoring

Most developer tools give you a read-only view: logs, maybe some metrics. The Aspire Dashboard is read-write. You can:

- **Start, stop, and restart** any resource with a button click
- **Run custom commands** — seed a database, flush a cache, trigger a migration
- **Inspect environment variables** — see exactly what connection strings and URLs were injected into each service
- **View health check status** — understand why a service is stuck in "Starting"
- **Browse endpoints** — click directly into your running services

## Custom Commands

This is the feature you will use every day. You define commands in your AppHost, and they appear as buttons in the Dashboard.

```csharp
builder.AddProject<Projects.BazaarApi>("api")
    .WithCommand("seed", "Seed Database", async context =>
    {
        var client = context.ServiceProvider
            .GetRequiredService<IHttpClientFactory>()
            .CreateClient();
        await client.PostAsync(
            "http://api/admin/seed", null, context.CancellationToken);
        return CommandResults.Success();
    })
    .WithCommand("flush-cache", "Flush Redis Cache", async context =>
    {
        // Clear all cached data with one click
        var redis = context.ServiceProvider.GetRequiredService<IConnectionMultiplexer>();
        var server = redis.GetServers().First();
        await server.FlushAllDatabasesAsync();
        return CommandResults.Success();
    });
```

Now open the Dashboard, click the dropdown on the "api" resource, and you see "Seed Database" and "Flush Redis Cache" as buttons. One click, done. No more curl scripts. No more "how do I reset the dev data?" in Slack.

## Resource Lifecycle Control

Beyond custom commands, the Dashboard has built-in lifecycle controls for every resource:

| Action | What it does |
|---|---|
| **Stop** | Gracefully stops the resource (SIGTERM for processes, docker stop for containers) |
| **Start** | Restarts a stopped resource |
| **Restart** | Stop + Start in sequence |

This is useful when you're iterating on a single service. Change your Python code, click Restart on the Python resource, and it picks up your changes — without restarting the entire app.

Click any resource name in the Dashboard to expand its details. The Environment Variables section shows every injected variable:

## Health Check Inspection

If a resource is stuck in "Starting" or shows "RuntimeUnhealthy," the Dashboard tells you why. Click the resource to see its health checks:

- **Check name** — which health check is failing
- **Status** — Healthy, Degraded, or Unhealthy
- **Last checked** — when it was last polled
- **Duration** — how long the check took (slow checks indicate infrastructure issues)

This is how you diagnose "my API won't start" — it's usually a WaitFor dependency whose health check hasn't passed yet, and the Dashboard shows you exactly which one.
