# Message-Based Communication

When services need to communicate asynchronously, a message broker decouples the publisher from the consumers. The API publishes events, and downstream services consume them independently.

## AppHost Wiring

```csharp
var rabbitmq = builder.AddRabbitMQ("messaging");

builder.AddProject<Projects.Api>("api")
    .WithReference(rabbitmq)
    .WaitFor(rabbitmq);

builder.AddExecutable("worker", "go", workerDir, "run", ".")
    .WithReference(rabbitmq)
    .WaitFor(rabbitmq);

builder.AddPythonApp("ml", mlDir, "app:app")
    .WithReference(rabbitmq)
    .WaitFor(rabbitmq);
```

All three services receive the RabbitMQ connection string as `ConnectionStrings__messaging`.

## Publishing from C#

```csharp
var connectionString = builder.Configuration.GetConnectionString("messaging");
var factory = new ConnectionFactory { Uri = new Uri(connectionString) };
using var connection = factory.CreateConnection();
using var channel = connection.CreateModel();

channel.BasicPublish("content-events", "content.created",
    body: JsonSerializer.SerializeToUtf8Bytes(new { Id = 42, Title = "New Article" }));
```

## Consuming in Go

```go
rabbitURL := os.Getenv("ConnectionStrings__messaging")
conn, _ := amqp.Dial(rabbitURL)
ch, _ := conn.Channel()

msgs, _ := ch.Consume("content-events", "", true, false, false, false, nil)
for msg := range msgs {
    fmt.Printf("Received: %s\n", msg.Body)
}
```

## Consuming in Python

```python
import os, pika

url = os.environ["ConnectionStrings__messaging"]
connection = pika.BlockingConnection(pika.URLParameters(url))
channel = connection.channel()

def callback(ch, method, properties, body):
    print(f"Received: {body}")

channel.basic_consume(queue="content-events", on_message_callback=callback, auto_ack=True)
channel.start_consuming()
```

## Why Messaging?

- **Decoupling** — the API does not need to know about the worker or ML service
- **Resilience** — if the worker is down, messages queue up and are processed when it restarts
- **Scaling** — add more worker instances to process messages in parallel

> 🌍 **Real World:** Netflix processes billions of messages per day through their messaging infrastructure. Even at much smaller scale, message queues prevent cascading failures — if your ML service goes down for 5 minutes, messages queue up in RabbitMQ and are processed when it recovers. No data loss, no retries needed from the API.
