# WaitFor — Health-Based Startup Ordering

`WaitFor(resource)` tells Aspire: do not start this service until the specified resource is running **and healthy**.

---

## The Problem

Without startup ordering, your C# API might start before PostgreSQL is ready:

```
[API] ERROR: Connection to postgres:5432 refused
[PostgreSQL] Initializing database...
[PostgreSQL] Ready to accept connections
[API] Still failing... connection refused
```

## The Solution

```csharp
var db = builder.AddPostgres("pg").AddDatabase("atlas-db");
var api = builder.AddProject<Projects.Api>("api")
    .WithReference(db)
    .WaitFor(db);
```

Now the sequence is:
1. DCP starts PostgreSQL
2. Health check polls PostgreSQL until it accepts connections
3. PostgreSQL reports healthy → API starts
4. API connects successfully on first attempt

## What "Healthy" Means

`WaitFor` does not just check if a container is running. It waits for the resource's **health check** to pass. Hosting integrations register health checks automatically:
- `AddPostgres` → checks if PostgreSQL accepts SQL connections
- `AddRedis` → checks if Redis responds to PING
- `AddRabbitMQ` → checks if the management API is reachable

The Dashboard shows resources in **Waiting** state while their dependencies are not yet healthy.

## WaitFor in Every Language

```csharp
api.WaitFor(db);                    // C#
await api.waitFor(db);              // TypeScript
api.wait_for(db)                    # Python
api.WaitFor(db)                     // Go
```

## Chaining

```csharp
builder.AddProject<Projects.Api>("api")
    .WithReference(postgres)
    .WithReference(redis)
    .WithReference(rabbitmq)
    .WaitFor(postgres)
    .WaitFor(redis)
    .WaitFor(rabbitmq);
```

All three health checks must pass before the API starts.
