# WithDataVolume for Persistent Storage

By default, container data disappears when the container stops. `WithDataVolume` creates a Docker named volume that persists database data across restarts.

## The Problem

Without persistence, every time you stop your Aspire app, all database data is lost. You'd need to re-run init scripts and re-seed data on each startup. During development iteration, this is usually undesirable.

## Adding a Data Volume

```csharp
var postgres = builder.AddPostgres("pg")
    .WithDataVolume();
```

This creates a Docker named volume mapped to the database's data directory (e.g., `/var/lib/postgresql/data` for PostgreSQL). The volume name is auto-generated based on the resource name.

## Named Volumes

You can provide an explicit volume name for reuse across sessions:

```csharp
var postgres = builder.AddPostgres("pg")
    .WithDataVolume("my-pg-data");
```

Named volumes persist until explicitly removed with `docker volume rm`.

## Data Bind Mounts

For cases where you want data stored in a specific host directory, use `WithDataBindMount`:

```csharp
var postgres = builder.AddPostgres("pg")
    .WithDataBindMount("./data/postgres");
```

| Feature          | Named Volume (`WithDataVolume`) | Bind Mount (`WithDataBindMount`) |
|------------------|---------------------------------|----------------------------------|
| **Managed by**   | Docker                          | Host filesystem                  |
| **Portability**  | Works on any Docker host        | Tied to specific host path       |
| **Performance**  | Optimized by Docker             | Depends on host filesystem       |
| **Visibility**   | Hidden in Docker storage        | Visible in your project          |

## Which Databases Support It?

All database hosting integrations support `WithDataVolume`:

- PostgreSQL, SQL Server, MySQL, MongoDB, Oracle
- Redis, Garnet, Valkey
- RabbitMQ, Kafka, NATS (for JetStream)
- Seq, Keycloak, Qdrant, Milvus

## When to Use vs. Skip

**Use volumes for:**
- Development iteration — keep data between restarts
- Demo environments — pre-populate once and persist

**Skip volumes for:**
- CI/testing — fresh data on every run
- Reproducing bugs — clean state guarantees consistency

## Combining with Init Scripts

A common pattern pairs init scripts with data volumes. Init scripts seed data on first run, and the volume persists it:

```csharp
var postgres = builder.AddPostgres("pg")
    .WithInitBindMount("./pg-scripts")
    .WithDataVolume();
```

Remember: init scripts only run when the data directory is empty. If the volume already contains data, scripts are skipped.

## Key Takeaways

- **`WithDataVolume()`**: Creates a Docker named volume for persistent storage.
- **`WithDataBindMount(path)`**: Maps a host directory for direct filesystem access.
- **Named volumes**: Auto-named by default; pass a name for explicit control.
- **Init + Volume**: Init scripts seed once; the volume preserves data thereafter.
- **CI/Testing**: Omit volumes for clean-state test runs.
