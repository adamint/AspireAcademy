# WaitForCompletion and Init Containers

`WaitForCompletion` is different from `WaitFor` — it waits for a resource to **finish and exit**, not to become healthy.

---

## The Use Case

Database migrations. You have a script that runs EF Core migrations:

```csharp
var migrator = builder.AddProject<Projects.Migrator>("migrator")
    .WithReference(postgres);

var api = builder.AddProject<Projects.Api>("api")
    .WithReference(postgres)
    .WaitForCompletion(migrator);
```

The sequence:
1. PostgreSQL starts and becomes healthy
2. Migrator starts, runs migrations, then **exits with code 0**
3. `WaitForCompletion` detects the exit → API starts
4. API connects to a fully-migrated database

## WaitFor vs WaitForCompletion

| Method | Waits For | Use Case |
|---|---|---|
| `WaitFor(resource)` | Resource reaches Running + healthy | Long-running services, databases |
| `WaitForCompletion(resource)` | Resource exits (code 0) | Init scripts, migrations, seed data |

## Init Containers

This pattern mirrors Kubernetes init containers. Common uses:

```csharp
// Database seeding
var seeder = builder.AddExecutable("seeder", "dotnet", seedDir, "run")
    .WithReference(postgres)
    .WaitFor(postgres);

var api = builder.AddProject<Projects.Api>("api")
    .WithReference(postgres)
    .WaitForCompletion(seeder);
```

## In TypeScript

```typescript
const migrator = await builder.addProject("migrator", "../Migrator");
await migrator.withReference(postgres);

const api = await builder.addProject("api", "../Api");
await api.withReference(postgres);
await api.waitForCompletion(migrator);
```

## Error Handling

If the resource exits with a non-zero exit code, `WaitForCompletion` reports a failure, and the dependent resource enters `FailedToStart`. The Dashboard shows the error.
