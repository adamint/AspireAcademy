# Seeding with Init Files

Init files let you run SQL scripts automatically when a database container starts for the first time. This is essential for seeding schema and test data during development.

## The WithInitFiles Pattern

`WithInitFiles` copies a local directory of scripts into the container's init script directory:

```csharp
var postgres = builder.AddPostgres("pg")
    .WithInitFiles("./pg-scripts");
```

**Signature:**
```csharp
IResourceBuilder<PostgresServerResource> WithInitFiles(
    this IResourceBuilder<PostgresServerResource> builder,
    string source)
```

The `source` path is relative to the AppHost project directory.

> **Note:** The older `WithInitBindMount` method is deprecated in favor of `WithInitFiles`. `WithInitFiles` copies files into the container rather than using bind mounts, making it work better with remote Docker hosts.

## How It Works

Each database engine maps init scripts to a specific container directory:

| Database   | Container Init Directory               |
|------------|----------------------------------------|
| PostgreSQL | `/docker-entrypoint-initdb.d/`         |
| MySQL      | `/docker-entrypoint-initdb.d/`         |
| MongoDB    | `/docker-entrypoint-initdb.d/`         |
| Oracle     | `/opt/oracle/scripts/startup/`         |

Scripts execute **only on first startup** when the data directory is empty. If you've attached a data volume with existing data, init scripts won't re-run.

## File Naming for Execution Order

Scripts execute in alphabetical order. Use numeric prefixes to control sequencing:

```
./pg-scripts/
  01-create-schema.sql
  02-seed-data.sql
  03-create-indexes.sql
```

## WithInitFiles vs. WithInitBindMount

`WithInitFiles` copies files into the container, while the deprecated `WithInitBindMount` uses Docker bind mounts:

```csharp
// Preferred — copies files into the container
var postgres = builder.AddPostgres("pg")
    .WithInitFiles("./pg-scripts");

// Deprecated — uses bind mounts (kept for backward compatibility)
var postgres = builder.AddPostgres("pg")
    .WithInitBindMount("./pg-scripts");
```

`WithInitFiles` is preferred because it works with remote Docker hosts and doesn't require the host directory to be accessible from the Docker daemon.

## WithCreationScript for Per-Database Init

For database-level scripts, use `WithCreationScript` on the database resource (available for PostgreSQL, SQL Server, and MySQL):

```csharp
var db = postgres.AddDatabase("catalog")
    .WithCreationScript("""
        CREATE TABLE Products (
            Id SERIAL PRIMARY KEY,
            Name TEXT NOT NULL
        );
        INSERT INTO Products (Name) VALUES ('Widget'), ('Gadget');
    """);
```

## Complete Example

```csharp
var builder = DistributedApplication.CreateBuilder(args);

var postgres = builder.AddPostgres("pg")
    .WithInitFiles("./pg-scripts")
    .WithDataVolume();

var catalogDb = postgres.AddDatabase("catalog");

builder.AddProject<Projects.CatalogApi>("catalog-api")
    .WithReference(catalogDb);

builder.Build().Run();
```

## Key Takeaways

- **`WithInitFiles(path)`**: Copies init scripts into the container's init directory.
- **First-run only**: Scripts execute only when the data directory is empty.
- **Alphabetical order**: Name files with numeric prefixes (`01-`, `02-`) for sequencing.
- **`WithInitBindMount(path)`**: Deprecated — uses bind mounts instead of copying files.
- **`WithCreationScript`**: Inline SQL for per-database initialization.
