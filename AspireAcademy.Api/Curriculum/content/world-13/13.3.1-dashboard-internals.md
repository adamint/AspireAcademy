# Dashboard Internals

The Aspire Dashboard is a Blazor Server application that receives resource state via gRPC and telemetry via OTLP. Understanding its architecture helps you contribute to the Dashboard codebase.

---

## Architecture

```
Browser (SignalR WebSocket)
  ^
  |
Blazor Server Engine
  |
  +-- Components/Pages/ResourcesPage.razor
  |     Renders resource table, subscribes to IDashboardClient
  |
  +-- Components/Pages/ConsoleLogs.razor
  |     Streams logs from IDashboardClient.SubscribeConsoleLogs()
  |
  +-- Components/Pages/StructuredLogs.razor
  |     Queries in-memory OTLP log store
  |
  +-- Components/Pages/Traces.razor
  |     Queries in-memory OTLP trace store
  |
  +-- Components/Pages/Metrics.razor
        Queries in-memory OTLP metric store
```

## IDashboardClient (gRPC)

The Dashboard connects to the AppHost via gRPC:

```csharp
// src/Aspire.Dashboard/ServiceClient/IDashboardClient.cs

public interface IDashboardClient : IAsyncDisposable
{
    // Subscribe to resource state changes (streaming)
    Task<ResourceViewModelSubscription> SubscribeResourcesAsync(
        CancellationToken cancellationToken);

    // Subscribe to console log lines (streaming)
    IAsyncEnumerable<IReadOnlyList<ResourceLogLine>> SubscribeConsoleLogs(
        string resourceName, CancellationToken cancellationToken);

    // Execute a command on a resource
    Task<ResourceCommandResponseViewModel> ExecuteResourceCommandAsync(
        string resourceName, string resourceType,
        CommandViewModel command, CancellationToken cancellationToken);

    // Get current resources (non-streaming)
    IReadOnlyList<ResourceViewModel> GetResources();
    ResourceViewModel? GetResource(string resourceName);
}
```

The implementation (`DashboardClient`) creates a gRPC channel:

```csharp
private GrpcChannel CreateChannel()
{
    var httpHandler = new SocketsHttpHandler
    {
        EnableMultipleHttp2Connections = true,
        KeepAlivePingDelay = TimeSpan.FromSeconds(20),
        KeepAlivePingTimeout = TimeSpan.FromSeconds(10),
    };

    return GrpcChannel.ForAddress(address, new()
    {
        HttpHandler = httpHandler,
        MaxReceiveMessageSize = 16 * 1024 * 1024  // 16 MB
    });
}
```

## OTLP Receiver

The Dashboard runs an OTLP (OpenTelemetry Protocol) receiver on a configured port. The `OTEL_EXPORTER_OTLP_ENDPOINT` environment variable points all services to this receiver.

Three signal types are stored in memory:

| Signal | Storage | Dashboard View |
|---|---|---|
| Traces | Ring buffer of recent spans | Waterfall timeline |
| Metrics | Time-series data points | Charts and gauges |
| Logs | Recent structured entries | Queryable log viewer |

## Resource Data Flow

```
1. DCP reports container state change
   via IKubernetesService.WatchAsync<Container>()
     |
     v
2. DcpExecutor updates ResourceNotificationService
   await notificationService.PublishUpdateAsync(resource, ...)
     |
     v
3. AppHost gRPC service streams to Dashboard
   DashboardService -> WatchResources RPC
     |
     v
4. DashboardClient receives ResourceViewModel update
   IDashboardClient.SubscribeResourcesAsync()
     |
     v
5. Blazor component re-renders
   ResourcesPage.razor - @foreach(var resource in resources)
     |
     v
6. Browser updates via SignalR
   (diff-based rendering, only changed elements)
```

Total latency: **milliseconds**. No polling anywhere in the chain.

## Why Blazor Server?

- **Real-time**: SignalR pushes diffs — no polling, no manual WebSocket management
- **Server-side data access**: No CORS issues, no API layer between Dashboard and gRPC
- **Small client**: No heavy JS bundle — the browser just renders HTML diffs
- **Consistent with .NET**: The Aspire team writes C#, not React

## In-Memory Storage

All telemetry is stored in memory and lost when the AppHost stops. This is intentional — the Dashboard is for development. For production monitoring, configure OTLP export to:
- **Seq** (structured logging)
- **Application Insights** (Azure)
- **Jaeger** (traces)
- **Prometheus** + **Grafana** (metrics)

> 🐛 **Common Gotcha:** All Dashboard telemetry (traces, metrics, structured logs) is stored in memory and lost when the AppHost stops. If you're investigating a tricky bug and restart your AppHost, all your traces are gone. For persistent telemetry during development, add a Seq or Jaeger container to your AppHost and configure OTLP export alongside the Dashboard.
