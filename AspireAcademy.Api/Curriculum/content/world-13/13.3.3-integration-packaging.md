# How Integrations Are Packaged

Every `Aspire.Hosting.*` integration follows strict naming, structure, and API conventions. Understanding these is essential for contributing new integrations.

---

## Project Structure

```
src/Aspire.Hosting.Redis/
  Aspire.Hosting.Redis.csproj       Project file
  RedisResource.cs                  Resource class
  RedisBuilderExtensions.cs         Extension methods (AddRedis, etc.)
  PublicAPI.Shipped.txt             API surface tracking
  PublicAPI.Unshipped.txt           Unreleased API changes

tests/Aspire.Hosting.Redis.Tests/
  RedisPublicApiTests.cs            API surface regression tests
  RedisResourceTests.cs             App model tests
  RedisFunctionalTests.cs           Integration tests (Docker)
```

## Naming Conventions

| Element | Convention | Example |
|---|---|---|
| **Package** | `Aspire.Hosting.{Service}` | `Aspire.Hosting.Redis` |
| **Resource class** | `{Service}Resource` | `RedisResource` |
| **Extension class** | `{Service}BuilderExtensions` | `RedisBuilderExtensions` |
| **Add method** | `Add{Service}()` | `AddRedis()` |
| **Sub-resource** | `{Parent}{Child}Resource` | `PostgresDatabaseResource` |

## PublicAPI Tracking

Every integration has `PublicAPI.Shipped.txt` and `PublicAPI.Unshipped.txt` files:

```
// PublicAPI.Shipped.txt
Aspire.Hosting.RedisResource
Aspire.Hosting.RedisResource.RedisResource(string! name) -> void
static Aspire.Hosting.RedisBuilderExtensions.AddRedis(
    this Aspire.Hosting.IDistributedApplicationBuilder! builder,
    string! name, int? port = null)
    -> Aspire.Hosting.IResourceBuilder<Aspire.Hosting.RedisResource!>!
```

These files are checked by Roslyn analyzers. If you change a public API without updating these files, CI fails.

## Analyzers

Two analyzer packages enforce correctness:

- **`Aspire.Hosting.Analyzers`** — validates AppHost usage patterns
- **`Aspire.Hosting.Integration.Analyzers`** — validates integration internals

Example: the analyzer warns if you call `.WithReference()` after `.Build()` (the model is frozen).

## Project File Pattern

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFramework>net9.0</TargetFramework>
    <Description>Aspire hosting integration for Redis</Description>
    <PackageId>Aspire.Hosting.Redis</PackageId>
  </PropertyGroup>

  <ItemGroup>
    <PackageReference Include="Aspire.Hosting" />
  </ItemGroup>

  <!-- Shared build properties from eng/ -->
  <Import Project="$(RepoRoot)eng/Versions.props" />
  <Import Project="$(RepoRoot)eng/Analyzers.props" />
</Project>
```

## Contributing a New Integration

1. Create `src/Aspire.Hosting.YourService/` following the naming conventions
2. Implement the resource class (inherit `ContainerResource` or `Resource`)
3. Implement `IResourceWithConnectionString` for connection string support
4. Write extension methods (`AddYourService()`)
5. Add health checks
6. Create `PublicAPI.Unshipped.txt` with all public API members
7. Write tests: app model tests, manifest tests, integration tests
8. Submit a PR following the contribution guide
