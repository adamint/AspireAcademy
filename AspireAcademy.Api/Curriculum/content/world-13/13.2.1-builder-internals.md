# DistributedApplicationBuilder Internals

When you call `DistributedApplication.CreateBuilder(args)`, a complex initialization sequence assembles the DI container, registers services, and prepares the execution context.

---

## CreateBuilder (from the source)

```csharp
// src/Aspire.Hosting/DistributedApplication.cs

public static IDistributedApplicationBuilder CreateBuilder(string[] args)
{
    WaitForDebugger();  // Pauses if --wait-for-debugger was passed
    ArgumentNullException.ThrowIfNull(args);
    var builder = new DistributedApplicationBuilder(
        new DistributedApplicationOptions() { Args = args });
    return builder;
}
```

## The Builder's Key Properties

```csharp
// src/Aspire.Hosting/DistributedApplicationBuilder.cs

public class DistributedApplicationBuilder : IDistributedApplicationBuilder
{
    private readonly HostApplicationBuilder _innerBuilder;

    public IHostEnvironment Environment => _innerBuilder.Environment;
    public ConfigurationManager Configuration => _innerBuilder.Configuration;
    public IServiceCollection Services => _innerBuilder.Services;
    public string AppHostDirectory { get; }
    public string AppHostPath { get; }
    public Assembly? AppHostAssembly => _options.Assembly;
    public DistributedApplicationExecutionContext ExecutionContext { get; }
    public IResourceCollection Resources { get; } = new ResourceCollection();
    public IDistributedApplicationEventing Eventing { get; }
        = new DistributedApplicationEventing();
    public IDistributedApplicationPipeline Pipeline { get; }
        = new DistributedApplicationPipeline();
}
```

Key takeaways:
- **`_innerBuilder`** is a standard `HostApplicationBuilder` — Aspire builds on top of the standard .NET host
- **`Resources`** is a `ResourceCollection` that grows as you call `AddRedis()`, `AddProject()`, etc.
- **`Eventing`** and **`Pipeline`** are created immediately — available before any resources are added

## The Constructor (Simplified)

```csharp
public DistributedApplicationBuilder(DistributedApplicationOptions options)
{
    _options = options;
    _innerBuilder = new HostApplicationBuilder(innerBuilderOptions);

    // 1. Parse execution context
    ExecutionContext = new DistributedApplicationExecutionContext(
        options.Operation ?? DistributedApplicationOperation.Run);

    // 2. Register core services
    Services.AddSingleton(ExecutionContext);
    Services.AddSingleton<ResourceNotificationService>();
    Services.AddSingleton<ResourceLoggerService>();
    Services.AddSingleton<ResourceCommandService>();
    Services.AddSingleton<IDistributedApplicationEventing>(Eventing);

    // 3. Mode-specific services
    if (ExecutionContext.IsRunMode)
    {
        Services.AddHostedService<DcpHost>();
        Services.AddSingleton<DcpExecutor>();
    }
    if (ExecutionContext.IsPublishMode)
    {
        Services.AddHostedService<PipelineExecutor>();
    }

    // 4. Register eventing subscribers from DI
    // (services implementing IDistributedApplicationEventingSubscriber)
}
```

## ExecutionContext

```csharp
public class DistributedApplicationExecutionContext
{
    public DistributedApplicationOperation Operation { get; }
    public string PublisherName { get; set; }
    public bool IsPublishMode => Operation == DistributedApplicationOperation.Publish;
    public bool IsRunMode => Operation == DistributedApplicationOperation.Run;
    public IServiceProvider ServiceProvider { get; }
}
```

The `Operation` is either `Run` (from `aspire run`) or `Publish` (from `aspire publish`). This single flag determines whether DCP starts or the pipeline generates artifacts.

## Build()

```csharp
public DistributedApplication Build()
{
    // Validate: no duplicate resource names (case-insensitive)
    foreach (var dup in Resources
        .GroupBy(r => r.Name, StringComparers.ResourceName)
        .Where(g => g.Count() > 1)
        .Select(g => g.Key))
    {
        throw new DistributedApplicationException(
            $"Multiple resources with the name '{dup}'.");
    }

    // Build the .NET host
    var application = new DistributedApplication(_innerBuilder.Build());
    return application;
}
```

`Build()` freezes the resource collection and builds the underlying `IHost`. After this point, no more resources can be added.

## Run()

```csharp
// DistributedApplication wraps IHost
public class DistributedApplication : IHost, IAsyncDisposable
{
    private readonly IHost _host;

    public async Task RunAsync(CancellationToken ct = default)
    {
        await _host.RunAsync(ct);
    }
}
```

`Run()` starts the .NET host, which starts all registered `IHostedService` instances. In run mode, that is `DcpHost`. In publish mode, that is `PipelineExecutor`.

> 🌍 **Real World:** The `ExecutionContext.IsRunMode` vs `IsPublishMode` split is why the same AppHost code works for both `aspire run` (local development) and `aspire publish` (deployment). The builder registers completely different service sets depending on the operation — DcpHost for running, PipelineExecutor for publishing. Your resource definitions stay the same; only the execution path changes.
