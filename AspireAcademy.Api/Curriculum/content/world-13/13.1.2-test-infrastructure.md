## Three Testing Patterns

### 1. App Model Tests (No DCP, No Docker)

Verify that extension methods produce the correct resource graph:

```csharp
[Fact]
public void AddRedis_CreatesResourceWithCorrectType()
{
    using var builder = TestDistributedApplicationBuilder.Create();
    builder.AddRedis("cache");

    var resources = builder.Resources;
    var redis = Assert.Single(resources.OfType<RedisResource>());
    Assert.Equal("cache", redis.Name);
}
```

`TestDistributedApplicationBuilder` is a lightweight builder that does not start DCP. Fast, no Docker required.

### 2. Manifest Tests (SDK Only)

Verify the JSON manifest output matches expectations:

```csharp
[Fact]
public async Task Redis_PublishesCorrectManifest()
{
    using var builder = TestDistributedApplicationBuilder
        .Create(DistributedApplicationOperation.Publish);
    var redis = builder.AddRedis("cache");

    var manifest = await ManifestUtils.GetManifest(redis.Resource);

    var expectedManifest = """
    {
        "type": "container.v0",
        "image": "docker.io/library/redis:7.4",
        "connectionString": "{cache.bindings.tcp.host}:{cache.bindings.tcp.port}"
    }
    """;

    Assert.Equal(expectedManifest, manifest.ToString());
}
```

### 3. Integration Tests (Real Docker)

Start real containers and verify runtime behavior:

```csharp
[Fact]
[RequiresDocker]
public async Task Redis_AcceptsConnections()
{
    var appHost = await DistributedApplicationTestingBuilder.CreateAsync<TestAppHost>();
    await using var app = await appHost.BuildAsync();
    await app.StartAsync();

    var rns = app.Services.GetRequiredService<ResourceNotificationService>();
    await rns.WaitForResourceAsync("cache", KnownResourceStates.Running);

    // Connect and verify
    var connectionString = await app.GetConnectionStringAsync("cache");
    var redis = ConnectionMultiplexer.Connect(connectionString);
    var pong = await redis.GetDatabase().PingAsync();
    Assert.True(pong > TimeSpan.Zero);
}
```

The `[RequiresDocker]` attribute skips the test if Docker is not running.

## Shared Utilities

- **`TestDistributedApplicationBuilder`** — lightweight builder for app model tests (no DCP)
- **`ManifestUtils`** — generates manifests without starting anything
- **`DistributedApplicationTestingBuilder`** — full test harness with Docker
- **`[RequiresDocker]`** — trait attribute that skips tests without Docker

> 🐛 **Common Gotcha:** Integration tests with `[RequiresDocker]` are automatically skipped in CI environments without Docker. But they're also skipped silently on YOUR machine if Docker isn't running. If you're wondering why your integration test "passes" but nothing seems to happen — check that Docker Desktop is actually running. The test was skipped, not passed.
