# DCP Communication

When you run an Aspire AppHost, the AppHost process does not start containers directly. It delegates to the **Developer Control Plane (DCP)** — a separate process that manages containers, executables, and networking.

---

## DCP as a Separate Process

```
┌─────────────────────┐       gRPC        ┌──────────────────────┐
│      AppHost         │ ───────────────▶  │        DCP           │
│  (App Model / C#)    │ ◀───────────────  │  (Local Orchestrator)│
└─────────────────────┘                    └──────────┬───────────┘
                                                      │
                                           ┌──────────▼───────────┐
                                           │  Container Runtime   │
                                           │  (Docker / Podman)   │
                                           └──────────────────────┘
```

## Key Classes

### DcpHost

`DcpHost.cs` is a `BackgroundService` that manages the DCP process lifecycle:
- Starts DCP as a child process
- Establishes the gRPC channel
- Monitors DCP health
- Tears down DCP on shutdown

### DcpExecutor

`DcpExecutor.cs` translates the app model into DCP resource definitions:
- Iterates over `DistributedApplicationModel.Resources`
- Creates DCP-specific objects (Container, Executable, ExecutableReplicaSet)
- Submits them via gRPC
- Handles state change notifications from DCP

### DcpKubernetesClient

DCP uses a **Kubernetes-inspired API**. `DcpKubernetesClient.cs` is a typed gRPC client that communicates using Kubernetes-style CRDs:

```
AppHost ──gRPC──▶ DCP
  CreateContainer("redis", image: "redis:7.4", port: 6379)
  CreateExecutable("api", path: "dotnet", args: ["run"])
  WatchResources() → stream of state changes
```

## DCP Resource Types

| DCP Type | Aspire Equivalent | Description |
|---|---|---|
| Container | `ContainerResource` | Docker/Podman container |
| Executable | `ProjectResource`, `ExecutableResource` | Local process |
| ExecutableReplicaSet | `ProjectResource` with replicas | Multiple process instances |

## The Lifecycle

1. AppHost calls `builder.Build().Run()`
2. `DcpHost` starts the DCP process
3. `DcpExecutor` submits resources via gRPC
4. DCP pulls images, starts containers, launches processes
5. DCP streams state changes back to `ResourceNotificationService`
6. Dashboard subscribes to notifications for real-time UI
7. On shutdown, DCP tears down all session-scoped resources
