# The Contribution Guide

How to contribute to the Aspire project — from finding issues to merging PRs.

---

## Finding Work

### Good First Issues

Filter GitHub issues by the `good first issue` label. These are specifically scoped for new contributors — clear requirements, bounded scope, and a maintainer available to help.

### Help Wanted

The `help wanted` label marks issues that the team needs external help with. These may be more complex but are well-defined.

### Areas That Need Contributors

| Area | Skills | Complexity |
|---|---|---|
| **New hosting integrations** | C# + Docker + target technology | Medium |
| **Dashboard improvements** | Blazor, CSS, data visualization | Medium |
| **CLI commands** | C#, System.CommandLine | Medium |
| **Polyglot SDK generators** | C# + target language (TS/Py/Go/Java/Rust) | High |
| **VS Code extension** | TypeScript, VS Code API | Medium |
| **Documentation** | Technical writing | Low |
| **Test coverage** | xUnit, Docker | Low-Medium |
| **Bug fixes** | Varies | Varies |

## The PR Workflow

1. **Fork** the repository on GitHub
2. **Clone** your fork locally
3. **Create a branch** from `main`:
   ```bash
   git checkout -b feature/add-meilisearch-hosting
   ```
4. **Make your changes** following the coding style (see `eng/.editorconfig`)
5. **Add tests** — app model tests (mandatory), integration tests (if applicable)
6. **Update PublicAPI files** if you changed public APIs
7. **Run tests locally**:
   ```bash
   dotnet test tests/Aspire.Hosting.YourService.Tests/
   ```
8. **Push and open a PR** with a clear description
9. **Address review feedback** — maintainers will review within a few days
10. **Merge** — maintainers squash-merge after approval

## Code Review Expectations

- **Focused PRs** — one feature or fix per PR. Do not mix formatting changes with logic changes.
- **Tests required** — new features must have tests. Bug fixes should have a regression test.
- **PublicAPI.txt** — any new public API must be listed in `PublicAPI.Unshipped.txt`
- **Documentation** — user-facing changes should update docs or at least note the need
- **Formatting** — must match the repo style (enforced by `.editorconfig` and CI)

## Where to Ask Questions

- **GitHub Discussions** on the `dotnet/aspire` repo
- **Discord**: [aka.ms/aspire/discord](https://aspire.dev/community)
- **Stack Overflow**: `dotnet-aspire` tag
- **PR comments** — maintainers are responsive to questions in PR threads

> 🌍 **Real World:** The `good first issue` label on dotnet/aspire is genuinely curated. Maintainers scope these issues specifically for newcomers — they include clear requirements, bounded scope, and a maintainer available to help. Many successful Aspire contributors started with a documentation fix or a test gap labeled `good first issue` and progressed to feature work within weeks.
