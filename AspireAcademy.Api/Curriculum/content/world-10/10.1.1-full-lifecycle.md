# Aspire's Full Lifecycle

The same AppHost that runs locally also drives production deployment — one model, two execution modes.

---

## The Two Modes

When you call `builder.Build()`, the `ExecutionContext` determines what happens:

```
aspire run    → ExecutionContext.IsRunMode = true
                → DCP starts resources locally
                → Dashboard opens

aspire publish → ExecutionContext.IsPublishMode = true
                → Pipeline generates deployment artifacts
                → No services are started
```

## From Dev to Prod

```
Development                     Production
─────────────                   ──────────
aspire run                      aspire publish --target azure
    ↓                               ↓
DCP starts containers           PipelineExecutor runs
Dashboard shows telemetry       Steps build, push, generate
You write code                  main.bicep → Azure resources
                                docker-compose.yml → containers
                                k8s-manifests/ → Kubernetes
```

## Why One AppHost

The app model captures _what_ your application is — resources, relationships, configuration. How those resources are realized changes between modes:

| Concept | Run Mode | Publish Mode |
|---|---|---|
| PostgreSQL | Docker container on localhost | Azure Database for PostgreSQL |
| Redis | Docker container on localhost | Azure Cache for Redis |
| API project | dotnet run on localhost | Container image pushed to ACR |
| Frontend | Vite dev server | Static build served by CDN |
| Connection strings | localhost:PORT | Managed identity + DNS |

The publishing pipeline transforms the app model from local to production automatically.

> 🌍 **Real World:** Aspire's single-AppHost philosophy means your dev environment is always in sync with production topology. If you add a new service reference in the AppHost, it automatically appears in generated deployment manifests — no separate IaC updates needed.
