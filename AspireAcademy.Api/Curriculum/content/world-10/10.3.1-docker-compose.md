# Docker Compose Publishing

Your client runs their own servers — no Kubernetes, no cloud PaaS. They need a `docker-compose.yaml` they can deploy with `docker compose up` and nothing else. Aspire generates it directly from your app model, translating resource references into service names, health checks into `depends_on` conditions, and volumes into named Docker volumes.

---

## Generating Compose Files

```bash
aspire publish --target docker-compose --output-path ./deploy/compose
```

## What Gets Generated

For Bazaar, the output looks like:

```yaml
services:
  pg:
    image: postgres:17
    environment:
      POSTGRES_PASSWORD: ${PG_PASSWORD}
    volumes:
      - pg-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD", "pg_isready"]

  cache:
    image: redis:7.4

  api:
    build: ../Bazaar.Api
    environment:
      ConnectionStrings__bazaar-db: "Host=pg;Port=5432;..."
      ConnectionStrings__cache: "cache:6379"
    depends_on:
      pg:
        condition: service_healthy
      cache:
        condition: service_started

  frontend:
    build: ../bazaar-web
    environment:
      services__api__http__0: "http://api:5000"
    depends_on:
      api:
        condition: service_started

volumes:
  pg-data:
```

## Key Translations

| Aspire Concept | Docker Compose Equivalent |
|---|---|
| `AddPostgres("pg")` | `services.pg` with postgres image |
| `WithReference(pg)` | Environment variables pointing to service name |
| `WaitFor(pg)` | `depends_on` with `service_healthy` |
| `WithDataVolume("pg-data")` | Named volume in `volumes:` |
| `WithLifetime(Persistent)` | No direct equivalent (Compose containers always persist) |

> 🐛 **Common Gotcha:** Aspire's `WaitFor()` translates to `depends_on` with `service_healthy` in Docker Compose — but this requires the dependency to have a `healthcheck` defined. If the upstream service has no health check, Compose will hang waiting for a health status that never comes.
