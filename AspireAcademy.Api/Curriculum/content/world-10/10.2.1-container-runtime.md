# IContainerRuntime — Docker and Podman

The container runtime abstraction lets Aspire work with both Docker and Podman for building, tagging, and pushing images.

---

## The Interface

```csharp
public interface IContainerRuntime
{
    string Name { get; }
    Task<bool> CheckIfRunningAsync(CancellationToken ct);
    Task BuildImageAsync(string contextPath, string dockerfilePath, ...);
    Task TagImageAsync(string localImage, string targetImage, CancellationToken ct);
    Task PushImageAsync(IResource resource, CancellationToken ct);
    Task LoginToRegistryAsync(string server, string username, string password, CancellationToken ct);
}
```

## Docker Implementation

Uses `docker buildx build`, `docker tag`, `docker push`. Supports:
- Multi-platform builds
- Build arguments and secrets
- OCI and Docker image formats
- Registry authentication

## Podman Implementation

Equivalent commands using Podman. Aspire auto-detects which runtime is available.

> 🐛 **Common Gotcha:** Aspire checks for container runtimes by looking for `docker` and `podman` on the system PATH. If both are available, Docker takes precedence. In CI environments using Podman (common on RHEL/Fedora runners), make sure Docker isn't also installed — otherwise Aspire will use Docker and potentially fail if Docker's daemon isn't running while Podman's is.

## In the Pipeline

```
build-prereq
    ↓
build (IContainerRuntime.BuildImageAsync)
    ↓
push-prereq
    ↓
push (IContainerRuntime.PushImageAsync)
    ↓
publish (generate manifests referencing pushed images)
```

The pipeline builds container images for your .NET projects, pushes them to a registry (ACR, Docker Hub, etc.), and then generates deployment manifests that reference the pushed image tags.
